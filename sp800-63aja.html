    ABSTRACT

    このガイドラインは、デジタル認証で使用する ID の登録および検証に焦点を当てている。このプロセスの中心は、申請者がクレデンシャル・サービス・プロバイダ（CSP） に自分自身を確実に識別する証拠を提供することで、CSP が有用な ID 保証レベルでその 識別をアサートできるようにする、身元確認として知られるプロセスである。本文書は、3 つの ID 保証レベルごとに技術要件を定義する。このガイドラインは、この目的以外の標準の開発または使用を制約することを意図するものではない。本書は、NIST 特別刊行物（SP）800-63A に取って代わるものである。

    序文

    本文書、および関連する[SP800-63]、[SP800-63B]、[SP800-63C] の目的は、指定された保証レベルでデジタル ID を管理するためのプロ セスと技術に関するガイダンスを組織に提供することである。

    本文書は、ID サービスへの登録またはオンライン・リソースへのアクセスを提供する目的で、各 ID 保証レベル（IAL）で個人の ID を証明するための要件を規定する。 この文書は、ネットワーク上または対面での個人の 身元確認に適用される。

    はじめに

    このセクションは参考になる。

    オンライン・サービスを提供する際の課題のひとつは、一連の活動を既知の一個人に関連付けることができることである。これが不要な状況もあるが、現実の対象者との関連付けを確実に確立することが重要な状況もある。この例には、政府サービスへのアクセスや金融取引の実行が含まれる。また、現実の対象者との関連付けが、規制（例えば、金融業界の「顧客識別プログラム」要件）により要求される状況や、リスクの高い行動（例えば、ダムからの放水速度の変更）に対する説明責任を確立するために要求される状況もある。

    このガイダンスは、オンライン・サービスにアクセスする対象者と実在の人物との関係を、 ある程度の保証をもって確立するプロセスとして 身元確認を定義する。本文書は、連邦機関、サードパーティのクレデンシャル・サービス・プロバイダ（CSP）、および 身元確認・サービスを提供または利用するその他の組織向けのガイダンスを提供する。

    身元確認に期待される成果

    身元確認に期待される成果には以下が含まれる：

    • ID の解決：CSP またはオンライン・サービスが提供するユーザの集団のコンテキスト内で、クレームされた ID が 1 人の固有の個人に対応することを判断する。
    • 証拠の検証：提供された証拠が真正、本物、有効であることを確認する。
    • 属性の検証：コア属性の正確さを確認する。コア属性は、身元確認に必要な最小限のセットである。
    • 本人確認：請求者が提示された証拠および属性の真正な所有者であることを確認する。
    • ID 登録：ID を証明された申請者を、CSP の ID サービスに加入者として登録する。
    • 不正行為の軽減：不正な ID を使用した給付、サービス、データ、資産へのアクセスを検出、対応、防止する。

    身元確認サービスには、データの最小化などプライバシー強化の原則を組み込むとともに、 期待される結果を達成しつつ申請者の負担を最小限に抑えるために、優れた使いやすさを 実践することが期待される。

    アイデンティティ保証レベル

    加入者の ID に対する保証（信頼）は、定義された ID 保証レベル（IAL）に関連するプロセスを使用 して確立される。連続する各 IAL は、より高い保証を達成するために、より低い IAL の要件に基づいて構築される。

    身元確認なし(No identity proofing)：申請者を特定の実在の人物と結びつける必要はない。対象者の活動に関連して提供される属性はすべて、自己主張であるか、自己主張として扱われる。証拠は検証されず、属性は検証も確認もされない。

    IAL1： 身元確認プロセスは、主張された ID の現実世界の存在をサポートし、申請者がその ID に関連 していることをある程度保証する。コア属性は、ID の証拠から取得するか、申請者が自己主張する。 すべてのコア属性（第 2.2 節参照）は、権威のある情報源または信頼できる情報源に対して検証され、 その属性を ID のプルーフィング・プロセスを受ける人物に結び付けるための手順が踏まれる。身元確認は、CSP の代表者（プルーフィング・エージェントまたは信頼できるレフェリー） の立会いの有無にかかわらず、リモートまたはオンサイトのプロセスを使用して実行される。ID プルーフが正常に完了すると、申請者はサブスクライバ・アカウントに登録され、サブスク ライバが提供する認証機能など、あらゆる認証機能がそのアカウントにバインドされる。IAL1 は、高度にスケーラブルな攻撃を制限し、合成 ID に対する保護を提供し、危殆化した PII を使用する攻撃に対する保護を提供するように設計されている。

    IAL2：IAL2 は、追加的な証拠の収集、および証拠の検証および ID 検証のより厳密なプロセスを要求す ることによって、身元確認プロセスにさらに厳密さを追加する。IAL2 は、IAL1 によって対処される脅威に加え、大規模攻撃および標的型攻撃を制限し、 基本的な証拠の改ざんおよび証拠の盗難に対する保護を提供し、基本的なソーシャル・エンジニ アリング戦術に対する保護を提供するように設計されている。

    IAL3：IAL3 では、訓練された CSP 担当者（プルーフィング・エージェント）が、現場で立会う 身元確認・セッションの一環として申請者と直接対話し、少なくとも 1 つのバイオメトリクスを収集する要件が追加される。現場での身元確認セッションの成功は、申請者の加入者アカウントへの登録、およびそのアカウ ントに関連付けられた（バインドされた） 1 つ以上の認証子の引き渡しで終了する。IAL3 は、より高度な攻撃を制限し、高度な証拠の改ざん、盗難、および否認に対する保護 を提供し、より高度なソーシャル・エンジニアリングの手口に対する保護を提供するように設計さ れている。

    身元確認の概要

    このセクションは規範的なものである。

    このセクションでは、身元確認および登録プロセスの概要、ならびに申請者が主張する ID の解決、検証、および検証をサポートする要件を示す。また、身元確認プロセスの追加的側面に関するガイドラインも提供する。 これらの要件は、申請された ID が実世界に存在し、申請者がその ID に関連する個人であるこ とを保証することを目的としている。

    さらに、本ガイドラインは、解決、検証、および確認を達成できる複数の方法を規定し、身元確認プロ セスをサポートする複数の種類の ID 証拠を提供する。CSP および組織は、手段、能力、および技術アクセスが異なる申請者のアクセスを促進するた めに、身元確認サービスおよびプロセスを実装する際にオプションを提供しなければならない（SHALL）。これらのオプションには、複数の種類および組み合わせの ID 証拠を受け入れること、複数のデータ検証ソースをサポートすること、ID を検証する複数の方法を可能にすること、関与のための複数のチャネル（現場、遠隔など）を提供すること、および申請者のための支援メカニズ ム（申請者の参照など）を提供することが含まれるべきである（SHOULD）。

    CSP は、提供される各 身元確認・オプション（身元確認の種類、検証ソース、支援メカニ ズムなど）に関連するリスクを評価し、必要に応じて不正行為を軽減する管理を実施するものとする。最低でも、CSP は、各オプションが全体として同等の保証を提供するように、各オプションを設計するものとする。

    身元確認と登録

    身元確認の目的は、指定された確実性レベルにおいて、身元確認・プロセ スに関与する申請者が本人であることを保証することである。本文書は、CSP が申請者を指定保証レベルで身元確認するための 3 段階のプロセスを提示する。第 1 段階である ID 解決は、適切な ID 証拠および属性情報を収集し、申請者が CSP がサービ スを提供する集団における固有の ID であり、実在の人物であることを判断する。第 2 段階の ID 検証では、第 1 段階で収集した証拠および属性情報の真正性、正確性、および妥当 性を検証する。第 3 のステップである ID 検証では、ID エビデンスを提示する申請者が、検証されたエ ビデンスが発行され、検証された属性が関連付けられたのと同じ個人であることを確認する。ほとんどの場合、指定された IAL に対して申請者の ID 検証に成功すると、CSP は申請者（現 在は ID サービスのサブスクライバ）に対して一意のサブスクライバ・アカウントを確立し、このア カウント内の証明された ID に 1 つまたは複数の認証者をバインドできるようにする。

    身元確認は、給付またはサービスに対する適合性または資格の決定をサポートする組織のビジネス・ プロセスの一部である。このガイドラインは、適切なレベルの ID 保証のガイダンスを提供するが、給付またはサービス の適合性および資格の決定は、これらの 身元確認プロセスとは異なるビジネス・プロセスの決定 であり、このガイドラインの範囲外である。

    身元確認の役割

    申請者およびリスク・シナリオのさまざまなニーズを満たす 身元確認の提供を支援す るために、プルーフィング・プロセスではさまざまな個人がさまざまな役割を果たすことが期待 される。これらのガイドラインの一貫した実施を支援するために、以下の ID プルーフ ィングの役割が定義されている：

    1. プルーフィング・エージェント- 現場または遠隔で 身元確認・セッションに出席し、ID 証拠を目視で検査し、証拠が改ざんされていないと判断するなど、限定的でリスクに基づく判断を行うよう訓練された CSP のエージェント。
    2. 信頼される審判員- 申請者が定義された IAL プルーフィング・プロセスの期待される要件を満たせな い場合に、申請者の 身元確認・ケースに関してリスク・ベースの判断を下す ための訓練を受けた CSP のエージェント。プルーフィング・エージェント（信頼されるレフェリーもこの役割を果たす場合がある） とは異なり、証拠の物理的検査および参照顔画像との申請者の視覚的比較による検証および検 証をサポートする能力に加え、欺瞞およびソーシャル・エンジニアリングの兆候を検出する訓練 を含む、より実質的な訓練レベルが期待される。信頼される推薦人の要件は、第 3.1.13.1 項に記載されている。

      注：信頼されるレフェリーは、申請者が必要な ID 証拠を所持していない場合、または証拠 上の属性がすべて申請された ID と一致しない場合（最近の名前や住所の変更など）を含め、例外処 理シナリオをサポートするために追加の研修およびリソースを受けるという点で、プルーフ ィング・エージェントとは異なる。

    3. 申請者参照- 申請者の身元、申請者に関連する特定の属性、または個人の状況に関連する条件 （緊急状態、ホームレスなど）を保証できる申請者の代表者。この個人は、身元確認プロセスにおいて申請者に代わって行動することはないが、 ID の主張を裏付けるために呼び出されるリソースである。申請者の参照に関する要件は、第 3.1.13.3 節に記載されている。
    4. プロセス・アシスタント- プルーフィング・プロセスを支援するが、意思決定またはリスクベースの評価を支援 しない個人（翻訳、テープ起こし、アクセシビリティ支援など）。プロセス・アシスタントは、CSP または申請者が提供する。

    CSP は、上記の役割のどれが ID サービスに適用されるかを特定し、第 3 節に規定される要件および期待に沿っ たトレーニングおよびサポート・リソースを提供しなければならないものと する 。

    身元確認の種類

    身元確認・プロセスの一部として解決、検証、および確認を提供する能力は、CSP がサービ スを提供する多様なユーザ、コミュニティ、および依存当事者をサポートするために、技術、通信チャネ ル、および 身元確認の役割を組み合わせて提供される。プルーフィングの種類は、2 つの具体的な要因（出席の有無と実施場所）に基づいて分類できる。

    1. リモート無人 身元確認- 解決、検証、および確認のプロセスが完全に自動化されており、プルーフィング・エージェントとの対話が不要な場合に実施される 身元確認。 プルーフィング・プロセスで使用される場所およびデバイスは、CSP によって管理されない。
    2. 遠隔出席型 身元確認-申請者がプルーフィング・エージェントとの安全なビデオ・セ ッションを通じて、解決、検証、および確認のステップを完了する場合に実施される 身元確認。プルーフィング・プロセスで使用される場所およびデバイスは、CSP によって管理されない。
    3. オンサイト無人 身元確認- 個人が管理されたワークステーションまたはキオスクと対話して行う 身元確認であるが、プルーフィング・エージェントとの対話は必要ない。プロセスは完全に自動化されているが、物理的な場所で、CSP によって承認されたデバイスを使用する。
    4. オンサイト立会 身元確認-申請者がプルーフィング・エージェントの立会いの下、 解決、検証、確認を含む 身元確認・プロセス全体を完了する、物理的な環境 で実施される 身元確認。プルーフ ィング・エージェントは、利用者と同席するか、キオスクまたは装置を介して利用者と対話 する。物理的な場所および装置はすべて、CSP によって承認される。

    各保証レベルにおける要件は、CSP が（必要に応じて）異なる保証レベルの要件を満たすために、証明タイ プのさまざまな組み合わせを実装できるように構成される。IAL1 および IAL2 サービスを提供する CSP は、リモート無人 ID 検証プロセスを提供し、少なくとも 1 つの有人 ID 検証プロセス・オプションを提供するものとする（ SHALL）。IAL1 および IAL2 サービスを提供する CSP は、申請者が 1 つのタイプで不合格になった場合に、 申請者が証明タイプを移行できるようにする身元確認プロセスをサポートすべきである（例えば、リ モート無人証明に不合格になった申請者がリモート有人証明に移行できるようにする）。

    コア属性

    身元確認・プロセスには、身元確認を達成するために必要な最小限の属性の提示と 検証が含まれる。これには、解決、検証、および検証を完了するために必要なものが含まれる。特定のユースケースに必要なコア属性は、サービスを提供するコミュニティの性質によって変 化するが、以下の属性は、証明プロセスをサポートするために CSP によって収集されるべ きである（SHOULD）：

    • ファーストネーム申請者の名前
    • ミドルネームまたはイニシャル申請者のミドルネームまたはイニシャル（あれば）。
    • 姓：申請者の姓または姓名。
    • 政府識別子：政府の記録で申請者に関連付けられた固有の識別子（SSN、TIN、運転免許証番号など）。
    • 物理的住所：物理的住所（PhysicalAddress）：プルーフィング・プロセスに関する連絡を申請者が受け取ることができる物理的住所：デジタルアドレス：申請者がプルーフィングプロセスに関する連絡を受け取ることができるデジタルアドレス（電話やEメールなど）。

    CSP および RP の要求に応じて、これらに追加の属性を追加することができる。CSP および RP は、すべてのコア属性をトラスト・アグリーメントおよびプラクティス・ ステートメントに文書化しなければならない（SHALL）。プライバシー・リスク評価後、CSP は、身元確認を完了するために必要ではないが、他の RP ビジネス・ プロセスをサポートする可能性がある追加属性を要求してもよい（MAY）。追加属性を要求するためのプライバシー要件の詳細については、第 3.1.3 節を参照のこと。

    アイデンティティの解決

    ID 解決の目標は、特定の集団またはコンテキスト内で個人を一意的かつ正確に識別するために、可能な限り 最小の属性セットを使用することである。このステップでは、申請者が収集した属性を、CSP がサービスを提供するユーザの記録 に保存されている属性と比較する。ID 解決は、潜在的な不正行為の初期検出を含む全体的な 身元確認プロセスの出発点であるが、決して完全で成功した 身元確認プロセスを示すものではない。

    アイデンティティの検証およびアイデンティティの証拠収集

    ID 検証の目標は、申請者から最も適切な ID 証拠を収集し、それが真正（改ざんまたは偽造 されていない）であり、正確（該当するデータが正しく、最新であり、申請者に関連している） であり、有効であることを判断することである。

    注：この文書では、有効期限が切れていたり、決められた期間外に発行されたものであっても、証 拠は ID を証明する有用な手段であり続けるという認識の下、期限切れではなく「有効」という用語を使 用している。

    ID 証拠収集は ID 検証プロセスを支援し、2 つのステップから構成される：1) 身元確認申請者による CSP への ID 証拠の提示、および 2) 提示された証拠が適用される強度要件を満たしていると いう CSP による判定。

    1. 発行の厳しさ、
    2. 正確性と真正性のチェックを含む、検証に対する信頼性を提供する能力。
    3. 証拠を提示する申請者の検証に対する信頼を提供する能力。

    アイデンティティの証拠と属性の検証

    ID エビデンスの検証では、提示されたエビデンスを検証して、それが真正であり（偽造または改ざん されていない）、正確であり（エビデンスに記載された情報が正しい）、有効である（有効期限が切れていないか、 CSP が定義した発行または有効期限内である）ことを確認する。属性の検証には、提示されたエビデンスから取得されたものであれ、自己主張によるものであれ、コア属性の 正確性を確認することが含まれる。以下のサブセクションでは、エビデンスおよび属性の検証で許容される方法を示す。

    エビデンスの検証

    CSP は、確認することにより、提示された証拠の真正性、正確性、および有効性を検証するものとする：

    • 証拠は正しい形式であり、ID 証拠の種類に対応する完全な情報が含まれている；
    • 証拠は偽造されたものではなく、改ざんもされていない；
    • セキュリティ機能
    • 証拠の情報は正確だ。

    エビデンスの検証方法

    提示されたエビデンスを検証するために許容される方法には、以下が含まれる：

    • 現場での身元確認のため、訓練を受けた担当者による目視および触覚検査；
    • 訓練を受けた職員による目視検査で、遠隔地での身元確認を行う；
    • 適切な技術を用いた文書検証プロセスの自動化。
    • デジタル証拠、または属性データオブジェクトの出所と完全性を暗号的に検証すること。

    属性の検証

    CSP は、Sec.2.2 に記載されるとおり、すべてのコア属性を、ID エビデンスから取得したものであれ 申請者が自己主張したものであれ、Sec.2.4.2.4 に記載されるとおり、権威ある情報源または信頼できる情報源で検証するものと する。

    検証ソース

    CSP は、以下の基準を満たす、権威ある情報源または信頼できる情報源を使用するものとする。

    権威のある情報源とは、ID の証拠または属性の発行元であるか、運転免許証データの州 DMV や社会保障カードおよび社会保障番号の社会保障庁など、発行元が保持する情報への直接アクセスを持つものである。権威ソースはまた、米国自動車管理者協会の運転免許証データ検証（DLDV）サービスなど、証 拠または属性の発行ソースへの直接アクセスを提供または可能にするものである。

    信頼できる情報源とは、ID の証拠および属性情報の正確性を提供または検証できる実体である。規制監督（公正信用報告法（FCRA）など）の対象であることに加え、信頼できる情報源は、正 確な情報源にトレースできる属性情報にアクセスできるか、正確性、一貫性、および最新性 のために相関する複数の情報源から得た ID 属性情報を保持している。信頼できる情報源の例は、FCRA の対象となる信用情報機関である。

    Identity Verification

    ID 検証の目標は、主張された有効な ID と 身元確認プロセスに関与する現実の申請者との間のリン ケージを、指定された信頼レベルで確立することである。言い換えれば、検証は、証拠を提示する申請者がその証拠の正当な所有者であることを保証 する。

    Identity Verification Methods

    CSP は、以下の方法の 1 つまたは複数を通して、申請された ID と身元確認プロセスに従事する申請 者との関連性を検証しなければならない（SHALL）。セクション 4では、各 IAL で許容される検証方法を示す。

    • 確認コードの検証（Confirmation Code Veriﬁcation）個人は、第 3.1.8 節に指定された要件に一致する確認コードの返送によって、ID 証拠の一部 の管理を証明することができる。
    • 認証およびフェデレーション・プロトコル（Authentication and Federation Protocols）個人は、認証またはフェデレーション・プロトコルの使用を通じて、デジタル・アカウント （オンライン銀行口座など）または署名済みデジタル・アサーション（検証可能クレデンシャルな ど）の制御を示すことができる。これは、デバイスまたは読み取り機へのクレデンシャルの提示によって直接行うこともできるが、 リモート ID プロファイル・セッション中に行うこともできる。
    • マイクロ・トランザクション（Micro Transaction）個人は、CSP とエビデンスの発行元（銀行口座など）との間で行われたマイクロ・トランザクションに基づく 値を返すことで、エビデンスの一部を管理していることを証明できる。
    • オンサイト対面（立会有り）での顔画像の視覚的な比較（Onsite-In-person (Attended) visual facial image comparison）プルーフィング・エージェントと申請者は、身元確認イベントのために対話する。プルーフィング・エージェントは、ID 証拠に提示された顔の肖像と、ID プ ロファイニング・イベントに参加した申請者の顔の視覚的比較を実行する。
    • リモート（立会有りまたは無人）での顔画像の視覚的な比較（Remote (Attended or Unattended) visual facial image comparison）プルーフィング・エージェントは、ID 証拠に提示された、または発行元によって保存された顔 の肖像と、ID プロファイニング・イベントに参加した申請者の顔画像の視覚的比較を実行する。プルーフィング・エージェントは、ID プ ロファイニング・イベントの一部または全部の間に申請者と直接対話する（有人）か、キャプチャされた ビデオまたは写真および証拠のアップロードされたコピーを使用して、後で比較を行う （無人）。比較を後で実行する場合、キャプチャされたビデオまたは写真が、ID プロファイ ン・イベント中にその場にいた実際の申請者から撮影されたものであることを確認するための手 順が取られる。
    • 自動（無人）でのバイオメトリクス比較（Automated (Unattended) biometric comparison）顔認識または他の完全に自動化されたアルゴリズム駆動型バイオメトリクス比較などの自動 化バイオメトリクス比較は、現場または遠隔の 身元確認イベントで実行することができる。ID 証拠上の顔画像または他のバイオメトリクス特性（指紋、掌紋、虹彩および網膜パ ターン、声紋、または静脈パターンなど）、または権威ある記録に格納されているものが、生身の 申請者の写真内の顔画像または 身元確認イベント中に申請者から収集された他のバイオメトリ ック・ライブ・サンプルと比較される。

    知識ベース検証（KBV）または知識ベース認証は、ID 検証に使用してはならない（SHALL NOT）。

    身元確認の要件（Identity Proofing Requirements）

    このセクションは規範的なものである。

    このセクションでは、各 IAL での 身元確認の要件など、身元確認および登録サービスを運営する CSP の要件を規定する。このセクションには、連邦機関が独自の ID サービスを運用するか外部 CSP を使用するかに関係なく、連邦機関に対する追加要件も含まれる。

    セクション4.1、4.2、および4.3では、特定の IAL における 身元確認の要件およびガイドラインを提 供する。セクション 4.4には、表 1 に IAL 別のこれらの要件の要約リストが記載されている。

    一般要件

    本セクションの要件は、あらゆる IAL において 身元確認を実施するすべての CSP に適用される。

    アイデンティティ・サービスの文書と記録

    CSP は、定義された IAL を達成するために実施されるすべての身元確認プロセスを詳述する実務記述 書に従って業務を遂行するものとする。実行ステートメントには、最低限以下を含めなければならない：

    1. 各提供保証レベルにおいて CSP が申請者の身元を証明するために従う特定の手順を含む、完全なサービ スの説明；
    2. 利用可能な 身元確認プロセスの種類、指定された IAL に対する証拠および属性収集要件、PII 収集の目的（第 3.1.3.2 節による）、およびバイオメトリクスの収集、使用、および保持（第 3.1.11 節を参照）について、申請者に通知を提供する CSP のポリシー；
    3. 申請者がすべての要件を満たした後、身元確認手続きが適時に完了することを保証する CSP の方針；
    4. CSP がエビデンス強度の要件を満たすために受け入れる ID エヴィデンスの種類；
    5. CSP の ID エビデンスの検証および検証のポリシーおよびプロセス。これには、検証および検 証の責任を持つ要員の研修要件および資格要件、ならびに CSP がエビデンスの検証および検証に 採用する特定の技術が含まれる；
    6. 身元確認プロセスを完了するために必要な ID 証拠を所有していない申請者個人のために、 CSP が 身元確認を完了するための代替プロセス1；
    7. CSP がコア属性と見なす属性、およびそれらの属性の検証に使用する、権威ある信頼できるソース。コア属性には、CSP が ID 解決を実行するために最低限必要な属性のセットのほか、CSP が 身元確認、不正行為の軽減、法律または法的手続きの遵守、または属性アサーションによる依拠当事者（RP）への伝達を目的として収集および検証する追加属性が含まれる；
    8. CSP のポリシーおよび身元確認の誤りに対処するためのプロセス；
    9. RP および影響を受ける個人との連絡を含め、不正の疑いまたは確認されたアカウントを特定し、是正す るための CSP のポリシーおよびプロセス；
    10. サービスの変更（データ・ソース、統合ベンダー、またはバイオメトリック・アルゴリズムの変更など） を管理し、RP に通知する CSP のポリシー；
    11. ユーザの再確認を必要とする条件（アカウントの回復、アカウントの放棄、規制上の「再認証」要件 など）に対する CSP のポリシー；
    12. 定期的な見直しのタイミング、およびプライバシー・リスク評価の更新のトリガーとなる特定の条件を 含む、プライバシー・リスク評価の実施に関する CSP のポリシー（第 3.1.3.1 項を参照）；
    13. 潜在的な衡平性への影響を判断するための評価を実施する CSP の方針。これには、定期的なレ ビューのタイミング、およびサイクル外レビューのトリガーとなる特定の条件が含まれる（第 3.1.4 項を参照）、
    14. すべての個人情報およびセンシティブ・データの収集、目的、保持、保護、および削除に関する CSP のポリシー（CSP が業務を停止した場合、または他の CSP に業務を合併もしくは移管した場合のすべての個人情報の取り扱いを含む）。

    注：800-63C は、連携関係にある IdP、CSP、および RP 間の要件を定義するために、トラスト・アグリーメントの 使用を参照している。CSP プラクティス記述は、これらのアグリーメントに直接含めてもよい（MAY）。

    不正管理

    身元確認プロセスの重要な側面は、ID 管理システムによって保護された給付、サービス、データ、 または資産へのアクセスを得ようとする不正な試みを軽減することである。多くの場合、解決、検証、および検証プロセスにより、多くの攻撃を軽減できる。しかし、絶え間なく変化する脅威環境では、追加的な検査および管理を重ねることで、証明 された ID に対する信頼性を高め、他の管理を破ることを意図した攻撃に対する保護を強化す ることができる。潜在的な不正の事例を特定、検出、および解決する能力は、CSP と RP にとって同様に重要な機能である。

    CSP不正管理

    1. CSP は、不正の識別、検出、調査、報告、および解決機能を提供する不正管理プログラムを確立し、 維持するものとする。このプログラムの具体的な機能および詳細は、CSP 実践ステートメントに文書化されるものとする。
    2. CSP は、すべての不正チェックおよび不正緩和テクノロジーについて、実装前にプライバシー・リスク・アセスメント を実施するものとする。
    3. CSP は、不正行為の被害者または身元確認プロセスへの関与を危うくしようとする試みを受けたと考え る対象者のために、自己報告メカニズムおよび調査機能を確立しなければならないものとする。
    4. CSP は、不合格となった申請者が、権威ある情報源または信頼できる情報源によって確認された自己主張 情報の正確性を推測することを防止するための手段を講じるものとする。 注：これはしばしば「データ洗浄」と呼ばれ、CSP が展開するインタフェースに応じて、多くの方法で防止でき る。そのため、本ガイドラインは、この行為を防止するための特定のメカニズムを指示するものではない。
    5. CSP は、すべての身元確認プロセスについて、以下の不正チェックを実施するものとする：
      • 死亡日チェック- 申請者が死亡していないことを、信頼できる、権威のある、または発行元で確認する。このようなチェックは、合成 ID 詐欺、盗まれた ID 情報の使用、および近親者による搾取の防止に役 立つ。
    6. CSP は、利用可能な ID 検証の種類、選択した技術、証拠、およびユーザベースに基づいて、ID 検証プロセ スに以下の不正チェックを実装すべきであるが、これらに限定されない：
      • SIM スワップ検出- 身元確認プロセスで使用される電話番号が最近新しいユーザまたはデバイスにポ ートされていないことを確認する。このようなチェックは、電話またはデバイスが標的型攻撃によって侵害されたことを示すことができる。
      • デバイスまたはアカウントの存続期間チェック- 電話またはその他のアカウントが、大幅な修正または変更なしに存続し ている期間を評価する。このようなチェックは、身元確認プロセスで使用されるデバイスまたは証拠の信頼性にさらなる確信を与えることができる。
      • トランザクション分析- IP アドレス、ジオロケーション、トランザクション速度など、予 測されるトランザクションの特性を評価し、より高いリスクや詐欺の可能性を示す異 常な行動やアクティビティを特定する。このようなチェックは、ID システムで実行されている特定の攻撃パターンを示すだけでなく、 規模化された自動化された攻撃に対する保護を提供することができる。
      • 不正インジケータ・チェック- 報告された、確認された、または過去の不正イベントなどの記録を評価し、 特定の申請者、申請者のデータ、またはデバイスに関連するリスクが高まってい るかどうかを判断する。このようなチェックは、ID 窃盗または危殆化の兆候を示すことができる。そのような情報が商業プラットフォーム間で収集、集約、または交換され、RP および他の CSP が使用できるようになる場合、利用者はこの慣行を認識しなければならないものとする。これは、連邦 RP にユーザのアクティビティを報告するすべてのウェブサイトにも適用される。
    7. CSPは、不正管理プログラムの一環として知識ベース検証（KBV）を採用してもよい。
    8. CSP は、不正防止機能および判断に不正関連データを考慮する場合、そのデータの最新性を考慮すべきである（SHOULD）。
    9. 有人プルーフィング・プロセスでは、CSP は、不正の指標を検出するためにプルーフィング・エージェントを訓練 し、さらなる処理と調査のために不正の疑いがあるイベントにフラグを立てるツールをプルーフィング・エージェントと信頼 されるレフェリーに提供しなければ ならない（ SHALL ）。
    10. CSP は、不正チェックおよび不正緩和テクノロジーのパフォーマンスを継続的に監視し、プラットフォーム間 または ID サービスが提供する人口統計グループ間でパフォーマンスが異なることに関連する問題を特定し、 改善するものとする。
    11. CSP は、不正の疑いがあり確認されたイベントを RP に伝達する、技術的またはプロセスベースのメカニズ ムを確立しなければならない（SHALL）。
    12. CSP は、NIST SP 800-63C-4 に記載されているように、不正イベントを RP にリアルタイムで伝達するため に、共有シグナリングを実装すべきである（SHOULD）。
    13. CSP は、補償管理として不正行為緩和手段を実装してもよい。これを実施する場合、本ガイドラインの規範的ガイダンスからの逸脱として文書化し、統合前にデジ タル ID アクセプタンス・ステートメントを通じてすべての RP に通知しなければならない （ SHALL）。

    RP不正管理

    1. RP は、CSP が不正データをやり取りする窓口を設置するものとする。
    2. プライバシー・リスク・アセスメントに従い、RP は、不正行為に対抗するため、または他のビジネス・ プロセスをサポートするために、CSP がコア属性として提供するもの以外の追加属性を要求することもでき る（MAY）。
    3. 適用される法規に従い、RP は、CSP およびその他のパートナーが自らの不正行為の特定、軽減、 および報告能力を改善できるように、肯定的な結果も否定的な結果も含め、不正行為の報告および調査 の結果を CSP およびその他のパートナーに伝える仕組みを確立すべきである（SHOULD）。
    4. RP は、その使命、規制環境、システム、アプリケーション、データ、およびリソースと整合 する不正管理プログラムを確立すべきである（SHOULD）。
    5. RP は、潜在的なプライバシー・リスクまたは予期せぬ危害を特定するために、CSP の不正チェック および軽減テクノロジーについてプライバシー・リスク評価を実施しなければならない。連邦機関の RP は、 第 3.1.7 項に記載される要件と整合するように、これを実施するものとする。
    6. RP は、CSP との信頼契約に、不正チェックおよび不正緩和技術に関するあらゆる要件を含めな ければならない（SHALL）。
    7. RP は、CSP の不正管理プログラム、不正チェック、および不正テクノロジーの定期的なレビューを実施し、閾 値を調整し、不正事象の調査をレビューし、不正管理の有効性と効果について判断を下すものとする。
    8. RP は、CSP によって代償的または補足的な管理として導入されたすべての不正緩和手段が、RP の内 部のリスク許容度と受容度に適合しているかどうかを検証しなければならない（SHALL）。RP は、統合前に、CSP の代償的管理策を自身のデジタル ID 受諾表明書に記録するものとす る。

    不正検査不合格の扱い

    不正チェックや軽減技術の有効性は、使用するデータソース、使用する技術、そしておそらく最も重要なことですが、申請者の母集団を含む多くの要因によって異なります。従って、不正管理措置から生じる失敗をどのように処理するかについて、十分に構造化され、文書化されたプロセスを持つことが極めて重要である。これらの失敗の処理には、以下の要件が適用される：

    1. CSP は、自らが実施する各不正検査に関連する閾値およびアクションを設定および文書化し、これらの閾値を RP に提供するものとする。
    2. CSP は、不正チェックおよび軽減技術に関連する問題を申請者が解決できるように、救済手順を確立するものとする（SHALL）。救済の詳細については、[SP800-63] のセクション 3.6 を参照してください。
    3. CSP は、無人リモート・プロセスで不正チェックに失敗したユーザに、信頼されるレフェリー・サービスを提供しなければならない（SHALL）。信頼されるレフリーは、リスク・ベースの意思決定プロセスに情報を提供するために、不正不合格の結果 の要約を提供されるものとする（SHALL）。

    以下のプライバシー要件は、どの IAL でも ID サービスを提供するすべての CSP に適用される。

    プライバシー・リスク評価

    1. CSP は、身元確認および登録に使用されるプロセスについて、プライバシー・リスク評価を実施し、 文書化しなければならないものとする。2最低限、プライバシー・リスク評価は、以下に関連するリスクを評価しなければなら ない：
      1. 身元確認、登録、または不正管理を目的とした、ID 属性、バイオメトリクス、画像、ビデオ、スキャン、 または ID 証拠のコピーを含む、PII の処理；
      2. 本書に規定される必須要件を超えて、CSP が申請者の身元を確認するために取る追加の手順；
      3. 法律または法的手続きに従う場合を除き、身元確認および登録の範囲外の目的で PII を処理する場合；
      4. ID 記録および PII の保存スケジュール；
      5. アルゴリズム（例えば、人工知能または機械学習ツール）によって集計または処理された場 合、個人を識別するために使用される可能性のある非 PII、
      6. CSP に代わってサードパーティ・サービスによって処理される PII。
    2. プライバシー・リスク評価の結果に基づき、CSP は、自身が処理する PII の分離可能性、予測可能性、管理可 能性、機密性、完全性、および可用性を維持するために取る手段を文書化するものとする。3そのような手段を決定する際、CSP は、NIST Privacy Framework [NIST-Privacy]や NIST Special Publication[SP800-53]などの関連ガイダンスおよび基準を適用するものとする。
    3. CSP は、PII の処理に影響を与える ID サービスの変更を行う場合は常に、プライバシー・リスクを再 評価し、プライバシー・リスク評価を更新しなければならないものとする。
    4. CSP は、プライバシー・リスク評価を定期的に見直し、PII の処理に関連する現在のリスクを正確に反映 していることを確認するものとする。
    5. CSP は、そのサービスを利用する組織がプライバシー・リスク評価の要約を利用できるようにしなければ ならない（SHALL）。その要約は、そのような組織がデューデリジェンス調査を行えるよう、十分 な詳細性を持たなければならない（SHALL）。
    6. CSP は、サブスクライバ・アカウントに保持されるすべての個人情報の処理について、プライバシー・ リスク評価を実施するものとする 。 第 5 項。

    その他のプライバシー保護措置

    1. PII の処理は、申請された ID の存在を検証し、申請された ID を申請者に関連付け、不正行為を 軽減し、RP が認可の判断に使用できる属性を RP に提供するために必要な最小限の処理に限定 されなければならない（SHALL）。
    2. CSP は、CSP の ID サービスに関連する機密情報にアクセスできるすべての人員およびサードパーティ・ サービス・プロバイダーに、プライバシー研修を提供するものとする。
    3. CSP は、Sec.3.1.3 のプライバシー要件に従って、ID 解決に必要な場合、属性として社会保障番号（SSN） を収集してもよい（MAY）。SSN を収集する場合、CSP は、SSN データの拡散と保持を制限するために、プライバシー保護技術（完全な属性値ではなく派生した 属性値を送信および受け入れるなど）を実装すべきである（SHOULD）。SSN の知識は、ID の証拠として機能するのに十分ではなく、証拠として使用する場合に社会保障カ ードの所持を検証する許容可能な方法とは見なされない。SSN が連邦、州、または地方の政府機関に代わって収集される場合、CSP は、適用される法律に従って、 収集の通知を申請者に提供しなければならない（SHALL）。
    4. 収集時に、CSP は、身元確認、登録、および不正行為の軽減に必要な PII および属性を収集する目的について、 申請者に明示的な通知を提供するものとする。これには、かかる PII および属性が、身元確認プロセスを完 了するために任意であるか必須であるか、CSP が申請者のその後のサブスクライバ・アカウントに保存する予定 の特定の属性およびその他のセンシティブ・データ、属性を提供しない場合の結果、および記録保持要件がある場合はそ の詳細が含まれる。

    一般資本要件

    衡平性の改善という目標を支援するため、また全体的なリスク評価プロセスの一環として、 CSP はアイデンティティ・サービスの要素を評価し、ある集団の構成員が他の集団と比較して不公 平なアクセス、処置、または結果をもたらす可能性のあるプロセスおよび技術を特定する。衡平性へのリスクが特定された場合、CSP は、人口統計学的グループ間の不一致を低減または解消す る緩和策を積極的に採用する。本文書の第 9 章では、不公平なアクセスまたは結果の対象となる可能性のある 身元確認プロセスおよび技 術の非網羅的リストと、可能な緩和策を示す。

    大統領令[EO13985]、Advancing Racial Equity and Support for Underserved Communities Through the Federal Governmentは、各連邦機関に対し、そのプログラムや政策が、有色人種やその他の十分なサービスを受けていないグループの機会や利益に対する制度的障壁を永続させているかどうか、またどの程度永続させているかを評価するよう求めている。さらに、大統領令[EO13988]「性自認や性的指向に基づく差別の防止と撲滅」は、性自認や性的指向に関係なく、すべての人が敬意を持って扱われるという方針を定め、この性差別禁止を執行するよう各機関に求めている。

    CSP は、OMB 報告書『A Vision for Equitable Data』[EO13985-vision]に概説されている評価、データ収集、 および救済の方法を検討すべきである ：CSP は、衡平性評価方針および慣行の策定にあたり、OMB 報告書『衡平性データのビジョン：衡平性データ作業部会からの提言 』[EO13985-vision]に概説されている評価方法およびデータ収集方法 を検討すべきである。

    以下の要件は、どの IAL でも ID サービスを提供するすべての CSP に適用される：

    1. CDS は、あるグループのメンバーにとって他と比較して不公平なアクセス、処置、または結果を招 く可能性のあるプロセスまたは技術を特定するために、身元確認・プロセスの要素を評価するものとする。
    2. 評価の結果に基づき、CSP は、不公平なアクセス、処置、または結果の可能性を軽減するために講じる措 置を文書化するものとする。
    3. CSP は、衡平なアクセス、処置、または成果に対するリスクを定期的に、および CSP がその ID サービスに変更を加え、それがプロセスまたはテクノロジーに影響を与える場合はいつでも、再評価しなければならないものとする。
    4. CSP は、申請者にこれらのリスク評価への参加を義務付けてはならない。
    5. CSP は、関連する緩和策を含め、エクイティ評価の結果を一般に公開するものとする。

    一般的なセキュリティ要件

    1. 身元確認プロセス内の各オンライン・トランザクションは、第三者が関与するトランザクショ ンを含め、認証された保護されたチャネルを介して行われなければならない（SHALL）。
    2. CSP は、身元確認プロセスに対する自動化された攻撃を防止する手段を実装するものとする。許容可能な手段には、ボット検出、緩和、および管理ソリューション、行動分析4、Web アプリケーション・ファイアウォー ル設定、およびネットワーク・トラフィック分析が含まれるが、これらに限定されない。
    3. 身元確認プロセスの一環として収集されるすべての PII は、静止時のデータの暗号化、認証され た保護されたチャネルを使用した情報交換など、情報の機密性および完全性を維持するために保 護されなければならない（SHALL）。
    4. CSP は、NISTリスク管理フレームワーク [NIST-RMF]または同等のリスク管理ガイドラインに従って、ID サービスの運用に関連するリスクを評価するものとする。少なくとも、CSP は、IAL に関係なく、[SP800-53]中程度のベースラインと一致する適切な管理を適用するものとする。
    5. CSP は、 [SP800-161] Cybersecurity Supply Chain Risk Management Practices for Systems and Organizations (システムおよび組織のためのサイバーセキュリティ・サプライチェーン・リスク管理プラクティス) で規定されるとおり、サードパーティ・サービスの使用に関連するリスクを評価し、適切な管理を適用するものとする。

    救済要件

    1. CSP は、申請者の苦情および身元確認プロセスから発生する問題（証明の失敗、遅延、および困難を 含むがこれらに限定されない）を救済するためのメカニズムを提供するものとする。
    2. これらの仕組みは、申請者が見つけやすく、利用しやすいものでなければならない。
    3. CSP は、苦情または問題の解決を達成する上で有効なメカニズムを評価するものとする。

    救済についての詳細は、[SP800-63]のセクション3.6を参照のこと。

    連邦政府機関に対する追加要件

    以下の要件は、連邦機関が独自の ID サービスを運用するか、ID サービスの一部として外部の CSP を使用するかに関係なく、連邦機関に適用される：

    1. 機関は、プライバシー担当上級機関職員（SAOP）と協議して、身元確認を実施するためのバイオメトリッ クを含む PII の収集がプライバシー法の要件を誘発するかどうかを決定する分析を実施しな ければならない（SHALL）。
    2. 機関は、SAOP と協議して、身元確認を実施するためのバイオメトリクスを含む PII の収集が 2002 年電子政府法（E-Government Act of 2002[E-Gov]）要件を誘発するかどうかを決定する分析を実施しなければならない（SHALL）。
    3. 当該機関は、該当する場合、当該収集を対象とする記録システム通知（SORN）を公表しなけれ ばならない5。
    4. 当該機関は、該当する場合、そのような収集を対象とするプライバシー影響評価（PIA）を公表しなければならない（SHALL）。
    5. 機関は、その機関の多様性、公平性、包括性、およびアクセシビリティ（DEIA）に責任を負 う上級職員、事務所、または統治機関と協議し、身元確認サービスがどのようにすべての対象者 のニーズを満たすことができるかを決定しなければならない（SHALL）。
    6. 機関は、組織内の広報およびコミュニケーションの専門家と相談し、身元確認に関連する要件を 含む新しいプロセスまたは既存のプロセスの更新の展開に伴う広報または一般認識戦 略を策定すべきかどうかを決定すべきである。これには、サービスに関連する技術の使用方法、よくある質問（FAQ）ページ、身元確認プロ セスに参加するための前提条件（必要な証拠など）、ウェビナーまたはその他のライブまたは事前 録画の情報セッション、または ID サービスの採用を支援し、申請者に質問、問題、および フィードバックを伝えるメカニズムを提供するその他のメディアに関する詳細情報を記載 した資料が含まれる。
    7. 省庁がサードパーティの CSP を使用する場合、省庁は、PIA プロセスの一環として、CSP のプライバシ・リスク・アセスメントを省庁のアセスメントのインプットとして使用し、独自のプライバシ・リスク・アセスメントを実施しなければならない（SHALL）。
    8. 省庁が第三者 CSP を使用する場合、当該 CSP による衡平性リスクの評価を、自らの衡平 性リスクの評価に組み込まなければならない（SHALL）。

    確認コードの条件

    このセクションには、確認コードの使用をサポートする CSP の要件が含まれる。

    確認コードは、将来の通信のために、申請者が郵便住所、電子メール・アドレス、または電話番号に アクセスできることを確認するために使用される。また、第 2.5.1 節に記載されているように、IALs 1 および 2 での ID 検証オプションとしても使用される。

    これらの目的のために使用される確認コードは、承認されたランダムビットジェネレータ（[SP800-63B]3.2.12 項を参照）から生成された、少なくとも 10 進数 6 桁（または同等のもの）を含むものとする。確認コードは、手動入力用の数字または英数字（Base64 など）、確認コードの表現を含む安全な（https など）リンク、または確認コードを含む QR コードなどの機械可読の光学ラベルとして提示することができる。

    確認コードの有効期限は、最大で次のとおりとする：

    • 米国本土内の有効な郵便宛先に送付された場合、21日間；
    • 米国本土以外の有効な住所に送付された場合は、30日間；
    • 有効な電話番号（SMSまたは音声）に送信された場合、10分間。
    • 有効なEメールアドレスに送信された場合、24時間。

    その使用時に、CSPは確認コードを無効にしなければならない（SHALL）。

    継続コードの条件

    このセクションには、継続コードの使用をサポートする CSP の要件が含まれる。

    継続コードは、不完全な 身元確認または登録プロセスへの申請者のリンクを再確立するために使用 される。CSP は、申請者が 1 回のセッションで身元確認および CSP の ID サービスへの登録を成功させる ために必要なすべてのステップを完了できない場合、または異なる身元確認タイプ間で切り替え を行う場合（リモート無人からリモート有人など）、継続コードを使用してもよい（MAY）。継続コードは、オフラインで保持し（印刷またはメモなど）、以前の不完全なセッションへのリン ケージを再確立する際に使用するために申請者が安全な場所に保管することを意図している。

    後続のセッションへの申請者の認証を容易にするため、CSP は、最初のセッションが停止する前に、申請者のために確立された記録またはアカウントに認証子を最初にバインドしなければならない（SHALL）。継続コードには、承認されたランダム・ビット・ジェネレータ（[SP800-63B]の Sec.）継続コードは、手動入力のために数字または英数字（例えば、Base64）として提示されてもよいし、継続コードを含むQRコードのような機械可読の光学ラベルとして提示されてもよい（MAY）。

    継続コードの検証は、[SP800-63B]の第3.2.2節で規定されるスロットリング要件に従わなければ ならない(SHALL)。継続コードは、FIPS 承認または NIST 推奨の一方向性関数を使用して、ハッシュ形式で格納されるものとする。継続コードの使用時に、CSP は継続コードを無効にしなければならない（SHALL）。

    身元確認に関する通知の要件

    証明通知は、申請者の有効な住所に送付され、申請者が 身元確認に成功したことを通知し、 身元確認イベントおよびその後の登録に関する情報（受信者が 身元確認の対象であったことを否認する方法など）を提供する。

    以下の要件は、どの IAL でもプルーフィングの通知を送信する CSP および RP に適用される。

    校正の通知：

    1. 有効な郵送先住所、電子メールアドレス、または電話番号に送信されなければならない（SHALL）。
    2. ID サービス名、ID プルーフが完了した日付など、身元確認イベントに関する詳細 を含めなければならない（SHALL）。
    3. 受取人が身元確認イベントを否認する場合に取るべき行動について、連絡先情報を含む明 確な指示を提供しなければならない（SHALL）。
    4. SHALL は、組織または CSP が収集する情報のセ キュリティおよびプライバシーの保護方法、ならびに ID サービスのサブスクライバとして受 信者が負う責任などの追加情報を提供するものとする。
    5. 加入者アカウントへのアクセス方法、または加入者がそのアカウントに 含まれる情報を更新する方法についての情報を提供するべきである[SHOULD]。

    加入者が ID サービスによる身元確認を否認した場合、CSP または RP は、その不正管理プログラム（第 3.1.2 節）に従って対応しなければならない（SHALL）。

    バイオメトリクス使用の要件

    バイオメトリクスとは、指紋、音声パターン、顔の特徴（生物学的特徴）、およびキーストロークのパ ターン、スマートフォンを持つ角度、画面の圧力、タイピング速度、マウスの動き、または歩行（行 動学的特徴）などの生物学的および行動学的特徴に基づいて（ただしこれらに限定されない）、個人を 自動的に認識することである。 本ガイドラインで使用されるバイオメトリクス・データとは、生物学的および行動学的特 徴の、取得、保存、または処理のあらゆる段階におけるアナログまたはデジタル表現を指す。これには、申請者からの生きたバイオメトリクス・サンプル（例：顔画像、指紋）、 および証拠から得たバイオメトリクス参照（例：運転免許証の顔画像、ID カードの指紋ミ ニュティエ・テンプレート）が含まれる。身元確認プロセスに適用されるように、CSP はバイオメトリクスを使用して、個人が ID 証拠の正当な対象者であることを検証したり、個人を新しい ID 証拠またはクレデンシャルに結び付け たり、不正行為を検出したりすることができる。

    以下の要件は、身元確認プロセスの一部としてバイオメトリクスを採用する CSP に適用される：

    1. CSP は、バイオメトリクスのすべての用途、どのようなバイオメトリクス・データが収集される か、どのように保管されるか、どのように保護されるか、および適用される法規制に準拠したバイオメトリクス情 報の削除方法に関する情報について、明確かつ一般に利用可能な情報を提供するものとする。
    2. CSP は、バイオメトリクス情報を収集する前に、すべての申請者から明示的なバイオメトリクス同意を 収集しなければならないものとする。
    3. CSP は、バイオメトリクス使用に対する加入者の同意の記録を保存し、それを加入者のアカウントと関連付けなければならない（SHALL）。
    4. CSP は、すべてのバイオメトリクス情報について、文書化され、一般に利用可能な削除プロセスお よび既定の保管期間を持つものとする。
    5. CSP は、規制、法律、または法令により別途制限されている場合を除き、加入者の要求があれば、 いつでも加入者のすべてのバイオメトリクス情報の削除をサポートするものとする。
    6. CSP は、バイオメトリック・アルゴリズムを、人口統計学的グループ間のパフォーマンスを含め、そのパ フォーマンス特性について、独立したエンティティ（認定ラボラトリーまたは研究機関など）によっ て定期的にテストさせるものとする。最低でも、CSP は、アルゴリズムが更新された後に再試験を受けるものとする。
    7. CSP は、システムの運用環境およびユーザ基盤と実質的に類似した条件下で、採用されたバイオメトリッ ク・テクノロジーの性能および人口統計学的影響を評価するものとする。ユーザ・ベースは、想定されるユーザの人口統計学的特性と、ユーザが使用すると想定される デバイスの両方によって定義される。このような評価に実世界のユーザが含まれる場合、ユーザの参加は任意とする。
    8. CSP は、検証シナリオにおけるバイオメトリクスの使用について、以下の最低パフォーマンス閾値を満たさなければならない（SHALL）：
      • 誤照合率：1:10,000 またはそれ以上。
      • 誤照合率：1:10,000 以上：1:100 またはそれ以上。
    9. CSP は、プライバシー・リスク評価に従って、解決または不正行為の検出をサポートするために 1:N マッチングを使用できるものとする。1:N シナリオでは、CSP は、偽陽性識別の最低パフォーマンス閾値である 1:1,000 を満たすものとする。これは人口統計グループごとに適用され、テストされるものとする。この要件を実証するテストでは、現在または意図される運用規模（N）の 90%以下のギャラリーを使用するものとする。
    10. 解決または不正防止の目的で 1:N バイオメトリクス照合を利用する CSP は、訓練を受けたプルーフ ィング・エージェントまたは信頼できるレフェリーが手動で審査して自動照合結果を確認し、その結果が 誤認識別でないことを確認することなく、ユーザの登録を拒否してはならない（例えば、同じ CSP の異なるアカウントで双子が登録する場合など）。
    11. CSP は、異なる人口統計学的集団（年齢、人種、性別など）の申請者に対して、同様のパフォーマンス特性を 提供するバイオメトリクス技術を採用するものとする。人口統計学的グループ間でパフォーマンスに著しい差が発見された場合、CSP は、影響を 受ける個人に救済手段を提供し、パフォーマンス・ギャップを埋めるために迅速に行動するものとす る。
    12. すべてのバイオメトリクス性能試験は、人口統計学試験を含め、ISO/IEC 19795-1:2021 および ISO/IEC 19795-10:2024 に適合しなければならない（SHALL）。
    13. CSP は、パフォーマンスおよび運用テストの結果を公開するものとする。

    以下の要件は、申請者からバイオメトリクスの特徴を収集する CSP に適用される：

    1. CSP は、バイオメトリクスが申請者から収集されたものであり、他の対象者から収集されたものではないという 合理的な保証を提供する方法で、バイオメトリクスを収集するものとする。
    2. バイオメトリクスをリモートで収集および比較する場合、CSP は、IAPAR パフォーマンス測定基準 <0.15 を満たす提示攻撃検出（PAD）機能を実装して、生身の人間の真の存在を確認し、スプーフィングおよびなりすましの試みを軽減するものとする。
    3. 現場でバイオメトリクスを収集する場合、CSP は、オペレータにバイオメトリクス・ソース（指、顔な ど）に非自然物質が存在しないか確認させ、プルーフィング・プロセスの一環としてそのような検査を実 行するものとする。すべてのバイオメトリクス提示攻撃検出テストは、ISO/IEC 30107-3:2023 に準拠しなければならない。

    証拠検証プロセス（真正性チェック）の要件

    証拠の検証は、リモート光学キャプチャおよび検査（文書認証または doc auth と呼ばれることが多い）によって実施することも、訓練されたプルーフィング・エージェントまたは信頼され た審判の目視検査によって実施することもできる。CSP は、ID 証拠の真正性を評価するために、いずれかまたは両方のプロセスを採用できる。

    以下の要件は、文書の真正性を判定する目的で光学式キャプチャおよび検査を採用する CSP に適用される：

    1. 自動証拠検証技術は、以下の性能指標を満たさなければならない（SHALL）：
      • 文書誤認率（DFAR）が 0.10以下であること。6
      • 文書誤廃棄率（DFRR）が 0.10 以下であること。7
    2. 証拠品に機械可読ゾーン（MRZ）またはバーコードが存在する場合、光学的捕捉および検査は、MRZ データと証拠品に印刷されたデータとの整合性を比較しなければならない（SHALL）。
    3. CSP は、検証プロセス中にドキュメントのライブ・キャプチャを実装するものとする。CSP は、例えば、ドキュメントの存在チェック（ドキュメントの有効性とも呼ばれる）を使用する、またはデバイスの特性を検査して仮想カメラまたはデバイス・エミュレータの存在を判断するなど、ドキュメント画像の注入を防止する技術的な管理を導入するべきである（SHOULD）。
    4. CSP は、システムの運用環境およびユーザ・ベースに実質的に類似した条件下で、採用されている光学 的キャプチャおよび検査技術の性能を評価するものとする。これらのテストは、CSP が光学式キャプチャおよび検査技術を使用した検証を許可する、利用可能なすべての ID 証拠タイプを考慮するものとする。加入者の文書、PII、または画像が試験の一部として使用される場合、それは任意ベースで、加入 者の通知および同意があるものとする。
    5. CSP は、そのエビデンス検証技術を、独立したエンティティ（認定ラボラトリーまたは研究機関な ど）によって定期的に性能特性試験を受けるべきである（SHOULD）。
    6. CSP は、テストの結果を公開するものとする。

    注：これらの要件は、物理的 ID 証拠の画像をキャプチャして検証する技術に適用される。これらの要件は、証拠自体に埋め込まれた PKI またはその他の暗号化技術に依存する検証技術には適用されない。

    以下の要件は、文書の真正性を判定する目的で、訓練を受けたプルーフィング・エージェントまたは信頼 できるレフェリーが証拠を目視検査する CSP に適用される：

    1. プルーフィング・エージェントおよび信頼できるレフェリーは、CSP によって裏付けされたあらゆる形 態の証拠を視覚的に検査するための研修を受け、リソースを提供されなければならない。この研修には以下が含まれるものとする：
      • 証拠の種類の正確なレイアウトおよび地形。
      • 物理的なセキュリティ特徴（浮き出し文字、ホログラフィック特徴、マイクロプリントなど）。
      • 特徴を評価するための技法（使用する道具、触覚検査が必要な場所、特定の特徴を見るための操作など）。
      • 一般的な改ざんの兆候（ラミネートの損傷、画像の修正など）
    2. セッティングがそれを可能にする場合（例えば、現場での立会いプルーフィングイベント）、プルーフ ィングエージェント及び信頼される審判員には、証拠の目視検査をサポートするための専門的な用具 及び機器（例えば、拡大鏡、紫外線ライト、バーコードリーダー）を提供しなければならない（SHALL）。
    3. 遠隔手段による目視検査を行うプルーフィング・エージェント及び信頼されるレフェリーは、提示され た証拠を効果的に検査できるよう、十分に高画質な画像をサポートする機器及びインターネット接続を提 供されなければならない（SHALL）。このような場合、目視検証は、証拠の真正性にさらなる信頼性を与える自動文書検証技術によって支援されるべきで ある（例えば、出席する遠隔セッションの前に証拠を提出し、検証する）。
    4. プルーフィング・エージェントおよび信頼される審判員は、継続的に証拠を目視検査する能力について審査され、少なくとも年1回の評価で評価され、認定されなければならない（SHALL）。

    注：ID 証拠の潜在的な数および組み合わせのため、これらのガイドラインは、セキュリティ機能 の包括的なリストを提供しようとするものではない。CSP は、受け入れる ID エビデ ンスの種類に固有のエビデンス検証トレーニングを提供する必要がある。

    例外とエラー処理

    身元確認プロセス全体を通して、エラーまたは障害が発生する可能性がある多くのポイントがあ る。標準的な 身元確認・ワークフローに対するこのような例外には、ユー ザーが必要な証拠を所持していない場合などのプロセスの失敗、統合サービスが利用で きない場合などの技術的失敗、および申請者がリモート検証ツールを使用するときに ID 証拠の鮮明な画像をキャプチャできない場合などのユーザー・エラーによる失敗がある。

    身元確認サービスのアクセシビリティ、使いやすさ、および公平性を高めるため、CSP は、エラーに対 処し、例外を処理するための運用プロセスを文書化しなければならない（SHALL）。このような文書化されたプロセスには、IAL 1 および 2 の要件を満たすことができない申請者をサポートする、信頼できる推薦者の提供を含めなければならない（SHALL）。さらに、CSP は、申請者の属性、条件、または身元を保証できる申請者参照者の使用をサポートすべきで ある（SHOULD）。

    信頼される審判員の条件

    アクセシビリティを高め、オンライン政府サービスへの平等なアクセスを促進する ために、CSP は信頼された審判を提供する。信頼されるレフリーは、特定の IAL に対する 身元確認の要件を満たすことができない個人の 身元確認および登録を促進するために使用される。このような個人および人口統計学的集団の例を示す非網羅的なリストには、必要な ID 証拠を所持しておらず取得できない個人、障害者、高齢者、ホームレス経験者、オンラ イン・サービスまたはコンピューティング・デバイスにほとんどまたは全くアクセスできない個人、 銀行口座を持たないか信用履歴が限られている人、ID 窃盗の被害者、自然災害で避難し たまたは被災した個人、および 18 歳未満の子どもが含まれる。信頼できる審判の利用には、以下の要件が適用される：

    1. CSP は、信頼されるレフェリー・サービスが利用可能であること、およびそのようなサービスを取得する方法 について、一般ユーザに通知を提供するものとします（SHALL）。
    2. CSP は、Sec.3.1.1 で規定されるとおり、信頼される審判員の使用に関するポリシーおよび手順を、その 実務記述の一部として文書化し、確立しなければならないものとする 。
    3. CSP は、申請者固有の状況に基づいて身元確認が成功するよう、信頼できるレフェリーがリスクに基 づいて判断できるよう研修し、認定するものとする。最低限、このような研修には以下を含めるものとする：
      1. 一般的なテンプレート、セキュリティ機能、レイアウト、地形など、文書の識別および検証。
      2. 破損、改ざん、変更、および材料の種類など、不正文書の指標。
      3. 提示された書類と申請者の照合を行うための顔および画像の比較
      4. 申請者が示す苦痛、混乱、強要などのソーシャル・エンジニアリングの指標
      5. 信頼できるレフリーの能力の年次再認証
    4. CSP は、信頼されるレフリーが関与する 身元確認セッションの記録を作成し、以下を含めるものとする：提示された証拠、完了したプロセス（検証または確認など）、および信頼されるレフリーが使用された理由（自動プロセスの失敗、申請者の要求、確立された例外ポリシーなど）。
    5. CSP は、オンサイト出席セッションまたはリモート出席セッションのいずれかについて、信頼される審判 サービスを提供してもよい（MAY）。これらのセッションは、プルーフィング・イベントの IAL に基づいて、これらのプルーフィ ング・タイプの要件と一致しなければならない（SHALL）。

    信頼できる審判の使用

    信頼されたレフリーは、他の手段で 身元確認を完了できない人にクリティカルパスを提供する。ただし、証明プロセスで発生する可能性のある失敗の数を考慮すると、CSP は、信頼され たリフェリーを独自のサービス提供に適用できる用途を定義することが不可欠である。以下の要件は、身元確認プロセスへの信頼されるレフリーの統合の定義に適用される：

    1. CSP は、どの種類の例外と障害が信頼されるレフェリーの使用に適格であるかを文書化しなければならない （SHALL）。
    2. CSP は、自動化された検証プロセス（例えば、バイオメトリック比較）の失敗に対して、信頼できる審判 サービスを提供しなければならない（SHALL）。
    3. CSP は、コア属性の不一致やレコード・ソースに申請者が存在しない場合など、自動検証プロセスの完了に失敗 した場合の信頼される審判サービスを提供するべきである（SHOULD）。
      1. CSP は、コア属性またはコア属性の変更の裏付けに使用される可能性がある、追加のエビデンス・タイプに 関するポリシーを提供するものとする。
      2. 信頼されるレフリーは、追加的なエビデンスの種類を、そのエビデンスで許容される最大限の程度まで、真正性 を審査するものとする。
      3. 検証を裏付ける権威ある記録または信頼できる記録がない場合、信頼されるレフリーは、 コア属性の一貫性を裏付けるために利用可能な最も強力な証拠と、追加的な証拠の属性を比較してもよい[MAY]。
      4. 権威ある記録とコア属性の部分的な不一致がある場合、信頼されるレフリーは、アサートされた 属性値の正当性を裏付ける証拠（例えば、最近の引っ越しや名前の変更）をレビューしなければならな い（SHALL）。

    申請者の推薦要件

    申請者参照人とは、申請者の身元、属性、または身元確認を完了する能力に関連する状況を保証 するために、申請者の身元確認に参加する個人である。申請者代表者は CSP の代理人ではなく、他の形態の証拠、検証、および確認が利用できない場合に、 身元確認の完了を補助するのに十分な知識を有する申請者の代表者である。

    IAL1またはIAL2で申請者の推薦状を使用する場合は、以下の要件が適用される：

    1. CSP は、申請者の参照の可否、および参照先と申請者の関係に関する要件について、一般に通知するものとする。
    2. CSP は、Sec.3.1.1 に規定されるとおり、実務記述書の一部として、申請者のリファレンスを使用するため の書面によるポリシーおよび手順を確立しなければならない。
    3. CSP は、申請者の参照先を、申請者に意図されたものと同じまたはより高い IAL に身元確認するものとする。CSP は、第3.1.3 節で要求されるとおり、申請者の参照の身元を証明するために収集、記録、および保持さ れる情報を、身元確認申請者のプライバシー・リスク評価に含めなければならないものとする。
    4. CSP は、サブスクライバ・アカウントにおける申請者参照の使用を記録し、申請者参照および申請者との 関係の記録を保持するものとする。
    5. RP は、リスクアセスメントを実施し、プルーフィングイベントの申請者参照を除外または含める ことに関連する適用可能性、業務要件、および潜在的なリスクを判断しなければならない。

    申請者の推薦状の用途

    申請者参照は、身元確認プロセスにおいて申請者をサポートするために、いくつかの異なるアクション を取ることができる。CSP および RP は、トラスト契約において、申請者リファレンスの許容されるすべての用途を定めなけれ ばならない（SHALL）。これには以下を含んでもよい：

    1. 申請者参照は、証拠及び属性の検証プロセスの一環として、申請者に関連する1つ以上の主張された中核的属性を保証してもよい（MAY）。
    2. 申請者参照は、身元確認・プロセスに関連する申請者の特定の状態またはステータ スを保証してもよい（MAY）（ホームレス、災害シナリオなど）。

      注：この情報は、身元確認イベントに関連するリスク判定を支援することを意図している。地位または給付の資格に関連する申請者参照記述の使用は、本ガイドラインの範囲外である。

    3. 十分な身元確認がない場合、申請者の身元を照会者が保証する場合がある。

    すべての場合において、CSPは、申請者のリファレンスがプロセスにおいて果たした役割の記録を作成し、 適用される法的要件および規制要件を裏付けるのに十分なこれらの行動を文書化するものとする。これには以下を含めることができる：

    1. 参考資料申請者の発言や主張を記録すること；
    2. 参照申請者の電子署名、デジタル署名、または物理的な署名を取得すること、
    3. 申請者の参考資料の記載が法的および法的責任に与える影響に関する同意と了解を得ること。

    CSP は、申請者リファレンスとしての参加によって生じる可能性のある法的影響および責任に関 する、明確かつ理解可能な情報を申請者リファレンスに提供するものとする。

    応募者リファレンス関係の確立

    多くの場合、業務上、法律上、または不正防止上の理由により、申請者と申請者照会先との関係を確認する必要があります。リスクアセスメントによりそのような措置が必要と判断された場合、以下の要件が適用されるものとする：

    1. CSP および RP は、申請者の参照関係の確認プロセスに関する要件を確立し、これを信託契約に文書化しなければならない。
    2. CSP は、関係性確認プロセスを開始する前に、申請者の参照先が利用可能な関係性の証拠リストを作成しな ければならないものとする。
    3. CSP は、申請者の関係の証拠（公証された委任状、専門家の証明書など）を要求するものとする。
    4. 申請者の身元確認に成功した場合、CSP は、申請者リファレンスと申請者の関係を確認するために使用され た証拠をサブスクライバ・アカウントに記録するものとする。

    未成年者と接する際の要件

    以下の要件は、あらゆる IAL において未成年者に 身元確認サービスを提供するすべての CSP に適用される。

    1. CSP は、所定の IAL の証拠要件を満たせない可能性のある未成年者の身元確認について、実務記述の 一部として書面によるポリシーおよび手順を確立しなければならない。
    2. 13 歳未満の人と対話する場合、CSPは、1998 年「児童オンライン・プライバシー保護法」（Children's Online Privacy Protection Act of 1998）[COPPA]、または未成年者保護を扱うその他の法律および規制（該当する場合）の遵守を保証するものとする。
    3. CSP は、18 歳未満の個人とやり取りする場合、申請者の参照情報の使用をサポートするものとする。

    加入者IALの向上

    CSP は、RP との高保証トランザクションをサポートするために、加入者アカウントに関連する ID 保証レベルを高めることを加入者に認めるべきである（SHOULD）。これらの機能をサポートする CSP については、以下が適用される：

    1. CSP は、保証レベルを高めるために承認されたアプローチを、実務記述書に文書化しなければなら ない（SHALL）。
    2. CSP は、アップグレード・プロセスを開始する前に、加入者に対し、そのアカウントで利用可能な最高 AAL での認証を要求するものとします（SHALL）。
    3. CSP は、より高い IAL を達成するために義務付けられているとおり、追加エビデンスを収集、検証、および 検証するものとする。

    • RP 固有のサブジェクト識別子を含め、加入者のために確立されたサブジェクト識別子。
    • 以下を含む、加入者のために完了した身元確認手順の記録：

    CSP は、加入者アカウントの情報に対する不正アクセスまたは潜在的な危殆化を報告する機能を、加入者 に提供するものとする。

    加入者アカウントの停止または終了

    以下のいずれかが発生した場合、CSP はサブスクライバのアカウントを速やかに停止または終了するものとする：

    • 加入者は、CSP の加入者アカウントを終了することを選択する。
    • CSP は、サブスクライバのアカウントが侵害されたと判断する。
    • CSP は、サブスクライバが CSP ID サービスへの参加に関するポリシーまたは規則に違反していると判 断する。
    • CSP は、CSP によって確立されたポリシーまたは規則に従って、サブスクライバのアカウントが非アク ティブであると判断する。
    • CSP は、信頼できる情報源からサブスクライバの死亡通知を受信する。
    • CSP は、加入者のアカウントを解約するための法的手段を裁判所から受け取る。
    • CSP は ID システムおよびサービスの運用を停止する。

    CSP は、サブスクライバのアカウントが一時停止または終了された旨の通知をサブスクライバに提供 するものとする。このような通知には、アカウントが一時停止または終了された理由、再アクティベーションまたは更新のオプション、およびアカウントが誤って一時停止または終了されたとサブスクライバが考える場合の救済オプションに関する情報を含めなければならない（SHALL）。

    CSP は、アカウント終了後、そのプラクティスステートメントセクション 3.1.1 に記載されるとおり、記録保持および廃棄要件に従って、サブスクライバのア カウント記録から個人情報または機密情報を削除するものとする 。

    収集とデータの最小化

    これらのガイドラインは、申請された ID を検証し、申請された ID を申請者に関連付け、不正行為を 軽減し、認可の決定に使用できる属性を RP に提供するために必要な PII のみを収集および処 理することを認めている。不必要な PII を収集すると、身元確認サービスに使用されない情報が収集される理由 について混乱が生じる可能性がある。これは、侵襲性または行き過ぎた懸念につながり、申請者の信頼を失う可能性がある。さらに、PII の保持は不正アクセスまたは不正使用に対して脆弱になる可能性がある。データの最小化により、不正アクセスまたは不正使用にさらされる可能性のある PII の量が減り、身元確認プロセスに対する信頼が高まる。

    社会保障番号

    これらのガイドラインは、ID 解決に使用する属性として SSN を CSP が収集することを認めている。しかし、SSN への過度の依存は誤用の一因となり、申請者を ID 窃盗などの危害のリスクにさらす可能性がある。それにもかかわらず、SSN は CSP、特に申請者を機関の記録と関連付けるために SSN を使用する連邦機関にとって、ID 解決を促進する可能性がある。本文書は、属性としての SSN の役割を認識し、その使用について適切な許容を行う。SSN の知識は、ID 証拠として機能するには十分ではない。

    可能な場合、CSP および機関は、身元確認プロセス中の SSN の拡散および露出を制限するメカニズ ムを検討する必要がある。これは、属性検証プロセス中に SSN がサードパーティ・ プロバイダに伝達される場合に特に適切である。可能な限り、プライバシー保護技法および技術を適用して、個人の SSN がサード・パーティ・システムで公開、保存、または維持されるリスクを低減する必要がある。この例としては、サードパーティによって不必要に送信されることを必要とせずに SSN の妥当性を確認するために、属性の主張（たとえば、バリデータからのはい／いいえ応答）を使用すること が考えられる。身元確認プロセスにおけるすべての属性と同様に、処理される各属性の価値およびリスクは、プライ バシ・リスク・アセスメントの対象となり、連邦機関は関連する PIA および SORN 文書でさらに取り 扱うことができる。SSN は、アプリケーションの保証およびリスク・レベルに関連する ID 解 決をサポートするために必要な場合にのみ収集されるべきである。

    通知と同意

    本ガイドラインは、CSP に対し、身元確認トランザクションを完了するためにかかる属性が 任意であるか必須であるかを含め、身元確認に必要な属性の記録を収集および維持する目的、お よび属性を提供しない場合の結果について、収集時に申請者に明示的な通知を提供することを要 求している。

    効果的な通知には、ユーザーエクスペリエンス設計基準や調査、収集によって生じる可能性のあるプライバシーリスクの評価が考慮される。属性が収集される理由、収集された情報が他のデータソースと組み合わされる可能性があることなどを、応募者が理解していると誤って推論することを含め、様々な要素を考慮する必要がある。効果的な告知は、応募者が読んだり理解したりしそうにない、複雑で法律的なプライバシーポリシーや一般的な利用規約へと導くポインタだけでは決してありません。

    さらに、RP は、CSP の選択に利用可能な選択肢、身元確認書類の要件、関連するプライバシー通知、およびサービ スにアクセスする代替手段について、追加ガイダンスを申請者に提供すべきである。

    使用制限

    このガイドラインは、CSP に対して、身元確認、認証、認可、または属性アサーション、関連する不正行為の 軽減、あるいは法律または法的手続きに従うこと以外の目的で属性を処理することから生じ得るプライバシ ー・リスクに見合った、予測可能性（PII および情報システムによるその処理について、個人、所有者、お よび運用者による信頼できる仮定を可能にする）および管理可能性（変更、削除、および選択的な開示を含む PII のきめ細かな管理機能を提供する）という目的を維持するための手段を使用するよう求めている。これらのリスクを管理し、プライバシー・リスク管理の原則をサポートするためのフレームワークは、[NISTIR8062]に記載されている。

    CSP は、加入者への非 ID サービスの提供を含め、属性を処理する様々な事業目的を有する可能 性がある。ただし、対象者に開示された目的以外の目的で属性を処理することは、追加のプライバシ ー・リスクを生じさせる可能性がある。CSP は、追加的な処理から生じるプライバシー・リスクに見合った適切な手段を決定できる。例えば、適用される法律、規制、またはポリシーがない場合、加入者が要求する非アイデンティティ・ サービスを提供するために属性を処理する場合、同意を得る必要はないかもしれないが、通知は、 加入者が処理について信頼できる前提（予測可能性）を維持するのに役立つかもしれない。属性の他の処理は、同意を得るか、または特定の属性の使用または開示について加入者により多くのコ ントロールを認めることを必要とする、異なるプライバシーリスクを伴うかもしれない（管理可 能性）。加入者の同意は意味のあるものである必要がある。したがって、CSP が同意手段を使用する場 合、追加的な使用に対する加入者の承諾を ID サービスの提供条件とすることはできない。

    連邦機関は、提案された処理が許可された処理の範囲外であるかどうか、または適切なプライバシーリスク軽減措置について疑問がある場合、SAOPを参照すべきである。

    リドレス

    ガイドラインは、CSP に対し、申請者の苦情や身元確認から生じる問題を解決するための効果的なメカニズ ムを提供し、申請者がそのメカニズムを見つけ、アクセスしやすくすることを求めている。

    個人情報保護法は、記録システムを保持する連邦政府 CSP に対して、申請者が記録にアクセスし、不正確な場合は修正できるような手続きに従うことを義務付けている。個人情報保護法に関する説明書には、該当する SORN（Sec.3.1.3.を参照）への参照を含めるべきであ る。SORN には、申請者がアクセスまたは訂正を要求する方法が記載されている。連邦政府以外の CSP は、第三者が情報源である場合の連絡先情報など、同等の手順を持つべきである。

    申請者が身元を確認できず、オンライン登録プロセスを完了できない場合、CSP は、プロセスを完了するための代替方法が利用可能であることを申請者に明示するものとする（例：カスタマー・サービス・センターでの対面）。

    注：身元確認プロセスが成功しなかった場合、CSP はその問題に対処する手順を申請者に通知す るべきであるが、登録が失敗した理由の詳細を申請者に通知すべきではない（例えば、「あなたの SSN は、当社が記録しているものと一致しませんでした」と申請者に通知してはならない）。

    プライバシー・リスク評価

    本ガイドラインは、CSP にプライバシー・リスク・アセスメントを実施するよう求めている。プライバシー・リスク・アセスメントの実施において、CSP は以下を考慮する必要がある：

    1. 申請者が取る行動（例えば、追加の検証ステップや記録の保持）が、情報への侵襲性や不正アクセスなど、申請者にとっての問題を引き起こす可能性。
    2. 問題が発生した場合の申請者への影響。CSP は、リスクの受諾、リスクの軽減、およびリスクの共有など、特定されたプライバシー・ リスクに対して取るあらゆる対応を正当化できるものとする。申請者の同意は、リスクを共有する一形態と見なされるため、申請者がこの共有リスクを評価し、受諾する能力があると合理的に期待できる場合にのみ使用されるべきである。

    省庁固有のプライバシー・コンプライアンス

    このガイドラインは、連邦 CSP の具体的なコンプライアンス義務をカバーしている。アイデンティティ・サービス開発の初期段階で省庁の SAOP を関与させ、プライバシー・リス クを評価および軽減し、アイデンティティ証明を実施するための PII 収集が 1974 年プライバシー法［PrivacyAct］または 2002 年電子政府法［E-Government Act of 2002］のプライバシー影響評価（Privacy Impact Assessment）の実施要件の引き金になるかどうかなど、コンプライアンス要件について省庁に助言することが重要である。たとえば、身元確認に関しては、身元確認を実施するために必要な PII またはその他の属性の収集および保守のために、プライバシ ー法要件がトリガされ、新規または既存のプライバシー法記録システム通知（SORN）によ るカバーが必要になる可能性が高い。

    SAOP も同様に、PIA が必要かどうかを判断する際に機関を支援することができる。多くの場合、デジタル ID ライフサイクル全体を包含する PIA および SORN を起草するか、または機関がオンライン・アクセスを確立しているプログラムまたは便益に ついて議論する、より大規模なプログラム的 PIA の一部として 身元確認プロセスを含めることが、最も理にかなって いる。

    デジタル ID ライフサイクルには多くの構成要素があるため、SAOP は個々の構成要素に対する認識と理 解を持つことが重要である。たとえば、プルーフィング・サービスを提供または使用する機関には、データ使用合意、コン ピュータ・マッチング合意など、他のプライバシー成果物が適用される場合がある。SAOP は、どのような追加要件が適用されるかを判断する上で、機関を支援することができる。さらに、デジタル認証の個々のコンポーネントを十分に理解することで、SAOP は、コンプ ライアンス・プロセスまたは他の手段によって、プライバシー・リスクを徹底的に評 価し、軽減することができる。

    ユーザビリティへの配慮

    このセクションは参考になる。

    ユーザー中心設計およびユーザビリティの標準的な用語に合わせるため、本セクション全体を通じて、「ユーザー」という用語は、人間を指すものとして使用される。ほとんどの場合、問題のユーザーは、本ガイドラインの他の箇所で説明されているように、対象者（申請者、請求者、または加入者の役割）になります。

    このセクションは、ID の証明および登録に関連するユーザビリティの考慮事項に対する実装 者の意識を高めることを意図している（典型的な認証機能使用および断続的なイベントに対する ユーザビリティの考慮事項については、[SP800-63B]の第 8 節を参照のこと）。

    [ISO/IEC9241-11]では、ユーザビリティを「システム、製品、またはサービスを、特定のユーザが、特定の使用状況において、有効性、効率性、および満足度をもって、特定の目標を達成するために使用できる程度」と定義している。この定義では、有効性、効率性、および満足度を達成するために必要な要素として、ユーザー、目標、および使用のコンテキストに焦点を当てています。ユーザビリティを達成するためには、これらの重要な要素を考慮した総合的なアプローチが必要です。

    身元確認および登録のユーザビリティの包括的な目標は、ユーザの負担（時間やフラストレーションな ど）および登録の摩擦（完了する手順の数や追跡する情報の量など）を最小限に抑えることで、ユー ザにとってスムーズで肯定的な登録プロセスを促進することである。この目標を達成するために、組織はまず利用者をよく知る必要がある。

    身元確認および登録プロセスは、ユーザと特定の CSP とのやり取り、およびユーザがアク セスするオンライン・サービスの舞台を設定する。否定的な第一印象は、その後のやり取りに対する ユーザの認識に影響を与える可能性があるため、組織はプロセス全体を通じて肯定的なユー ザ・エクスペリエンスを促進する必要がある。

    ユーザビリティは断片的な方法では達成できない。登録および 身元確認プロセスについてユーザビリティ評価を行うことは非常に重要である。ユーザビリティ評価は、代表的なユーザ、現実的な目標およびタスク、および適切な使用状況 を用いて実施することが重要である。登録および 身元確認のプロセスは、ユーザが正しいことをするのが簡単で、間違ったことをする のが難しく、間違ったことが起こったときに回復するのが簡単であるように設計および実装す る。[ISO/IEC9241-11]、[ISO16982]、および[ISO25060]は、ID サービスの全体的な使いやすさを評価する方法と、使いやすさを向上させる ための追加の考慮事項に関する指針を提供している。

    ユーザの視点から見ると、身元確認および登録の 3 つの主なステップは、登録前の準備、登録お よび証明セッション、および登録後の行動である。これらのステップは、1 回のセッションで発生する場合もあれば、各セッションの間 にかなりの時間（たとえば、数日または数週間）が経過する場合もある。

    一般的なユーザビリティと各ステップ特有のユーザビリティについては、以下のサブセクションで説明する。

    ガイドラインと考慮事項は、ユーザーの視点から記述されている。

    1973 年リハビリテーション法第 508 条[第 508 条]は、情報技術における障壁をなくし、連邦政府機関に対し て障害者が電子および情報技術にアクセスできるようにすることを要求するために制定された。これらのガイドラインは、第 508 条からの要件を直接主張するものではないが、ID サービス・ プロバイダは第 508 条の規定に準拠することが期待されている。セクション 508 に準拠するだけでなく、連邦機関およびそのサービス・プロバイダは一般に、ID システムのライフサイクル全体を通じてアクセシビリティが優先されるように、障害者の経験を考慮してサービスおよびシステムを設計することが期待されている。

    身元確認および登録時の一般的なユーザーに関する考慮事項

    このサブセクションでは、登録プロセスのすべてのステップに適用できるユーザビリティに関する考慮事項を示す。各ステップに固有のユーザビリティに関する考慮事項は、第 8.2 節、第 8.3 節、および第 8.4 節に詳述されています。

    • ユーザーのフラストレーションを避けるため、登録に必要なプロセスを合理化し、各ステップをできるだけ明確かつ簡単にする。

    • 技術支援を受ける方法と場所を明確に伝える。例えば、オンライン・セルフサービス・ポータルへのリンク、チャット・セッション、ヘルプデスク・サポートの電話番号など、役立つ情報を提供する。理想的には、外部からの介入なしに、利用者自身が入学準備に関する質問に答えられるよう、十分な情報を提供することである。

    • どのような個人データが収集されるのか、データの収集は任意か否かを 明確に説明すること。さらに、データが誰と共有され、どこに保管され、どのように保護されるかを示す情報を提供すること。

    • 提示されるすべての情報が利用可能であることを確認する。
      • すべてのユーザー向け資料（データ収集のお知らせや記入可能なフォームなど）については、優れた情報デザインの実践に従うこと。
      • 資料は平易な言葉で書き、専門用語は避ける。適切であれば、対象者の識字レベルに合わせて言語を調整する。アクティブボイスと会話形式を使用し、主要な点を論理的に並べ、混乱を避けるために同義語ではなく同じ単語を一貫して使用し、読みやすさを助けるために必要に応じて箇条書き、数字、書式を使用する。
      • フォントのスタイル、サイズ、色、周囲の背景とのコントラス トなど、テキストの読みやすさを考慮する。最もコントラストが高いのは白地に黒です。ユーザーの視力レベルはそれぞれ異なるため、テキストの読みやすさは重要です。読みにくいテキストは、ユーザーの理解ミスや入力ミス（記入可能なフォームへの入力時など）の原因となります。電子資料にはサンセリフ書体を、紙資料にはセリフ書体を使用する。可能な限り、混同しやすい文字（文字「O」と数字「0」など）を明確に区別できないフォントは避ける。これは、確認コードでは特に重要です。文字サイズは、表示サイズに収まる限り、最低12ポイントを使用する。
    • 各ステップについて、代表的なユーザによるユーザビリティ評価を実施します。ユーザビリティ評価のために、現実的な目標とタスク、および適切な使用状況を設定してください。

    入学前の準備

    このセクションでは、登録前の十分な準備を促進する効果的なアプローチについて説明し、 ユーザが困難で苛立たしい登録セッションを回避できるようにする。ユーザが登録セッションに対して可能な限り準備できるようにすることは、身元確認および登録プロセ スの全体的な成功と使いやすさにとって非常に重要である。

    このような準備は、利用者が適切な時間枠で使用可能な形式で必要な情報（必要な文書な ど）を受け取る場合にのみ可能である。これには、どのような ID 証拠が要求されるかを正確に利用者に認識させることが含まれる。利用者は IAL について何も知る必要はなく、必要な ID 証拠が「公正」、「強力」、「優れ ている」とスコア付けされているかどうかも知る必要はないが、組織は特定のシステムへのアクセ スにどのような IAL が必要かを知る必要がある。

    登録手続きを進めるかどうか、セッションに必要なものは何かについて、利用者が十分な情報を得た上で判断できるようにするため、利用者に以下のことを提供する：

    • 各ステップで何を期待するかなど、プロセス全体に関する情報。
      • 利用者がそれに応じて計画を立てられるよう、予想される期間についての明確な説明。

    • 利用者が価値提案を理解できるように、身元確認の必要性とメリットを説明する。

    • 意図された IAL に対する身元確認要件、および受理可能な証拠書類のリスト。

    • 入会金がある場合、その金額と支払い方法。さまざまな支払い方法を提供することで、利用者は好みの支払い方法を選ぶことができる。

    • 利用者の入会セッションが対面式か遠隔チャンネルを介した対面式か、および利用者が選択できるかどうかに関する情報。許可されたセッション・オプションに関連する情報のみを提供する。
      • 場所に関する情報、利用者が希望する場所を選べるかどうか、対面または遠隔チャンネルを介した対面セッションに必要な物流情報。紛失または盗難にさらされる機会が増えるため、利用者は特定の公共の場所（銀行対ス ーパーマーケット）に ID の証拠を持ち込むことを嫌がる場合があることに注意する。
      • 遠隔セッションの技術要件（インターネット・アクセスの要件など）に関する情報。
      • 待ち時間を最小限にするために、対面またはリモート・チャネルを介した対面 身元確認セ ッションの予約を設定するオプション。ウォークインが許可されている場合は、予約なしでは待ち時間が長くなる可能性がある ことを利用者に明確にする。
        • 登録セッションの予約設定、リマインダー、および既存の予約の再スケジュール方法について の明確な指示を提供する。
        • 予約リマインダーを提供し、利用者が希望する予約リマインダーの方法（郵便、ボイスメール、Eメール、テキストメッセージなど）を指定できるようにする。ユーザーは、日付、時間、場所、必要な身元確認の説明などの情報が必要です。
    • 許可され、要求される ID の証拠および属性に関する情報、それぞれの部分が任意であるか 義務であるか、および ID の証拠の完全なセットを提供しない場合の結果。利用者は、ID 証拠の一部（出生証明書上の隆起印など）に固有の要件など、ID 証拠の特定の組 み合わせを知る必要がある。これは、必要な ID 証拠の調達が困難になる可能性があるため、特に重要である。
      • 可能な場合は、必要な ID 証拠を入手しやすくするツールを導入する。
      • 未成年者または独自のニーズを持つ人々に対する特別な要件を利用者に知らせる。たとえば、申請者照会および／または信頼できるレフェリー・プロセスが利用可能かどう か、またそれらのプロセスを利用するために必要な情報を利用者に提供する（第 3.1.13 節を参照）。
      • 書式が必要な場合：
        • フォームが必要な場合：記入可能なフォームを入学前および入学手続時に提供する。利用者がプリンターを利用できるようにする必要はない。
        • フォームが長いと利用者は苛立ちやすく、ミスを犯しやすいので、利用者がフォームに入力する情報量は最小限にする。可能であれば、フォームにあらかじめ情報を入力しておく。

    身元確認と登録

    身元確認および登録に特有のユーザビリティの考慮事項には以下が含まれる：

    • 身元確認セッションの開始時に、利用者に手順を思い出させる。登録前の準備段階で説明した手順を覚えていることを期待しないでください。登録前の準備の直後に登録セッションが行われない場合は、証明および登録段階を完了す るための一般的な時間枠をユーザに明確に思い出させることが特に重要である。
    • ID プルーフ方法（リモートまたはオンサイト無人など）に応じて、ID プルーフプロセ スのステップバイステップのチュートリアルを提供する別のビデオウィンドウを提供する。このようなタイプのチュートリアルまたは例を提供する場合、サービス・プロバイダは、 幅広いユーザをカバーするためにさまざまなサポート・オプションを提供する必要がある。ビデオ・ウィンドウに代わるものとして、口頭または書面による指示がある。
    • 必要な場合は、身元確認の予約の時間または種類を変更するオプションをユーザーに提供 する。
    • 許可され、要求される ID 証拠を記載したチェックリストを提供し、該当する場合は登録 コードなど、登録セッションを進めるために必要な ID 証拠をユーザが持っているこ とを確認する。利用者が完全な ID 証拠一式を持っていない場合は、部分的な 身元確認セッションを完 了するか、信頼できるレフェリーを介して例外処理を使用するか、または必要に応じて 身元確認の例外処理のための申請者レファレンスを使用するかについて、利用者に通知する必要がある。これは、IAL の 身元確認要件を達成するために、ID 証拠の種類およびデータ、 サービス、および検証情報源へのアクセスが容易または容易に利用できない可能性がある国 外の利用者にも適用される。信頼できるレフェリーおよび申請者参照は、例外処理が必要なそのようなタイプの利用者に対し て、代替の 身元確認ワークフローおよびリスクに基づく判断のための機能を提供することを目 的としている。
    • どのような情報が破棄されるか、どのような情報があれば将来のフォローアップ・セッション のために保持されるか、将来のセッションを完了するためにどのような ID の証拠が必 要かについて、利用者に通知する。理想的には、ユーザは部分的な 身元確認セッションを完了したいかどうかを選択できる。
    • 登録セッションの結果に関するユーザの期待値を設定する。これは、以前の ID 検証の経験によってユーザの期待値が左右される可能性があるためである（例： 運転免許証を直接受け取る、パスポートを郵送で受け取る）。

    • 1) ユーザは、登録セッションが成功した時点で直ちに認証機を受け取るのか、2) 認証機を直接受け取るためにフォローアップのアポイントメントを取る必要があるのか、3) またはユーザは認証機を郵送で受け取るのか、受け取る場合はいつ受け取れるのかを明確に示す。

    • 登録セッション中、CSP によってどのようなデータが収集され処理されるかなど、身元確認時にユー ザに明示的な通知を提供するための要件がいくつかある。(通知に関する詳細要件については、第 3.1 節および第 7 節を参照）。CSP は、身元確認、認証、認可、または属性アサーション以外の目的での属性の使用についてユー ザに同意を求めると、ユーザが不快に思う可能性があることを認識する必要がある。ユーザが追加的な収集または使用によってどのような利 益を受けるかを認識していない場合、ユーザは同意を提供したりプロセスを継続したりすることを嫌がったりためらったりする可能性がある。したがって、CSP は、個人情報の追加的な処理によって利用者がどのような利益を得る可能性が あるか、およびそのような処理に関連するリスクを軽減するために CSP が講じる措置について、利用 者に徹底的な説明を提供することが推奨される。さらに、CSP は、追加的な処理をオプトアウトする能力を利用者に提供すべきである。

    • 確認コードが発行される場合
      • 確認コードが発行される場合：利用者に対して、確認コードが発行されること、いつ頃発行される予定であるか、確認コードの有効期間、確認コードの送付方法（郵便、SMS、固定電話、電子メールなど）を事前に通知する。
      • 確認コードがユーザーに届く際には、どのサービスに加入しているのかをユーザーに思い出させ、コードの使用方法とコードの有効期間に関する指示を含める。これは、セクション3.1.8で規定されている有効期間が短いことを考えると、特に重要である。
      • QRコード（Sec.3.1.8参照）のような機械読み取り可能な光学ラベルを発行する場合、QRコード の読み取り機能を取得する方法（例えば、使用可能なQRコードアプリケーション）に関する情報を利用 者に提供する。
      • 登録コードの有効期限が切れたり、使用前に紛失したりした場合は、登録プロセスを 繰り返す必要があることを利用者に知らせる。
      • すべての利用者が公平にテクノロジーにアクセスし利用できるとは限らないため、利用者に代替手段を提供する。例えば、利用者がこの方法を実現するために必要な技術を持っていない場合がある。
    • 登録セッションの終了時：
      • 登録が成功した場合、加入者に、身元確認および登録が成功したことを確認する証明通知（第 3.1.8 節を参照）と、次に取る必要がある手順（いつ、どこで認証機を受け取るのか、いつ郵送されるのかなど） に関する指示を送信する。
      • 登録が部分的に完了した場合（ユーザが完全な ID 証拠一式を持っていない、ユーザがプロセ スの停止を選択した、またはセッションのタイムアウトが原因）、ユーザに通知する：
        • どのような情報が破棄されるか；
        • 今後のフォローアップ・セッションのために保持される情報がある場合は、その情報；
        • 情報の保持期間
        • 将来のセッションにどのような ID 証拠を持参する必要があるか。
      • 登録に失敗した場合は、別の 身元確認および登録オプション（たとえば、リモート証明 を完了できないユーザに対する対面証明など）についての明確な指示をユーザに提供する。

    • 利用者が登録セッション中に認証機を受け取る場合、利用者に認証機の使用および保守に関す る説明書を提供する。例えば、使用方法（特に初回使用時または初期化時に異なる要件がある場合）、認証機の有効 期限に関する情報、認証機の保護方法、認証機が紛失または盗難された場合の対処方法などである。

    • 対面および遠隔の 身元確認には、さらにユーザビリティに関する考慮事項が適用される：
      • 登録セッションの開始時に、オペレータまたは係員はその役割をユーザに説明する 必要がある（オペレータまたは係員が登録セッションを通してユーザを案内するのか、黙って 監視し必要に応じてのみ対話するのかなど）。
      • 登録セッションの開始時に、セッション中に離席してはならないこと、およびセッション中も利用者の行動を監視しなければならないことを利用者に知らせる。
      • 登録セッション中にバイオメトリクスを収集する場合は、収集プロセスを完了する方法について利用 者に明確な指示を与える。指示は、プロセスの直前に行うのが最適である。生身のオペレータによるガイダンスを伴う口頭指示が最も効果的である（例えば、バイオメトリッ ク・センサーがどこにあるか、いつ開始するか、センサーとどのように対話するか、バイオメトリッ ク収集がいつ完了するかを指示する）。
    • リモート 身元確認はオンラインで行われるため、一般的な Web ユーザビリティの原則に従っ てください。たとえば
      • 登録プロセスを通じてユーザを誘導するユーザ・インタフェースを設計する。
      • ユーザーのメモリ負荷を軽減する。
      • インタフェースに一貫性を持たせる。
      • 連続する手順を明確に表示する。
      • 開始点を明確にする。
      • 複数のプラットフォームやデバイスサイズに対応するように設計する。
      • 一貫性があり、見つけやすく、たどりやすいナビゲーションにする。

    入会後

    ポストエンロールメントとは、登録直後で、認証機を最初に使用する前の段階を指す(典型的な 認証機の使用および断続的なイベントに対するユーザビリティの考慮については、[SP800-63B]の セクション10を参照のこと)。上述したように、ユーザは、登録セッションの終了時に、どのような配送(または受け取り) メカニズムで認証機を受け取る予定であるかを既に知らされている。

    入会後に考慮すべきユーザビリティには、以下のようなものがある：

    • ユーザが、認証器が到着するまでの待ち時間を最小限にする。待ち時間が短くなれば、ユーザは情報システムやサービスに迅速にアクセスできるようになる。

    • ユーザが認証子を受け取るために物理的な場所に行く必要があるかどうかを通知する。以前に確認した、予約とリマインダに関するユーザビリティの考慮事項がそのまま適用されます。

    • これには、使用説明書、特に初回使用時または初期化時に異なる要件がある場合、認証機 能の有効期限に関する情報、認証機が紛失または盗難された場合の対処法などが含まれる。

    • ソーシャル・エンジニアリングやフィッシング攻撃など、ID アカウントおよび関連する認証機 関に対する一般的な脅威からユーザを保護する方法について、ユーザに情報を提供する。

    エクイティに関する考察

    このセクションは参考になる。

    本セクションは、第 3.1.4 節で要求されるように、ID サービスを使用する個人の不公平なアクセス、処遇、 または結果に関連するリスクを評価するための指針を CSP および RP に提供することを意図している。このガイダンスは、不公平の対象となる可能性のある 身元確認・プロセスにおける潜在的な領域と、適用可能な緩和策の非網羅的なリストを提供する。CSP および RP は、不公平なアクセス、処置、または結果のリスクが ID サービス内に存在する場 所を検討するための出発点として、このセクションを使用できる。以下の指針は、ID サービスに関連する不公正リスクの決定的で包括的なリストと見なされることを 意図するものではない。

    衡平性リスクの評価において、CSP および RP は、そのオンライン・サービスが提供する全体的なユ ーザー人口を考慮することから始める。さらに、CSP および RP は、その母集団の中で、サービスを利用する際に不公正なアクセス、処遇、ま たは結果を受ける可能性がある、共通の特性を持つ利用者のグループを特定する。CSP および RP は、影響を受ける利用者グループに対する影響を評価することで、緩和策の有効性を 評価することが推奨される。第 8 節で規定されるユーザビリティの考慮事項は、ID サービスを使用するすべての人にとって の全体的なユーザビリティおよび衡平性の改善に役立つ衡平性リスクの緩和を適用する場合にも考慮す べきである。

    大統領令13985[EO13985]「連邦政府を通じた人種的公平性の推進と十分なサービスを受けていないコミュニティへの支援」に従い、OMBは「公平性を評価する方法を特定するための調査」[OMB-Equity]を発表した：大統領への報告書 [OMB-Equity]では、「人種、民族、宗教、所得、地理、性自認、性的指向、障害に 関する衡平性を評価する上で、各省庁を支援するための、適用法に沿った最善の方法」を特定した。CSP および RP は、ID サービスの衡平性を評価するためにどのアプローチおよび方法を使用す るかを決定する際に、この調査を参照することが奨励される。

    リモート 身元確認は、オンライン ID サービスへの登録の使いやすさとアクセシビリティを広 げることができる。以下の小節では、一部のグループおよび個人に不公平な扱いのリスクを生じさせる可能性のある 身元確認プロセスに関する考慮事項を示し、リモート 身元確認に関連するそのようなリスクを軽減するのに役立つ信頼された審判の使用を示す。ただし、信頼される審判の使用は、インターネットにアクセスできない、またはスマートフォ ンやコンピューティング・デバイスに簡単にアクセスできない集団の間に、排除のさらなるリ スクを生じさせないことが重要である。信頼される審判に対面式の選択肢を提供することは、デジタル・デバイドの影響を受けている 人々も CSP または RP が提供するサービスにアクセスできるようにするのに役立つ。

    さらに、CSP および RP は、これらの考慮事項の実施により 身元確認プロセスに遅延が生じ る可能性があるかどうかを評価し、オンライン・スケジューリング・ツールや需要のピーク時用の 追加人員配置などの適切な方法を採用して、これらの遅延を軽減すべきである。

    また、このセクションで提供される考慮事項および緩和策が積極的に採用され、ID サービスがサービ スを提供する人口にとってより公平な 身元確認の経験になることが意図されている。CSP は、サービスのパフォーマンスを継続的に監視し、必要に応じて改善更新を行うことが期待さ れる。これには、サービスの不公平なアクセ ス、処置、または結果に関する利用者の報告を是正するための方針およびプロセスが含まれる。

    説明ID サービス設計では、申請者は欧米の名前形式（姓、名、オプションのミドルネームなど） を使用して名前を入力する必要がある。

    本人確認と公平性

    ID 検証は、身元確認プロセスを受ける申請者と、ID 解決および検証ステップを通じて確立され た有効な現実世界の ID との間の結合を証明することを含む。多くの場合、身元確認イベント中に撮影された申請者の写真（顔画像キャプチャ） を収集し、提示され検証された ID 証拠の一部に含まれる写真と比較する。

    このセクションでは、ID 検証フェーズに関連する不公平な扱いまたは結果について、考えら れる一連の問題と緩和策を示す：

    説明顔画像キャプチャ技術では、特定の肌色や顔の特徴を、比較を実行するのに十分な品質でキャプチャする能力が不足しています。

    可能な緩和策には以下のようなものがある：

    1. さまざまな肌色、顔の特徴、照明状況に対応できることが実証されている、高性能アルゴリズムによる堅牢な画像キャプチャ技術を採用。
    2. 画像キャプチャ技術の運用テストを実施し、民族、人種、出生時の性別、その他の人口統計学的要因の違いを超えて公平に機能するかどうかを判断し、不公平を是正するために必要に応じてアップグレードする。
    3. 画像撮影のための照明や条件の改善方法について、申請者に指導を行う。
    4. 信頼される審判員（3.1.13.1項）など、残留バイアスや技術的限界を補う、リスクに基づく代替プロセ スの提供
    5. 申請者に、最先端の顔認証および生体認証技術を採用したCSP管理キオスクを利用する選択肢を提供する。
    6. 申請者が有人ワークフローのオプションに切り替えられるようにすること

    説明顔画像を含むバイオメトリクス比較の場合、宗教上の目的で着用されている顔の覆いは、 申請者の顔画像を収集する能力を妨げる可能性がある。他のバイオメトリクス特性を含むバイオメトリクス比較の場合、年齢が使用可能な指紋を収集する能力に影響を及ぼすなど、人口統計学的要因が使用可能なバイオメトリクスサンプルを収集する能力を阻害する可能性がある。

    1. 特定の申請者の状況に基づいてリスクに基づく判断を行うことができる、信頼できるレフリーを提供すること（第3.1.13.1項）。
    2. 対面での証明など、本人確認を達成するための代替方法を提供する。
    3. 代替バイオメトリック収集および比較機能を提供。

    1. 独自にテストされたアルゴリズムを使用し、人口統計グループや画像の種類を問わず、一貫したパフォーマンスを実現する。
    2. 残留バイアスと技術的限界を補うための代替プロセスのサポート
    3. 継続的な品質モニタリングと運用テストを実施し、人口統計グループ間のパフォーマンスのばらつきを特定し、必要に応じて是正措置を実施する（アルゴリズムの更新、機械学習など）。

    3. 継続的な品質モニタリングの実施と、偏見、不公平な治療、または結果が確認された場合の是正措置の実施

    ユーザー・エクスペリエンスと公平性

    本文書の「ユーザビリティに関する考慮事項」セクション（第 8 節）は、申請者に円滑で肯定的な ID 証 明体験を提供する方法に関する指針を CSP に提供する。第 8 セクションで提供される具体的な考慮事項に加え、本セクションは、ユーザ体験の公平性を考 慮する際の追加的な考慮事項を CSP に提供する。

    説明必要な技術（例えば、接続されたモバイル機器やコンピュータ）へのアクセスの欠如や、必要な技術を使用することの困難さは、一部のユーザーグループに不当に負担を強いる。

    可能な緩和策には以下のようなものがある：

    1. 身元確認要件を満たすことができる申請者を支援するプロセス・アシスタントの使用 を認める。
    2. 一般に利用可能なデバイス（コンピュータやタブレットなど）の使用を許可し、申請者 が所有していないコンピュータやデバイスで 身元確認プロセスを完了するためのオンラ イン・ヘルプ・リソースを提供する。
    3. 対面での校正オプションの提供
    4. 身元証拠や顔画像のアップロードを簡素化する自動キャプチャなどの技術を採用する。

    説明リモートまたは対面での 身元確認プロセスには、障害者にとって課題がある。

    リモート 身元確認の緩和策として考えられるものは以下のとおりである：

    1. さまざまなニーズや障がいを持つ人々とコミュニケーションを図り、支援するための訓練を受けた信頼できる審判員（第3.1.13.1項）を提供する（例：手話に堪能である）。
    2. 申請者の推薦状の使用を認める（第3.1.13.3項）
    3. 音声指示、スクリーンリーダー、音声認識技術など、アクセシビリティやその他の技術の使用をサポートする。
    4. 身元確認の要件を満たすことができる申請者を支援するために、必要な技術および活動を 使用するプロセス・アシスタントの使用を許可する。

    対面での 身元確認の緩和策として考えられるものは以下のとおりである：

    1. さまざまなニーズや障害を持つ人々とコミュニケーションをとり、支援する訓練を受けたオペレーターの提供（手話が堪能など）
    2. さまざまな高さや角度に調整できる機器やワークステーションを選ぶ
    3. 利便性が高く、ADAアクセシビリティ・ガイドラインに準拠した場所の選定

    用語集

    このセクションは参考になる。

    デジタル ID の領域ではさまざまな用語が使用される。多くの定義は SP 800-63 の以前のバージョンと一貫しているが、この改訂で変更されたものもある。これらの用語の多くには単一の一貫した定義がないため、ここでの用語の定義方法には注意が必 要である。

    申請者
    身元確認および登録のプロセスを受ける対象。
    応募者レファレンス
    申請者の身元、申請者に関連する特定の属性、または個人の状況に関連する条件（緊急事態、ホームレス状態など）を保証できる申請者の代理人。
    公認暗号
    暗号化アルゴリズム、ハッシュ関数、ランダム・ビット・ジェネレータ、または類似の技法で、連邦情報処理標準（FIPS）承認またはNIST推奨のもの。承認されたアルゴリズムおよび技法は、FIPS または NIST 勧告に規定または採用されている。
    アサーション
    サブスクライバの認証イベントに関する情報を含む、IDPからRPへのステートメント。アサーションには、サブスクライバの ID属性を含めることもできる。
    属性
    誰かまたは何かに帰属する品質または特性。ID 属性は、加入者の ID に関する属性である。
    属性検証
    一連の属性が正確であり、現実の ID に関連付けられていることを確認するプロセスまたは行為。検証を参照。
    認証する
    認証を参照。
    認証
    請求者が、加入者アカウントにバインドされた 1 つ以上の認証子を所有し管理していることを証明 し、そのアカウントに関連する加入者であることを証明するプロセス。
    認証保証レベル（AAL）
    認証プロセスの強度を表すカテゴリー。
    認証者
    加入者が所有し管理するもの（暗号モジュールまたはパスワードなど）で、請求者の身元を認証 するために使用されるもの。認証子タイプおよび多要素認証子を参照。
    信憑性
    データがその出所とされる場所に由来するという性質。
    権威筋
    発行元からの正確な情報へのアクセスまたは検証済みコピーを持ち、CSPがその発行元がID プルーフ中に申請者から提供された ID 属性または証拠の妥当性を確認できると高い信頼性を持つエンティティ。発行元は、権威ある情報源である場合もある。多くの場合、権威ある情報源は、身元確認検証段階で使用する前に、機関または CSP の方針決定によって決定される。
    認可する
    通常、対象者の 属性を評価することによって自動化される。
    バイオメトリック参照
    例えば、パスポートにデジタル保存された顔画像、国民IDカードの指紋ミニュチュアテンプレート、話者認識のためのガウス混合モデルなど。
    バイオメトリックサンプル
    指紋画像を含む記録など、バイオメトリクス特徴抽出前のバイオメトリクス特徴のアナログまたはデジタル表現。
    バイオメトリクス
    生物学的または行動的特徴に基づく個人の自動認識。生物学的特徴には、指紋、掌紋、顔の特徴、虹彩・網膜パターン、声紋、静脈パターンが含まれるが、これらに限定されない。行動的特徴には、キーストローク、スマートフォンを持つ角度、画面の圧力、タイピング速度、マウスや携帯電話の動き、ジャイロスコープの位置などが含まれるが、これらに限定されない。
    クレーマー
    1 つまたは複数の認証プロトコルを使用して身元が検証される対象。
    請求先住所
    連絡可能な物理的な場所。個人の居住地の住所が含まれ、郵送先住所が含まれる場合もある。
    クレームドアイデンティティ
    申請者による、検証されていない個人的属性の申告。
    コア属性
    CSPが身元確認に必要であると判断し、文書化した身元属性のセット。
    クレデンシャル・サービス・プロバイダ（CSP）
    信頼されるエンティティ。その機能には、ID サービスへの申請者の 身元確認、および加入者ア カウントへの 認証子の登録が含まれる。CSP は独立した第三者である場合もある。
    信頼できる情報源
    ID の証拠および属性情報の正確性を提供または検証できる実体。信頼できる情報源は、身元確認・プロセスを通じて検証された属性情報、または権威ある情報源にトレースできる属性情報にアクセスできる。または、正確性、一貫性、および最新性についてデータの相関性をチェックされた複数の情報源から得た ID 属性情報を保持する。
    デジタルアイデンティティ
    与えられた文脈の中で、対象を一意に記述する属性または属性の集合。
    デジタル署名
    秘密鍵を使用してデータにデジタル署名し、公開鍵を使用して署名を検証する非対称鍵操作。デジタル署名は、真正性保護、完全性保護、否認防止を提供するが、機密性やリプレイ攻撃保護は提供しない。
    解離性
    システムの運用要件を超え、個人またはデバイスに関連付けることなく、PII またはイベントの処理を可能にする。[NISTIR8062]。
    電子認証
    デジタル認証を参照。
    入会
    CSP/IdPが、身元確認に成功した申請者にサブスクライバ・アカウントを提供し、持続的アクセ スを付与するために認証子をバインドするプロセス。
    エントロピー
    攻撃者が秘密の値を決定する際に直面する不確実性の量。エントロピーは通常ビットで表される。nビットのエントロピーを持つ値は、一様に分布したn ビットの乱数値と同じ不確実性を持つ。
    エクイティ
