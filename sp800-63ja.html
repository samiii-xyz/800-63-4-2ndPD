    ABSTRACT

    これらのガイドラインは、ネットワークを介して政府の情報システムと相互作用するユーザ （職員、請負業者、または個人など）の ID 証明および認証を対象としている。ID プルーフィング、登録、認証子、管理プロセス、認証プロトコル、フェデレーション、 および関連アサーションの各分野における技術要件を定義する。また、有用な提案として意図された技術的推奨およびその他の情報的文章も提供する。このガイドラインは、この目的以外の標準の開発または使用を制約することを意図するものではない。本書は、NIST 特別刊行物（SP）800-63-3 に取って代わるものである。

    序文

    本書およびその付属書である[SP800-63A]、[SP800-63B]、および[SP800-63C] は、デジタル ID サービスを実装するための技術ガイドラインを組織に提供する。

    はじめに

    このセクションは参考になる。

    過去数年間のオンライン・サービスの急速な普及により、信頼性が高く、公平で、安全で、プライ バシを保護するデジタル ID ソリューションの必要性が高まっている。デジタル ID は、オンライン・サービスのコンテキストでは常に一意である。しかし、人は複数のデジタル ID を持つことがあり、デジタル ID はオンライン・サービスのコンテキスト内で固有の特定の意味を伝えるが、デジタル ID の背後にある個人の現実の ID は分からないことがある。オンライン・サービスへのアクセスを提供するために、個人の現実の身元を信頼する必 要がない場合、組織は匿名または仮名のアカウントを使用することができる。その他のすべての使用例では、デジタル ID は、デジタル ID の保持者と、オンライン・サービ スの反対側にいる個人、組織、またはシステムとの間の信頼を示すことを意図している。しかし、このプロセスには課題がある。ミス、ミスコミュニケーション、なりすまし、および他人のデジタル ID を 詐称するその他の攻撃の機会が複数存在する。さらに、個人のニーズ、制約、能力、および嗜好の幅が広いことを考慮すると、オンライン・ サービスは、デジタル機器およびサービスへの広範かつ永続的な参加およびアクセスを確保す るために、公平性、使いやすさ、および柔軟性を考慮して設計されなければならない。

    デジタル ID リスクは動的であり、連続性に沿って存在する。したがって、組織のデジタル ID リスク管理アプローチは、組織固有のニーズを満たすように設計された成果ベースのアプロ ーチを使用してリスクを管理するように努めるべきである。このガイダンスは、ID 関連のリスクに対処しようとする組織に共通のポイントを提供するように設 計された基本管理セットとして機能する特定の保証レベルを定義する。保証レベルは、機関がリスク管理に取り組む際の出発点、異なる機関間の相互運用性をサポ ートするための共通の構造など、複数の利点を提供する。しかし、リスク、脅威、または組織が ID ソリューションを展開するときに直面する考慮事項 の全範囲に包括的に対処できる保証レベルを作成することは非現実的である。このため、本ガイドラインは、コンプライアンス指向のアプローチではなく、デジタル ID ソリューションの実装に対するリスク指向のアプローチを促進し、組織は本ガイドラインで定義されたプロ セスに基づいて制御の実装を調整することが推奨される。

    さらに、デジタル ID に関連するリスクは、オンライン・サービスを提供する組織への潜在的な 影響を超えて広がる。本ガイドラインは、個人、コミュニティ、および他の組織に対するリスクをより確実かつ 明確に考慮するよう努める。組織は、セキュリティを優先するデジタル・アイデンティティの決定が、組織のプログラムおよびサ ービスと相互作用する個人にどのような影響を与えるか、または対応する必要があるかを考慮す べきである。個人のプライバシー、公平性、およびユーザビリティは、セキュリ ティとともに考慮する必要がある。さらに、組織は、デジタル ID のアプローチを、コール・センターや直接の対話で使用されるもの などの ID 管理の他のメカニズムと一緒に考慮する必要がある。ミッションの提供に対して人間中心で継続的に情報に基づいたアプローチを取ることで、 組織は、サービスを提供するさまざまな集団との信頼を徐々に築き、顧客満足度を向上させ、 問題をより迅速に特定し、文化的に適切で効果的な救済手段を個人に提供する機会を得る。

    ID サービスの構成、モデル、および可用性は、SP 800-63 の最初のバージョンがリリースされてか ら大幅に変化しており、多様なユーザ・コミュニティに安全で、プライベートで、使用可能で、公正 なサービスを展開する際の考慮事項や課題も変化している。この改訂は、全体的なデジタル ID モデルの下でエンティティが果たす可能性のある機能に基づいて要件を 明確にすることにより、これらの課題に対処するものである。

    さらに本書は、クレデンシャル・サービス・プロバイダ（CSP）、検証者、および依拠当事者 （RP）向けに、NIST リスク管理フレームワーク ［NISTRMF］およびその構成要素である特別 出版物を補足する指示を提供する。この文書は、組織がデジタル ID サービスを実装するために従うべきリスク管理プロセスを説明し、NIST RMF を拡張して、衡平性とユーザビリティへの配慮をどのように組み込むべきかを概説している。また、企業の業務および資産だけでなく、個人、他の組織、およびより広範に社会への影響 を考慮することの重要性を強調している。さらに、ID 証明、認証、およびフェデレーションのためのデジタル ID 管理プロセスは、通常、 個人情報の処理を伴うため、プライバシー・リスクが生じる可能性がある。そのため、このガイドラインには、潜在的な関連プライバシー・リスクを軽減するのに役立つプライバシ ー要件および考慮事項が含まれている。

    最後に、本ガイドラインは、ネットワークを介してデジタル・システムにアクセスする対象者 のデジタル ID を確立、維持、および認証するための技術的要件および推奨事項を組織に提 供するが、障壁や悪影響に対処し、公平性を育成し、任務目標を成功裏に達成するためには、 情報技術チームの権限外の追加的な支援オプションが必要になる場合がある。

    範囲と適用性

    このガイダンスは、対象者（居住者、ビジネス・パートナー、政府機関など）に関係なく、あるレベル のデジタル ID が必要なすべてのオンライン・サービスに適用される。本書では、「人」は自然人のみを指す。

    本ガイドラインは、公的給付を利用する住民や、コラボレーションスペースにアクセスする民間パートナーなど、外部ユーザーとやりとりする組織サービスを主な対象としている。ただし、職員および請負業者がアクセスする連邦システムにも適用される。連邦職員および請負業者の個人 ID 検証（PIV）標準［FIPS201］および対応する一連の特別 出版物および組織固有の指示は、個人 ID 検証（PIV）カードの発行および管理、派生 PIV クレデンシャルとしての追加認証子のバインド、および PIV システムとのフェデレーション・アーキテクチャおよびプロトコルの使用に関する追加の技術的管理お よびプロセスを提供することによって、連邦エンタープライズ向けにこれらのガイドライン を拡張する。

    このガイダンスの対象とならないオンライン・サービスには、合衆国法律集第 44 編第 3552 条(b)(6)に定義されている国家安全保障システムに関連するものが含まれる。デジタル・プロセスでさまざまなレベルのデジタル ID 保証を必要とする民間組織、州、 地方自治体、および部族政府は、必要に応じてこれらの基準の使用を検討することができる。

    本ガイドラインは、オンラインシステム、サービス、およびアプリケーションへの論理的アクセスに対応するものである。物理的なアクセス制御プロセスについては特に触れていない。ただし、本ガイドラインで規定されるプロセスは、適切な場合、物理的アクセスのユースケースに適用することができる。さらに、本ガイドラインは、機械間認証、相互接続されたデバイス（モノのインターネット（IoT） デバイスなど）、または被験者に代わってアプリケーション・プログラミング・インタフェース（API） にアクセスすることを含むが、これらに限定されないいくつかの主題を明示的に扱っていない。

    このSPスイートの使い方

    これらのガイドラインは、ID 証明、認証、およびフェデレーションの ID システム機能中に発生するエ ラーによる悪影響の軽減を支援する。第 3 節「デジタル・アイデンティティのリスク管理」では、リスク評価プロセスの詳細、およびリスク評 価の結果と追加的な状況から、ID プルーフィング、認証、およびフェデレーション・プロセスを保護す るための制御をどのように選択するかについて説明している。制御は、リスクおよび使命に基づいて、特定のサービスに適用される各タイプのデジタル ID エラーを軽減するために必要な保証レベルを決定することによって選択される。

    具体的には、組織は、実行される各機能に対応する保証レベル1 を個別に選択することが求められる：

    • ID 保証レベル（IAL）とは、ID 証明プロセスを指す。
    • 認証保証レベル（Authentication Assurance Level：AAL）は、認証プロセスを指す。
    • フェデレーション保証レベル（FAL）は、RP がフェデレーション・プロトコルを介して CSP または IdP に接続されている場合のフェデレーション・プロセスを指す。

    SP 800-63は、以下の一連のボリュームとして構成されている：

    • SP 800-63デジタル ID ガイドラインは、デジタル ID モデル、リスク評価方法、および ID 証明、認証、およびフェデレーションの保証レベルを選択するためのプロセスを提供する。SP 800-63 には、規範と参考資料の両方が含まれている。
    • [SP800-63A]：3つの各IALのリソースへのアクセスを希望する申請者の身元証明および登録（リ モートまたは対面）の要件を規定する。SP 800-63A には、規範および参考資料の両方が含まれる。
    • [SP800-63B]は、3 つの AAL のそれぞれで使用される認証子の選択など、認証プロセスに関する要 件を規定している。SP 800-63B には、規範と参考資料の両方が含まれる。

    概要

    SP 800-63 ガイドラインは、現在市場で利用可能な技術とアーキテクチャを反映したデジタル ID モデルを使用している。これらのモデルには、さまざまなエンティティおよび機能があり、複雑さもさまざまである。単純なモデルは、加入者アカウントの作成や属性の提供などの機能を単一のエンティティの下に グループ化する。より複雑なモデルは、これらの機能をより多くのエンティティに分割する。デジタル ID モデルに見られるエンティティおよび関連機能には、以下のものがある：

    サブジェクト：本ガイドラインでは、対象者は個人であり、デジタル ID プロセスのどこにいるかによって 3 つの役割の 1 つで表される。

    • 申請者-身元を証明され、登録される対象。
    • 加入者 - - 身元証明および登録プロセスまたは認証に成功した対象者（すなわち、対象者がアク ティブなオンライン・セッションにいる場合）。
    • 権利主張者 - 認証を受ける資格があると「主張する」対象。

    サービス・プロバイダ：サービス・プロバイダは、クレデンシャル・サービス・プロバイダ、依拠当事者、検証者、 ID プロバイダなど、オンライン・サービスへのアクセス許可および配信に関与するあらゆる機能の 組み合わせを実行できる。

    クレデンシャル・サービス・プロバイダ（CSP）：CSP 機能には、ID サービスへの申請者の ID 証明、および加入者アカウントへの認証子の登録が含まれる。加入者アカウントは、加入者、加入者の属性、および関連認証子に関する CSP の確立された記録である。CSP 機能は、独立したサードパーティによって実行される場合がある。

    依拠当事者（RP）：RP 機能は、通常、デジタル・トランザクションを処理するため、または情報またはシス テムへのアクセスを許可するために、CSP からのサブスクライバ・アカウント内の情報に依存する。フェデレーションを使用する場合、RP は、ID プロバイダからのアサーションを通じてサブスクライバ・アカウントの情報にアクセスする。

    ベリファイア：ベリファイアの機能は、認証プロトコルを使用して、請求者が 1 つまたは複数の認証子 を所有し管理していることを検証することにより、請求者の身元を確認することである。これを行うには、ベリファイアは、認証子とサブスクライバ・アカウントのバインディングを 確認し、サブスクライバ・アカウントがアクティブであることをチェックする必要がある。

    ID プロバイダ（IdP）：フェデレーションを使用する場合、IdP はサブスクライバのプライマリ認証子 を管理し、サブスクライバのアカウントに由来するアサーションを発行する。

    認証機関

    [SP800-63B]は、許可される認証子タイプ、その特性（フィッシング耐性など）、および各 AAL に適した認証プロセスに関する規範的な記述を提供する。

    本ガイダンスでは、認証に使用される 3 種類の認証要素を定義している：

    • 知っていること（パスワードなど）
    • 持っているもの（暗号鍵の入ったデバイスなど）
    • あなたが何者かであること（例：指紋やその他のバイオメトリック特性データ）

    一要素認証は、上記の要素のうち1つだけを必要とし、多くの場合「あなたが知っている何か」を必要とする。同じ要素が複数ある場合も、単一要素認証となる。例えば、ユーザが生成した暗証番号とパスワードは、どちらも "知っている何か "であるため、2つの要素にはなりません。多要素認証（MFA）とは、複数の異なる要素を使用することを指す。

    本ガイダンスでは、認証子には常に秘密が含まれるか、または秘密で構成されることを規定する。認証子に含まれる秘密は、鍵ペア（すなわち非対称暗号鍵）または共有秘密（対称暗号鍵、ワンタイムパス ワード（OTP）を生成するためのシード、パスワードを含む）に基づいている。非対称鍵ペアは、公開鍵と関連する秘密鍵で構成される。秘密鍵は、認証機 器に保存され、認証機を所有し管理する請求者のみが使用できる。サブスクライバの公開鍵（公開鍵証明書など）を持つベリファイアは、認証プロトコルを使用 して、請求者が認証子に含まれる関連秘密鍵を所有し管理しているサブスクライバであることを 検証することができる。対称鍵は一般にランダムに選択され、ネットワークベースの推測攻撃を阻止するのに十分な複雑さと長さを持ち、サブスクライバが管理するハードウェアまたはソフトウェアに格納される。パスワードは通常、暗号鍵よりも文字数が少なく、複雑性が低いため、脆弱性が増大し、これを軽減するために追加の防御が必要となる。

    多要素認証機能の起動要因として使用されるパスワードは、起動秘密と呼ばれる。アクティベーション・シークレットは、認証に使用される保存されたキーの復号化に使用されるか、ローカルに保持され保存されたベリファイアと比較され、認証キーへのアクセスを提供する。いずれの場合も、アクティブ化シークレットは、認証機 関および関連するユーザ・エンドポイント内に残る。アクティブ化秘密の例は、PIV カードをアクティブ化するために使用される PIN である。

    バイオメトリクス特性とは、認証の時点で物理的に存在する人の身元を確認するために使用で きる、固有の個人的属性である。これには、顔の特徴、指紋、および虹彩パターンが含まれるが、これらに限定されない。バイオメトリクス特性は、1 要素認証には使用できないが、物理的な認証子（つまり、持 っているもの）と組み合わせて使用する場合は、多要素認証の認証要素として使用できる。

    対面での対話に使用される認証方法の中には、デジタル認証に直接適用できないものがある。たとえば、物理的な運転免許証は、あなたが持っているものであり、人間（たとえば、警備員）に対して認証するときに役立つかもしれないが、オンライン・サービスの認証手段ではない。

    一般的に使用される認証方法の中には、秘密を含まないか、秘密から構成されるものがあり、こ のため、このガイドラインの下では使用が認められない。例えば

    • 知識ベースの認証では、請求者だけが知っていると思われる質問に回答するよう求 められるが、これはデジタル認証に許容される秘密にはならない。
    • バイオメトリクスの特性は秘密を構成せず、単一要素認証として使用することはできない。

    連盟とアサーション

    規範となる要件は、[SP800-63C]の「フェデレーションとアサーション」に記載されている。

    OMB[M-19-17] Enabling Mission Delivery through Improved Identity, Credential, and Access Managementのセクション III では、政府横断的な ID フェデレーションと相互運用性をサポートするよう各省庁に指示 している。フェデレーションという用語は、異なる信頼ドメイン間で情報を共有するいくつかの異なるアプ ローチに適用できる。これらのアプローチは、ドメイン間で共有される情報の種類によって異なる。このガイドラインは、フェデレーション・アサーションを通じて、一連のネットワーク化されたシ ステム間で信頼協定に基づく ID および認証情報の伝達を可能にするフェデレーション・プロセ スを取り上げている。

    フェデレーテッド・アーキテクチャを使用することには、以下のような多くの利点がある：

    • ユーザーエクスペリエンスの向上（例えば、対象者の身元証明は 1 回で済むが、加入者ア カウントは複数の RP で使用できる）。
    • 加入者（認証者の削減など）と組織（情報技術インフラの削減やアーキテクチャの合理化な ど）の双方にとってのコスト削減。
    • 個人情報を収集、保管、廃棄する必要のないRPにおけるデータの最小化。
    • アカウント値を各アプリケーションにコピーする代わりに、仮名識別子と派生属性値を使用することで、RPに公開されるデータを最小限に抑える。
    • 組織は複雑な ID 管理プロセスに集中するリソースを減らす必要があるため、ミッションの実現が 可能になる。

    一般に、フェデレーション・プロセスは、RP と IdP が共通のセキュリティ・ドメインの下で 共に管理されていない場合の認証に望ましいアプローチであるが、フェデレーションは、アカウント 管理の一元化や技術的統合を含むさまざまな利点のために、単一のセキュリティ・ドメイン 内で適用することもできる。

    SP 800-63 ガイドラインは、組織が選択する ID 証明、認証、およびフェデレーション・アーキテク チャに不可知論的であり、組織は独自の要件に従ってデジタル ID スキームを展開できる。ただし、組織または個々のアプリケーションにローカルな ID サービスを確立するよりも、フェデレーショ ンの方が効率的かつ効果的である可能性がある、組織が遭遇する可能性のあるシナリオがある。以下に、組織がフェデレーションを実行可能なオプションと見なす可能性のあるシナリオを詳 細に示す：

    • 潜在的なユーザは、必要な AAL 以上の認証子をすでに持っている。
    • すべての可能なユーザー・コミュニティをカバーするには、複数のタイプの認証子が必要である。
    • 組織には、加入者アカウントの管理をサポートするのに必要なインフラがない（アカウントの回復、 認証子の発行、ヘルプデスクなど）。
    • RP の実装を変更することなく、プライマリ認証機能の追加やアップグレードを可能にすること が望まれている。
    • フェデレーション・プロトコルはネットワーク・ベースであり、さまざまなプラットフォームや言語での実装が可能であるため、サポートすべき環境はさまざまである。
    • 潜在的なユーザーは複数のコミュニティから集まり、それぞれが既存の ID インフラを持つ。
    • 組織は、アカウントの失効や新しい認証子のバインドを含む、アカウントのライフサイクルを一元管理する機能を必要としている。

    以下のいずれかに該当する場合、組織は連携 ID 属性の受け入れを検討するとよい：

    • 偽名性は、サービスにアクセスする利害関係者にとって必要、必要、実現可能、または重要である。
    • サービスにアクセスするには、定義された属性のリストが必要です。
    • サービスへのアクセスには、少なくとも1つの派生属性値が必要です。
    • 組織は、必要な属性の権威ある情報源や発行元ではない。
    • 属性は、使用中（例えば、アクセス決定を行うため）に一時的に必要とされるだけであり、組織はデータを保持する必要はない。

    デジタル ID リスク管理（DIRM）プロセスは、（1）ID システムによって対処される可能性のあるオンラ イン・サービスに対するリスク、および（2）実装される ID システムによるリスクという 2 つの側面に従っ たリスクの特定と管理に重点を置く。

    リスクの第 1 の次元は、初期の保証レベルの選択に情報を提供し、ID システムによって対処される可 能性のあるオンライン・サービスの危険に関連するリスクを特定しようとするものである。たとえば、以下のようになる：

    • ID の証明：偽者が正当な利用者の ID を使用してサービスにアクセスしたりクレデンシャルを受け取ったりした 場合（たとえば、攻撃者が誰かになりすますことに成功した場合）、どのような悪影響が合理的に予 想されるか。
    • 認証：虚偽の請求者が正当でないアカウントにアクセスした場合、どのような悪影響が合理的に予想される か（例えば、認証機を危殆化したり盗んだりする攻撃者）。
    • 連邦：誤った対象者がオンライン・サービス、システム、またはデータへのアクセスに成功した場合、どのような悪影響が合理的に予想されるか（例えば、アサーションの侵害や再生）。

    この3種類のエラーはすべて、誤った対象者がオンラインサービス、システム、またはデータにアクセスすることを成功させてしまう可能性がある。

    ID システムで対処できるオンライン・サービスの侵害に関連するリスクがあると判断された場 合、初期の保証レベルが選択され、次にリスクの第 2 の次元が検討される。リスクの第 2 の次元は、ID システムによってもたらされるリスクを特定しようとするものであり、 調整プロセスに情報を提供する。テーラリングは、最初に評価した保証レベルを修正し、代償的または補足的な制御を実施し、 継続的な詳細リスク評価に基づいて選択した制御を修正するプロセスを提供する。

    1. 依拠当事者にとって、このプロセスの目的は、オンライン・サービスと、それらのサービスを構成する、 または影響を受けるアプリケーション、トランザクション、およびシステムを保護するために必要な保証レベ ルと調整を決定することである。これは、CSP サービスの選択、開発、および調達に直接貢献する。連邦 RP は、すべてのオンライン・サービスに対して DIRM プロセスを実施するものとする。
    2. クレデンシャル・サービス・プロバイダおよび ID プロバイダにとって、このプロセスの意図は、 定義された保証レベルの要件を満たし、ID システムの危殆化から継続的に保護し、RP の ニーズを満たすサービス提供を設計することである。 サービス提供が規範ガイダンスから逸脱する場合は常に、その逸脱は、サービスを利用する RP に 明確に通知されなければならない。すべての CSP は、提供するサービスに対して DIRM プロセスを実装し、各提供のデジタル ID アクセプタンス・ステートメント（DIAS）をすべての現在または潜在的な RP が利用できるようにしなければならない（ SHALL ）。CSP は、サポートすることを希望する、予想される、または代表的なデジタル ID サービスに基 づいて評価を行なってもよい。このリスク評価を作成する際、CSP は、実世界の RP から、そのユーザ集団および予想され るコンテキストに関する情報を求めるべきである（SHOULD）。

    このプロセスは、連邦情報セキュリティ近代化法（Federal Information Security Modernization Act[FISMA]）が要求するリスク管理プロセスを補強するものである。オンライン・サービスの DIRM 影響評価の結果は、基盤となるアプリケーションまたはシステムの FISMA 影響レベルとは異なる場合がある。ID プロセスの障害は、さまざまなユーザグループに対して異なる影響レベルになる可能性 がある。たとえば、決済システムの全体的に評価された FISMA 影響レベルは、システ ムによって処理される機密性の高い金融データのために「FISMA 中程度」の影響カテゴリになる場 合がある。しかし、永続的なアカウントが確立されていないゲスト決済を行う個人については、 関連データが保持されたりアクセス可能になったりしない可能性があるため、認証および 証明の影響レベルは低くなる可能性がある。省庁の認可担当者は、オンライン・サービスをサポートする基本情報システムの運用権限 （ATO）の一部として、DIRM プロセスの遵守を証明する文書を要求すべきである。省庁の権限付与担当者は、CSP との統合のための調達または ATO プロセスの一環として、DIRM の遵守を証明する文書を CSP に要求すべきである（SHOULD）。

    DIRMのプロセスには5つのステップがある：

    1. オンラインサービスを定義する：出発点として、組織は、オンラインサービスの機能範囲、サービスを提供するユーザグループ、各ユーザグループが利用可能なオンライントランザクションの種類、およびオンラインサービスがそのインタフェースを通じて処理する基礎データという観点から、オンラインサービスの説明を文書化する。オンラインサービスがより広範なビジネスプロセスの一要素である場合、その役割は文書化され、オンラインサービスによって収集・処理されるデータの影響も文書化される。さらに、組織は、オンラインサービス及びその一部である広範なビジネスプロセスによって影響を受けるエンティティを決定する必要がある。その結果、オンラインサービス、そのユーザー、およびその機能によって影響を受ける可能性のあるエンティティが記述される。
    2. 初期影響評価の実施：このステップでは、組織はユーザ集団を評価し、ID システム（ID プルーフ、認証、またはフェデレーショ ン）によって対処される可能性のあるオンライン・サービスの危殆化の影響を評価する。オンライン・サービスの各機能は、定義された一連の危害および影響カテゴリに対して評価される。オンライン・サービスの各ユーザ・グループは、そのユーザ・グループが利用できるトランザクショ ン（すなわち、オンライン・サービスのデータおよび機能に対してそのグループに付与される 許可）に基づいて個別に検討される。このステップの結果は、オンラインサービスの各ユーザグループの影響区分と関連する影響レベル（低、中、高）の文書化されたセットである。
    3. 初期保証レベルの選択このステップでは、影響区分と影響レベルを評価して、オンラインサービスを不正アクセスや不正行為から保護するための初期保証レベルを決定する。保証レベルを使用して、組織は、コンパニオンボリューム[SP800-63A]、[SP800-63B]、及び[SP800-63C]の要求事項に基づき、各ユーザグループの IAL、AAL、及び FAL の基本的な管理策を特定する。このステップの結果は、各ユーザグループに対して、該当する初期 IAL、AAL、及び FAL を特定することである。
    4. 保証レベルの決定を調整し、文書化する：このステップでは、最初に選択した保証レベル及び関連する管理が、プライバシー、衡平性、使いやすさ、及び現在の脅威環境に対する耐性に与える潜在的な影響を判断するために、詳細な評価を実施又は活用する。調整により、当初評価した保証レベルの修正、代償的又は補足的な管理策の特定、又はその両方が生じる可能性がある。すべての評価と最終決定は文書化され、正当化される。その結果、定義され実施可能な保証レベル及びオンラインサービスの最終的な管理策を含むDIAS（3.4.4節参照）が作成される。
    5. 継続的な評価と改善：このステップでは、ID 管理手法のパフォーマンスに関する情報を収集し、評価する。この評価では、ビジネスへの影響、不正率への影響、ユーザ・コミュニティへの影響な ど、さまざまな要因を考慮する。この情報は、選択した保証レベルおよび管理がミッション、ビジネス、セキュリ ティ、および（該当する場合は）プログラム・インテグリティのニーズを満たしているかど うかを判断する上で極めて重要である。また、プライバシーや公平なアクセスに影響する意図せざる損害の監視にも役立つ。進化する脅威の状況を注意深く監視し、それらの脅威に対抗できる、あるいは使いやすさ、公平性、プライバシーを改善できる新しい技術や方法論を調査することによって、改善の機会も考慮すべきである。 このステップの成果は、パフォーマンス測定基準、評価および救済のための文書化された透明な プロセス、および ID 管理手法の継続的な改善である。

    オンラインサービスの定義

    オンライン・サービスを定義する目的は、組織のリスクマネジメントの意思決定に影響を 与える文脈と状況について共通の理解を確立することである。このステップで確認されたコンテキストに富んだ情報は、DIRM プロセスの後続のステップに情報を提供することを意図している。オンラインサービスの役割は、より広範なビジネス環境及び関連プロセスの一部として 文脈化され、その結果、オンラインサービスの機能、ユーザーグループ及びその期待、 処理されるデータ及びその他の関連する詳細について文書化される。

    RP は、少なくとも以下を含むオンラインサービスの説明を作成しなければならない（SHALL）：

    • オンラインサービスによってサポートされる組織の使命と事業目的
    • オンラインサービスに関連するミッションとビジネスパートナーの依存関係
    • オンラインサービスに適用されるプライバシーおよび市民的自由の義務を含む、法律、規制、および契約上の要件
    • オンライン・サービスの機能と、それが処理することが期待される基礎データ
    • オンライン・サービスにアクセスする必要のあるユーザー・グループ、および各ユーザー・グループが利用できるオンライン・トランザクションの種類と権限
    • 機能性、特徴、本人確認および認証オプション、アクセシビリティおよび言語要件、文化的に対応可能な代替コミュニケーションなど、オンライン・サービスに対するユーザーの期待。
    • 既存の DIRA 評価の結果（入力として）、および既存の ID 技術の現状（プルーフィング、 認証、またはフェデレーションなど）。
    • サービスを提供するすべての利用者全体にわたり、身元証明を必要とするサービスの身元証明プロセ スを支援する身元証明書フォームの推定利用可能性。

    さらに、組織は、オンライン・サービスおよびその一部である広範なビジネス・プロセスによって影 響を受けるエンティティを決定する必要がある。デジタル ID システムの障害によって不正なユーザがオンライン・サービスにアクセスした結果、 さまざまなエンティティ、集団、または人口集団に予期しない望ましくない影響が及ぶことを 考慮することが不可欠である。たとえば、攻撃者が発電所を制御するアプリケーショ ンに不正アクセスした場合、悪意のある行為者によって実行された行動は、施設の近くに住む地域住民に壊滅 的な環境影響を及ぼすだけでなく、発電所がサービスを提供する地域の停電を引き起こす可能性がある。

    本文書で説明されているように、ユーザーグループと 影響を受けるエンティ ティを区別することが重要である。オンライン・サービスは、ユーザー・グループに提供される機能の種類に基づいて、いくつかのユーザー・グループに分割される可能性のあるユーザー・セットへのアクセスを許可する。例えば、所得税の申告・審査オンラインサービスでは、(i)個人確定申告の状況を確認する必要がある市民、(2)顧客の代わりに確定申告を行う税務申告者、(3)必要に応じて異なるユーザーグループに権限を割り当てたり、新しいユーザーグループを作成したりするシステム管理者、といったユーザーグループが存在する。対照的に、影響を受ける主体には、オンライン・サービスとその機能によって影響を受ける すべての集団が含まれる。例えば、水処理施設を制御、操作、監視するためのリモートアクセスを可能にするオンラインサービスには、以下の種類の影響を受ける主体が含まれる可能性がある：(1)その水処理施設の水を飲む人々、(2)水処理施設を制御および操作する技術者、(3)施設を所有し運営する組織、(4)施設および適用される規制への遵守を監督する監査人およびその他の職員。

    したがって、影響評価には、組織自身だけでなく、オンラインアプリケーションを使用する個人も含めるものとする。さらに、組織は、ミッション及びビジネス上の必要性に基づいて、特に含める必要のある他のエンティティ（例えば、ミッションパートナー、コミュニティ、及び[SP800-30]で特定されたエンティティ）を特定するべきである（SHOULD）。少なくとも、機関は影響評価を実施する際に、影響を受けるすべてのものを文書化しなければならない（SHALL）。

    このステップのアウトプットは、オンラインサービスが提供する機能によって影響を受けるエンティティのリストを含む、オンラインサービスの文書化された説明である。この情報は、以下のセクションで詳述する影響評価を効果的に適用するための基礎となり、その背景を確立する。

    初期影響評価の実施

    DIRM プロセスのこのステップは、リスクの第 1 の次元（すなわち、ID システムに対するリスク）に対応 し、ID システムによって対処される可能性のあるオンライン・サービスに対するリスクを特定し ようとするものである。

    初期影響評価の目的は、オンライン・サービス特有の ID プルーフィング、認証、およびフェデレーショ ンの失敗がもたらす潜在的な悪影響を特定し、初期保証レベルを得ることである。RP は、このステップを実行するときに、過去のデータおよびユーザ・フォーカス・グループ の結果を考慮すべきである（SHOULD）。 影響評価には、以下を含めるものとする：

    • 一連の影響カテゴリーと、各影響カテゴリーに対する潜在的な危害を特定する、
    • 影響レベルの特定
    • 各ユーザーグループに対する影響度の評価。

    第 3.1 節で特定した各ユーザグループの影響レベルは、そのユーザグループが利用できるトランザク ションに基づいて個別に検討しなければならない（SHALL）。ユーザ・グループを個別に評価することで、組織は各ユーザ・グループに適した ID アプローチおよび保証レベルを選択し、実施する上で最大限の柔軟性を確保できる。

    このアセスメントのアウトプットは、各ユーザグループに対して定義された影響レベル（すなわち、「低」、「中」、「高」）である。これは、最初の保証レベル選定のための主要なインプットとなる。この作業では、組織全体で一貫した適用を促進するために、影響度評価の定義と文書化に重点を置く。

    影響カテゴリーと潜在的危害の特定

    オンラインサービスの初期保証レベルは、最低限、以下の各項目の潜在的な影響を評価することによ って決定されなければならない（SHALL）：

    • ミッション・デリバリーの劣化
    • 信用、地位、評判の毀損
    • 情報への不正アクセス
    • 財務上の損失または財務上の責任
    • 人命の損失、人命の安全、人の健康、環境の健康に対する危険

    組織は、その使命および事業目的に基づき、必要に応じて、追加の影響区分を含めるべ きである（SHOULD）。各影響区分は、組織が提供する様々なオンラインサービスにおいてDIRMプロセスを実施する際に、文書化され、一貫して適用されなければならない。

    害とは、主体が経験するであろうあらゆる悪影響を指す。これらは、影響区分及びそれらがオンラインサービスによって影響を受ける特定の主体にどのように適用され得るかを効果的に理解するための手段を提供する。各影響区分について、機関は、第 3.1 項で特定された影響を受ける主体ごとに、潜在的な危害を考慮しなけれ ばならない。

    各カテゴリーに関連する害の例には、これらに限定されないが、以下が含まれる：

    • 任務遂行の低下：
      • 個人にとっての損害には、受給資格のある政府サービスまたは給付を利用できないことが含まれる。
      • 組織（オンライン・サービスを提供する組織およびオンライン・サービスによって支援される組織を含む）に対する損害には、現在の使命／業務機能を十分なタイミングで、十分な信頼性および／または正確性をもって、あるいは計画されたリソースの制約の範囲内で遂行できないこと、あるいは将来的に使命／業務機能を遂行できないこと、またはその能力が制限されることが含まれる。
    • 信用、地位、評判の毀損：
      • 個人に対する損害には、なりすましの結果としてのイメージや評判への損害が含まれる。
      • 組織に対する損害には、既存の信頼関係、イメージ、評判を損なったり、将来の潜在的な信頼関係を築くことができなくなったりすることによる評判への損害が含まれる。
    • 情報への不正アクセス：
      • 個人に対する危害には、PII またはその他の機密情報の漏えいが含まれ、金銭的損失、生命 の喪失、身体的または精神的傷害、なりすまし、ID の盗用、または持続的な不都合などの二次的危害 をもたらす可能性がある。
      • 組織に対する危害には、知的財産の流出、削除、劣化、暴露、または機密資料や非機密管理情報（CUI）などの他の情報資産の不正な開示が含まれる。
    • 経済的損失または責任：
      • 個人に対する損害には、詐欺やその他の危害の結果として発生した債務や資産の喪失、信用、実際または潜在的な雇用、収入源の損傷や喪失、利用可能な住居の喪失、および／またはその他の経済的損失が含まれる。
      • 組織に対する損害には、詐欺やその他の犯罪行為に関連する費用、資産の損失、評価の低下、事業の損失などが含まれる。
    • 人命の損失、または人の安全、人の健康、環境の健康に対する危険：
      • 個人に対する危害には、死亡、身体的、精神的、または情緒的な健康への損害または喪失、あるいは地域環境の居住不能をもたらす可能性があり、潜在的または実際の損害に対処するために何らかのレベルの介入を必要とする環境の健康への影響が含まれる。
      • 組織に対する損害には、組織の労働力に対する損害や喪失、あるいは、組織が活動できなくなる、あるいは活動能力が低下するような安全でない状況の影響などが含まれる。

    この活動の成果は、セクション3.1で特定された主体への影響を評価するために使用される影響カテゴリーと危害のリストとなる。

    潜在的影響レベルの特定

    デジタル・トランザクションの初期保証レベルは、組織が検討するために選択した影響カテゴリ（第 3.2.1 節から）ごとに、ID システムによって対処される可能性のあるオンライン・サービスの侵害によって生じる潜在的な影響レベ ルを評価することによって決定される。影響レベルは、以下の潜在的影響値のいずれかを使用して割り当てることができる：

    • 低：限定的な悪影響が予想される
    • 中程度：重篤な悪影響が予想される
    • 高い：深刻または壊滅的な悪影響が予想される。

    このステップでは、権限のない個人によるアクセスの影響を、各ユーザーグループ、各影響カ テゴリ、及び各影響対象エンティティについて考慮しなければならない（SHALL）。各カテゴリーにおける潜在的な影響の例を以下に示す。しかし、影響レベルを割り当てるためのより客観的な根拠を提供するために、組織は、各影響カ テゴリーの影響レベルの閾値と例を作成すべきである（SHOULD）。これを行う場合、特に具体的に定量化可能な値を定義する場合、これらの閾値を文書化し、組織 全体の DIRM 評価において一貫して使用し、リスクの共通理解を可能にしなければならない（SHALL）。

    • 任務遂行能力の低下：
      • 低：組織の主要機能の遂行は可能であるが、その有効性が著しく低下するような、限定的なミッション能力の低下をもたらすことが予想される。
      • 中程度：組織が主要機能を果たすことはできるが、その有効性が著しく低下するような、重大なミッション能力の低下をもたらすことが予想される。
      • 高：組織がその主要機能の一つまたはそれ以 上を遂行できなくなるような期間にわ たって、深刻または壊滅的なミッション 能力の低下または喪失をもたらすと予想され る。
    • 信用、地位、評判の毀損：
      • 低：いかなる当事者に対しても、限定的かつ短期的な不便、苦痛、困惑をもたらすと予想される。
      • 中程度：深刻な短期的または限定的な長期的不便、苦痛、または当事者の地位や評判への損害をもたらすと予想される。
      • 高：当事者の地位や評判に深刻または深刻な長期的不便、苦痛、損害をもたらすと予想されるもので、通常、特に深刻な影響を及ぼすか、多くの個人に影響を及ぼす可能性のある状況に限られる。
    • 情報への不正アクセス：
      • 低： [FIPS199]で定義されているように、組織の運営、組織の資産、または個人への悪影響が限定的であると予想される。
      • 中程度：FIPS199]で定義されているように、組織の運営、組織の資産、または個人に深刻な悪影響を及ぼすことが予想される。
      • 高：[FIPS199]で定義されているように、組織の運営、組織の資産、または個人に深刻または壊滅的な悪影響を及ぼすことが予想される。
    • 財務上の損失または財務上の責任：
      • 低：いかなる当事者に対しても、金銭的損失や賠償責任は限定的であると予想される。
      • 中程度：重大な経済的損失または責任をいずれかの当事者にもたらすと予想されるもの。
      • 高：いかなる当事者に対しても、深刻または壊滅的な金銭的損失または責任をもたらすと予想される。
    • 人命の損失、人命の安全、人の健康、環境の健康に対する危険：
      • 軽度：軽度の傷害または急性健康問題が生じると予想されるが、その問題は自然 に解決し、精神衛生上の治療を含む医療措置は必要ない；または、環境衛生への影響につ いては、被害の拡大防止または既存の被害の回復のために、せいぜい限定的な介入が必要で ある。
      • 中程度：軽度の傷害の中程度のリスク、または精神衛生上の治療を含む医療措置を必要とする傷害の限定的なリスク、居住不能期間をもたらし、さらなる被害を防止または既存の被害を回復するための介入を必要とする環境衛生への影響、または複数の低影響事象の複合的な影響をもたらすと予測されるもの。
      • 高：重傷者、外傷者、または死亡者が出ることが予想される場合、長期的または永続的な居住不能をもたらす環境衛生への影響が予想される場合、可能であれば、さらなる被害の防止または既存の被害の回復のために重大な介入を必要とする場合、または複数の中程度の影響事象の複合的な影響が予想される場合。

    本ガイダンスでは、3つの影響レベルを提示している。しかし、各省庁は、初期影響評価活動のために、より詳細な影響レベルを定義し、独自の方法論を開発してもよい。

    影響分析

    影響分析では、ID システム機能（すなわち、ID 証明、認証、およびフェデレーション）によって対処 される可能性のあるオンライン・サービスの侵害の影響レベル（すなわち、低、中、または高）を検 討する。影響分析では、以下の次元を考慮する：

    • ユーザーグループ第3.1項
    • 影響を受ける事業体第3.1項
    • 影響カテゴリー第3.2.1項
    • 影響レベル第3.2.2項

    どのような主体についても、与えられた影響カテゴリーに危害や影響がない場合、影響レベルは「なし」とマークすることができる。

    各ユーザーグループについて、影響度分析は、影響を受けるエンティティの種類ごとに、各影響 カテゴリーの影響度を検討しなければならない（SHALL）。各ユーザグループには異なるトランザクションセットが利用可能であるため、この分析では各ユーザ グループを個別に検討することが重要である。

    例えば、水処理施設の制御・運用・監視を可能にするオンラインサービスの場合、オンラインサービスを通じてそのユーザーグループが利用可能なトランザクションに基づいて、各ユーザーグループ（例えば、施設を制御・運用する技術者、監査・監視担当者、システム管理者など）を個別に検討する。言い換えれば、影響分析では、悪質な行為者がユーザーグループのメンバーとしてオンラインサービスに不正アクセスし、何らかの悪質な行為を行ったかどうか、また、考慮される影響カテゴリーごとに、様々な影響を受ける主体（例えば、水を飲む市民、施設を所有する組織、監査人、監視関係者など）に対する影響のレベル（すなわち、低、中、高）を判断しようとします。

    影響分析は、オンライン・サービスにアクセスできるユーザ・グループごとに実行しなければなら ない（SHALL）。 各影響カテゴリについて、ID 管理機能の障害によってオンラインサービスが侵害された結果、影 響を受けるエンティティごとに影響レベルが推定される。

    この影響度分析の出力は、各ユーザーグループに対する影響度のセットであり、 以下の次 のサブセクションに従ってさらなる分析を行うために、適切な形式で文書化さ れなければな らない（SHALL） 。

    各ユーザーグループの複合的な影響レベルを決定する

    前のステップから生成された各ユーザ・グループの影響アセスメント・レベルの結果が組み合わ され、そのユーザ・グループに対する単一の影響レベルが確立される。この単一の影響レベルは、そのユーザ・グループの ID 証明、認証、および／またはフェデレーショ ン機能の侵害によって生じる、影響を受けるエンティティに対するリスクを表す。

    組織は、この組み合わせ分析に様々な方法を適用して、各ユーザーグループに対する効果的な影響レベルを決定することができる。いくつかのオプションがある：

    • 様々な影響カテゴリーと影響を受ける事業体にわたるハイウォーターマーク・アプローチを使用し、実効影響レベルを導き出す。
    • 異なる影響カテゴリーおよび／または影響を受ける主体に異なる重みを割り当て、平均をとって実効影響レベルを導き出す。
    • 組織の使命と優先事項に沿った、その他の組み合わせ論理

    組織は、影響度評価をその定義された利用者グループごとに総合的な影響度スコアにまとめるために使用するアプローチを文書化し、そのオンラインサービスすべてに一貫して適用しなければ ならない。組合せ分析が終了した時点で、組織は、各利用者グループに対する影響を文書化しなければならない。

    このステップの結果は、ID 管理システム機能（すなわち、ID 証明、認証、フェデレーション） の危殆化による各ユーザ・グループの実効影響レベルである。

    初期保証レベルとベースライン・コントロールの選択

    最後のステップの初期影響分析によって、各ユーザ・グループの ID 証明、認証、およびフェデレーショ ンの初期保証レベルを選択するプロセスの主要な入力となる有効影響レベル（すなわち、低、 中、または高）が得られる。

    初期保証レベルの目的は、付属書[SP800-63A]、[SP800-63B]、および[SP800-63C]の要件およびガイドラインから、各 ID 管理機能の基本デジタル ID 管理（プロセスおよび技術要素を含む）を特定することである。

    選択されたデジタル ID 管理およびプロセスの初期セットは、ID 管理システムによって発生する 可能性のあるリスクに基づいてステップ 4 で評価され、調整される。

    保証レベル

    オンライン・サービスの機能と展開されたアーキテクチャに応じて、ID 管理機能（ID プルーフ、認証、およびフェデレーション）の 1 つ以上のサポートが必要になる場合がある。これらの機能の強度は、保証レベルで記述される。RP は、オンライン・サービスに適用される保証レベルのタイプを以下から特定しなけれ ばならない（SHALL）：

    • IAL：個人の身元を決定するための身元証明プロセスの堅牢性。IAL は、潜在的な身元証明の失敗から生じるリスクを軽減するために選択される。
    • AAL： 認証プロセス自体の堅牢性、および認証子と特定の個人の識別子との間のバインディング。AAL は、潜在的な認証の失敗から生じるリスクを軽減するために選択される。
    • FAL：IdP から RP に認証情報および属性情報を伝達するために使用されるフェデレーションプロ セスの堅牢性。FAL は、潜在的なフェデレーションの失敗から生じるリスクを軽減するために選択される。

    アイデンティティ保証レベル

    • IAL1： クレームされた ID の現実世界の存在を裏付ける。コア属性は、ID の証拠から取得するか、申請者が主張する。すべてのコア属性は、権威のある情報源または信頼できる情報源に対して検証され、その 属性を身元証明プロセスを受ける人物に結び付けるための手順が踏まれている。

    • IAL2：IAL2は、追加的な証拠の収集と、証拠を検証し身元を確認するためのより厳格なプロセスを要求することにより、厳密性を増している。

    • IAL3：IAL3 では、訓練を受けた CSP 担当者（すなわち、プルーフィング・エージェント） が、少なくとも 1 つのバイオメトリックの収集と同様に、オンサイトの立会 ID プルーフ ィング・セッションの一環として申請者と直接対話する要件が追加される。

    • FAL1：FAL1 により、サブスクライバは、フェデレーション・プロトコルの IdP からのアサーション を使用して、RP に対して認証を行うことができる。FAL1 は、アサーションが特定の IdP からのものであり、特定の RP を意図したものであることを保証する。

    • FAL2：FAL2 は、さらに、フェデレーション・トランザクションの前に IdP と RP の信頼契約が確立されること、および RP が攻撃者からのアサーションの注入に対して強固な防御を持つことを要求する。

    • FAL3：FAL3 はさらに、サブスクライバがバインドされた認証子を使用して RP に直接認証し、IdP からのアサーションを提示することを要求する。さらに、IdP と RP は、多くの場合手動である高度に信頼されたプロセスを通じて、互いの ID と暗号鍵資料を確立する。

    初期保証レベルの選択

    各ユーザ・グループに対する全体的な影響レベルは、評価対象の組織のオンライン・サービスのデジ タル ID 機能に対する初期保証レベルおよび関連する技術的およびプロセス制御の選択の基礎と して使用される。これらの初期保証レベルおよび管理の選択は、主に、権限のないエンティティがオンライン・サ ービスにアクセスできるようにするデジタル ID 機能内の障害から生じる影響に基づいている。初期の保証レベルおよび管理は、DIRM プロセスの次のステップでさらに評価され、必要に応じて調整 される。

    組織は、デジタル ID アプローチにおける障害の潜在的な影響に基づいて初期保証レベルおよび管理 を選択するためのプロセスおよびガバナンス・モデルを開発し、文書化しなければならない（SHALL）。このセクションでは、そのプロセスに含める主な要素についてガイダンスを示す。

    オンライン・サービス・プロバイダは、そのアプリケーションを保護するために適切な xAL を評価し、決定しなければならないが、これらの保証レベルの選択は、オンライン・サービス・プロバイダが独自に制御を実装しなけ ればならないことを意味するものではない。オンライン・サービス・プロバイダが実装を選択した ID モデルに基づいて、保証レベ ルの一部またはすべてをサードパーティ CSP や IdP などの外部エンティティが実装することができる。

    イニシャルIALの選択

    初期保証レベルを選択する前に、RP は、オンライン・サービスの利用者に ID プルーフが必要かどうかを判断する必要がある。オンライン・サービスがデジタル・トランザクションの実行に個人情報を必要としない場合、 ID の証明は必要ない。個人情報が必要な場合、RP は、検証された属性が必要か、自己主張の属性が許容される かを判断する必要がある。自己主張属性を受け入れることによる潜在的な害が重要でない場合、システムは ID プルーフなしで動作することもできる。このような場合、[SP800-63A]に記述されている ID プルーフ処理は、システムには適用されない。

    • 低いインパクトIAL1
    • 中程度の影響IAL2
    • 高いインパクトIAL3

    組織は、自己のアプリケーションに ID プルーフが必要かどうかを文書化し、必要な場合は、 第 3.2.4 節の実効影響レベルの決定に基づいて、各ユーザーグループの初期 IAL を選択しなければ ならない（SHALL）。

    IAL は、申請者が主張された現実の ID を保持していることの保証レベルを反映する。最初の選択は、ID 証明プロセスにおける失敗の潜在的な影響がより高い保証プロセスによって 軽減されるはずであることを前提としている。

    初期AALの選択

    すべてのオンライン・サービスに認証が必要なわけではない。公開情報へのアクセスを提供し、サブスクライバ・アカウントを利用しないオンライン・ サービスは、必ずしも認証メカニズムを実装する必要はない。しかし、個人情報、保護された情報、または加入者アカウントへのアクセスを提供 するオンライン・サービスでは、認証が必要である。これらのガイドラインで義務付けられている影響評価に加え、認証保証レベルおよび認証メカニズ ムの適用に関する決定を行う場合、組織は、オンライン・サービスを管理する法律、規制、ま たはポリシー要件を考慮することが重要である。たとえば、[EO13681]は、「デジタル・アプリケーションを通じて市民が個人データにアクセス できるようにするすべての組織は、複数の認証要素を使用する必要がある」と述べており、こ の基準を満たすアプリケーションには、最低でも AAL2 を選択する必要がある。

    オンライン・サービスに認証機能を実装する必要がある場合、初期AALは、以下のような単純なマッピング・プロセスによって選択される：

    • 影響が少ない：AAL1
    • 中程度の影響AAL2
    • 高いインパクト：AAL3

    組織は、オンラインサービスに認証が必要かどうかを文書化し、認証が必要な場合 は、 セクション3.2.4の実効影響レベル決定に基づいて、各ユーザーグループの初期AALを選 択しなければならない（SHALL）。

    AAL は、請求者がクレデンシャルまたは認証子が発行されたのと同じ個人であることを保証するレベ ルを反映する。最初の選択では、認証プロセスで失敗した場合の潜在的な影響は、より高い保証プロセスによって 軽減されるはずであると想定している。

    イニシャルFALの選択

    ID フェデレーションは、冗長でコストがかかり、多くの場合時間のかかる ID プロセスを回避す る便利なユーザ体験など、多くの利点をもたらす。汎用 IdP モデルまたはサブスクライバ制御ウォレット・モデルによるフェデレーションの利点は、[SP800-63C]の第 5 節で説明されている。ただし、すべてのオンライン・サービスが、リスクに基づく理由であれ、法律または規制要件に よるものであれ、フェデレーションを利用できるわけではない。M-19-17]と同様に、オンラインサービスを運営する連邦機関は、ユーザーアクセスのオプションとしてフェデレーションを実装すべきである（SHOULD）。

    オンライン・サービスがIDフェデレーションを実装している場合、初期FALは以下のような単純なマッピング・プロセスによって選択される：

    • 低いインパクトFAL1
    • 中程度の影響FAL2
    • 高いインパクト：FAL3

    組織は、そのオンラインサービスにフェデレーションが使用されるかどうかを文書化し、使用される場合は、 第 3.2.4 項の実効影響レベルの決定に基づいて、各ユーザーグループの初期 FAL を選択しなければ ならない（SHALL）。

    FAL は、認証プロセスの結果および関連 ID 情報を RP オンライン・サービスに伝える ID アサーションの保証レベルを反映する。予備的な選択では、連携 ID アーキテクチャにおける失敗の潜在的な影響は、より高い保証プロセ スによって軽減されるはずであると想定している。

    ベースライン・コントロールの特定

    該当する各 ID 機能（すなわち、IAL、AAL、および FAL）の初期保証レベルの選択は、付属文書[SP800-63A]、[SP800-63B]、および[SP800-63C] のガイドラインからベースライン・デジタル ID 管理を選択するための基礎となる。第 3.4 節で説明するように、ベースライン管理には、追加の潜在的影響に対して評価される技術お よびプロセス管理が含まれる。

    このステップの出力には、以下のように、各ユーザーグループに関連するxALとコン トロールを含めなければならない[SHALL]：

    • SP800-63A]の初期IALと関連技術およびプロセス管理
    • SP800-63B]の初期AALと関連技術およびプロセス管理
    • SP800-63C]の初期FALと関連技術およびプロセス管理

    保証レベルの調整と文書化

    デジタル ID リスク管理プロセスで扱うリスクの第 2 の側面は、ID 管理システムから生じるリスクに 焦点を当てる。これらのリスクは、テーラリング・プロセスに情報を提供し、第 3.3.4 節の xALs の初期選択および関連する技術的・プロセス的管理から生じるリスクと予期せぬ結果を特定しようとする。

    テーラリングは、最初に評価された保証レベルを修正し、継続的な詳細なリスク評価に基づ いて補償的又は補足的な管理策を実施するプロセスを提供する。これは、柔軟性を確保するための道筋を提供し、組織がそれぞれの状況、利用者及び脅威環境に合致したリスク管理目標を達成することを可能にする。このプロセスは、予期せぬリスクと公平性、プライバシー、ユーザビリティへの影響、 及び特定の環境脅威の評価に重点を置く。特定のリスク領域や結果を優先するものではない。組織の成果を達成するために、様々な種類のリスクのバランスをとる決定を行うことは、依然として組織の責任である。 組織は、特定の状況、ユーザ、および脅威環境にデジタル ID 管理を合わせる目的で、テーラリ ングを採用するべきである（SHOULD）。

    調整ステップの中で、組織は、社会から疎外された集団または歴史的に十分なサービスを受けていない 集団に不釣り合いな影響をもたらす ID 管理の実施によるミッションの提供への影響に重点を置かなけ ればならない。組織は、特定の意図された対象者がオンライン・サービスにアクセスできない可能性だけでなく、 ID 管理の結果として経験する負担、不満、および摩擦も考慮しなければならない。

    調整プロセスの一環として、組織は、使用する、または使用する予定の CSP および IdP の影響評価 文書および実施声明1 をレビューするものとする。ただし、組織は、組織の特定の使命およびオンライン・サービスが提供するコミュニティが、 調整の目的で十分に考慮されるように、独自の分析も実施しなければならない（SHALL）。その結果、組織は、選択した CSP に対し、組織の使命およびサービスを提供するコミュニティに対する リスクおよび予期せぬ影響に対処するために、特定の管理の実施における強化またはオプションの 提供を求めることができる。

    組織間の相互運用性と一貫性を促進するため、サードパーティ CSP は、本文書の規範的なガイダンスと 一貫した（評価済みまたは調整済みの）xAL を実装すべきである（SHOULD）。ただし、本ガイドラインは、特定のミッション・ニーズを満たし、独自のリスク選好に対応し、安全でアクセシブルなオンライン・サービスを提供するために、組織が最初の xAL および関連コントロールを調整できるように柔軟性を提供する。そうすることで、CSP は、本ガイドラインの規範的な記述とは異なる、カスタマイズされた管理 セットを提供し、組織はそれを利用しても よい （ MAY ）。

    \クリアページ

    したがって、組織は、xALテーラリングプロセスを確立し、文書化しなければならない（SHALL）。最低限、このプロセスは次のとおりである：

    • 意思決定を可能にするために、文書化されたガバナンス・アプローチに従わなければならない。
    • 評価された xAL、修正された xAL、補完および補償コントロールを含む、調整プロセスにおけるすべ ての決定をデジタル ID Acceptance Statement に文書化しなければならない（第 3.4.4 項を参照）。
    • 最初に評価した xAL に対するすべてのリスクベースの決定または変更を、デジタル ID 受諾表明書 （第 3.4.4 項参照）で正当化し、文書化しなければならない。
    • テーラリングプロセスにおけるxAL選択の影響に関する主題分析を支援するために、部門横断的な能力を確立すべきである（例えば、プライバシー、ユーザビリティ、詐欺及びなりすましの影響、公平性、及びその他の重要な分野に関連するリスク及び考慮事項について語ることができる主題専門家）。
    • 選択されたxAL管理策の影響を評価するために、実運用データを組み入れる継続的なプロセスであるべきである。

    テーラリングプロセスは、個人の公平性、プライバシー、使いやすさをサポートしながら、ミッションの成功を可能にする方法で、オンラインサービス、システム、データを保護するために、リスクと影響のバランスをとる構造的な手段を促進する。

    プライバシー、公平性、使いやすさ、脅威への耐性を評価する

    特定のオンラインサービスの保証レベルを選択し、調整する場合、プロセスへの洞察及びインプッ トは、第 3.2 節の最初の影響評価を超えて拡張することが重要である。第3.3.4節の初期保証レベルの選定から最終的なxALの選定及び実施に移行する際、組織は、初期に選定されたxALについて定義された統制の詳細な評価を実施し、運用環境における潜在的な影 響を特定しなければならない（SHALL）。

    最低限，組織は，次の分野に関連する影響及び予期せぬ結果の可能性を評価しなければならない（SHALL）：

    • プライバシー- 評価された xAL での管理の対象となる個人、およびデジタル ID の確立、管理、 またはフェデレーションに関連する組織またはサードパーティの慣行によって影響を受ける個 人のプライバシーに対する意図しない結果を特定する。プライバシー評価は、プライバシー評価プロセスへのインプットとして、既存のプライバシーしきい 値評価（PTA）およびプライバシー影響評価（PIA）を活用すべきである。ただし、プライバシー・アセスメントの目標は、最初の保証レベルの選択から発生するプライバシ ー・リスクを特定することであるため、基盤となる情報システムについては、保証レベルのベースライン 管理に固有の追加的なアセスメントおよび評価が必要になる場合がある。
    • 衡平性- 初期保証レベルの実施によって、コミュニティ間の不平等が生じたり、維持 されたり、悪化したりしないかどうかを判断する。衡平性評価では、技術への習熟度やアクセス、必要な技術機能（カメラなど）を備えたエンドデバイスの利用可能性、コンピューティングやデバイスの共有シナリオ、住宅状況、インターネットへのアクセス、インターネットの速度、世帯の所得層、クレジットスコア、障害の有無、性別、肌の色、年齢、母国語、英語の流暢さ、学歴などの要素を考慮して、サービスを受ける地域社会への影響を評価しなければならない（shall）。この評価の意図は、社会から疎外され、歴史的に十分なサービスを受けていない集団に対す る潜在的影響を緩和し、ID 管理機能の要件から不釣り合いな影響を制限することである。
    • ユーザビリティ- 初期保証レベルの実装がエンドユーザ・エクスペリエンスに課題をもたらすか どうかを判断する。ユーザビリティ評価では、ID 管理制御によって生じるユーザビリティの影響を考慮し、サービスを 提供するコミュニティに過度の負担、不満、または摩擦が生じないこと、およびあらゆる能力のユーザにアクセシ ビリティを提供する経路があることを確認しなければならない（SHALL）。
    • 脅威への耐性- 運用環境、その脅威要因、および既知の戦術、技法、手順（TTP）に基づき、定義され た保証レベルおよび関連する管理がオンライン・サービスに対する特定の脅威に対処でき るかどうかを判断する。脅威アセスメントは、ID 管理機能の実装環境内の特定の既知の脅威、脅威行為者、および TTP を考慮しなければならない（SHALL）。たとえば、特定の給付プログラムは、家族の脅威または共謀の影響を受けやすい場合がある。オンライン・サービスが提供するコミュニティ内の特定の脅威に対処するために、補足的な制 御を実装する必要がある場合がある。逆に、脅威アセスメントによって、その環境に基づいて脅威態勢を縮小することが適切であるこ とが示された場合、機関は、評価した xAL を縮小したり、ベースライン制御を修正したりしてもよい。

    組織は、調整プロセスが、サービスを提供する主体およびコミュニティの制約に確実に対応す るよう、協議およびフィードバックを活用するべきである（SHOULD）。 組織は、周縁化されたグループと協働する市民社会組織が、感じられる、または感じられる可能性のある影響について意見を提供できるメカニズムを確立してもよい。

    さらに、組織は、ここに記載されていないミッション及びドメイン固有の考慮事項を完全に表すために、適宜、 ビジネス固有の追加評価を実施すべきである（SHOULD）。これらの評価は、第3.4.2 項及び第3.4.3 項で定義されているように、あらゆる代償的または補足的な管理にまで拡大しなければ ならない（SHALL）。

    このステップの結果は、プライバシー、公平性、使いやすさ、脅威への耐性、その他の側面に関する一連のリスクアセスメントであり、初期保証レベルの調整と、補償的・補足的コントロールの選択に反映される。

    補償コントロールの特定

    代償的管理とは、定義されたxALsの規範的管理の代わりに組織が採用する管理上、運用上又は技術上の管理である。これらは、実行可能な最大限の範囲で、規範的管理策と同じリスクに対処することを意図している。

    組織は、ベースラインコントロールを実施できない場合、又はリスクアセスメントにより、代償コントロールが組織のリスク許容度に合致してリスクを十分に軽減することが示された場合、代償コントロールを実施することを選択してもよい（MAY）。このコントロールは、本ガイドラインで定義される規範的記述の修正であってもよいが、アプリケーショ ン、デジタルトランザクション、又はサービスライフサイクルの他の場所に適用してもよい（MAY）。例えば

    • 連邦機関は、本ガイドラインに基づく権威ある情報源による ID 証拠の検証要件を補うために、Personal Identity Verification [FIPS201]で参照されているように、連邦政府の身元調査およびチェックを使用す ることを選択できる。
    • 組織は、エンドユーザ集団に必要な証拠が不足しているため、より弱い形式の ID 証拠を使用した検証プロセスが受け入れられた支払アプリケーションに、より厳密な監査およびトランザクショ ン審査プロセスを導入することを選択できる。

    代替コントロールが実装される場合、組織は、代替コントロール、逸脱の根拠、選択された代替の比 較可能性、および結果として生じる残留リスク（もしあれば）を文書化しなければならない。代替コントロールを実装する CSP および IDP は、RP がそのユースケースに対する代替コントロールの可否 を評価および判断できるように、統合前にこの情報をすべての潜在的な RP に伝達するものとする。

    テーラリングのプロセスにより、機関およびサービスプロバイダは、xAL および関連制 御の実装方法に関してリスクベースの決定を行うことができる。また、3.4.4 節で説明するデジタル ID 受諾表明を通じて、決定を文書化し伝達するメカニズ ムも提供する。

    補足的コントロールの特定

    補足的な管理とは、組織が選択した保証水準に対して規定された基本的な管理をさらに強化するために追加される可能性のある管理である。組織は、基本的な管理策では対処できない運用環境の特定の脅威に対処するために、補足的な管理策を特定し、実施すべきである（SHOULD）。例えば

    • プルーフィング・プロセスを完了するために、組織は、詐欺的な試みが多発するため、保証レベ ルで要求されるものを超えて、追加の ID 証拠に対してエンド・ユーザを検証することを選択で きる。
    • 組織は、AAL2において、ユーザをフィッシング耐性認証のみに制限することができる。
    • 組織は、アクセス試行が特定のリスク要因を示す場合にユーザーの身元を確認するために、再証明メカニズムと組み合わせたリスクスコア分析を実装することを選択できる。

    補足的な管理策はすべて、組織の保証レベルを調整するために使用されたのと同じ要素に基 づいて影響を評価しなければならず（SHALL）、文書化しなければならない（SHALL）。

    デジタル ID アクセプタンス・ステートメント（DIAS）

    組織は、組織が管理する各オンラインサービスについて、デジタル ID リスク管理プロセスの結果を 文書化するデジタル ID 受諾表明書（DIAS）を作成しなければならない（SHALL）。CSP/IdP は、DIAS および実践ステートメントを RP が利用できるようにしなければならない。特定の CSP/IdP を使用する予定の RP は、後者の DIAS および実務記述書を確認し、各オンライン・サービ スの組織の DIAS に関連情報を組み込むものとする。

    DIASは最低限、以下を含むものとする：

    • 初期の影響評価結果、
    • 当初はxALを評価した、
    • 調整されたxALが当初評価されたxALと異なる場合は、調整されたxALとその根拠、
    • すべての代替コントロールとその比較可能性または残存リスク。
    • すべての補助コントロール。

    連邦機関は、 [NISTRMF]に記述されている情報システム認可パッケージにこの情報を含めるべきである。

    継続的な評価と改善

    脅威主体は適応し、利用者の能力、期待、およびニーズは変化し、季節的急増が発生し、 ミッションは進化する。そのため、リスク評価と ID ソリューションは継続的に改善されなければならない。脅威および技術環境に対応することに加え、継続的改善は、もし対処されなければ、リスク管理目 的のバランスをとる方法で ID 管理システムの実装を妨げる可能性のあるプログラム上の ギャップを示すための重要なツールである。たとえば、ある組織は、オンライン・サービスによってサービスを提供することを意図している 対象人口の一部が、リモート ID 証明をサポートするのに必要な手頃な価格の高速インターネット・ サービスを利用できないと判断することができる。組織は、地域内のローカルなプルーフィング機能を実装するプログラム、または地域のコミュ ニティセンター、最寄りの郵便局、提携ビジネス・パートナーの施設、または個人の自宅など、 よりアクセスしやすく便利な住所で個人と面会するプルーフィング代理人とのアポイントメントを提供す ることによって、このギャップに対処することができる。

    運用環境の変化に対応し、サービス能力のギャップに迅速に対処するために、組織は、ID 管理システ ムと相互作用したエンド・ユーザからの情報およびオンライン・サービスのパフォーマンス測定基準を活 用する継続的な評価および改善プログラムを実施しなければならない（SHALL）。このプログラムは、収集される測定基準、パフォーマン ス評価を可能にするために必要なデータ源、および継続的改善プロセスに基づいて適時に措置を講じる ためのプロセスを含めて文書化しなければならない（SHALL）。このプログラムおよびその有効性は、成果が達成されていること、およびプログラムがタイムリーに問題に対処していることを確認するために、定められたベースで評価されるべきである（SHOULD）。

    さらに、組織は、最新の脅威と詐欺の手口を常に把握するために、進化する脅威の状況を監視しなければならない（SHALL）。組織は、最新の脅威及び不正の手口に対する現在のセキュリティ対策及び不正検知能力の有効性を定期的に評価しなければならない（SHALL）。

    評価インプット

    ID システムのパフォーマンスを完全に理解するために、組織は評価プロセスへの重要なインプッ トを特定する必要がある。最低限、以下を含めるものとする：

    • 統合された CSP、IdP、および認証機能のほか、必要に応じて検証、検証、および不正管理シス テム。
    • 苦情処理、ヘルプデスク統計、その他のユーザーフィードバック（調査、インタビュー、フォーカスグループなど）などの顧客フィードバックメカニズム
    • 脅威分析、脅威レポート、脅威インテリジェンスフィードを組織で利用できるようにする。
    • 不正の傾向、不正調査の結果、不正の指標を組織で利用できるようにする。
    • 継続的な公平性評価、プライバシー評価、ユーザビリティ評価の結果。

    組織は、CSP、IdP、またはその他の統合 ID サービスの測定基準、報告要件、およびデータ入力を 文書化し、パートナーおよびベンダーに期待事項が適切に伝達されるようにしなければならない（SHALL）。

    公平性の評価と成果を支える測定

    継続的改善の主な目的は、さまざまな利用者集団に対する公平性およびアクセシビリティの結果を 改善することである。その結果、組織によって収集された測定基準は、支援されるコミュニティおよび人口統計に対す る ID 管理システムのパフォーマンスに関する洞察を提供するためにさらに評価されるべきである。可能な場合、このような取り組みでは、追加の個人情報の収集は避けるべきであり、その代わ りに、潜在的な格差の指標を提供するのに役立つ代理データの情報に基づく分析を使用すべきである。これには、郵便番号、地理的地域、年齢、性別など他の利用可能なデータに基づいて、 異なる利用者集団間のパフォーマンスの偏差を識別するための測定基準の比較やフィルタリングを含 むことができる。

    各団体は、OMB 報告書「公平なデータのためのビジョン（A Vision for Equitable Data）」を参照することが推奨される： 平等なデータ作業部会からの提言 [EO13985-vision]を参照し、パフォーマンス指標を人口統計学的グループ間の公平性評価に組み込み、公平なパフォーマンス結果を評価するための統計的推定値を作成するための指針を得ることを推奨する。

    リドレス

    さまざまな人々をサポートするサービスを設計する上で重要なのは、問題を裁定し、必要に応じて救済2を行うプロセスを組み込むことである。サービスの障害、紛争、その他の問題は、通常業務の一環として発生する傾向があり、その影響は、軽微な不便から大きな混乱や損害まで、多岐にわたる。アクセスの障害、サイバーセキュリティ事件やデータ漏洩は、影響を受ける個人にとって現実的な結果をもたらす。さらに、ある個人やコミュニティが不便を感じているのと同じ問題が、他の個人やコミュニティ、特に現在他の被害や障壁を経験している個人やコミュニティに、不釣り合いに大きな損害を与えることもある。これらの問題を放置しておくと、既存の不公平を悪化させ、排除の体系的な連鎖を継続させる害をもたらす可能性がある。

    ID 関連の不正行為やサイバーセキュリティの脅威を抑止しながら、重要なサービスへの公平なアクセ スを可能にするには、組織が潜在的な問題を計画し、公正で透明性があり、正当な請求者が容易にナビゲーシ ョンでき、搾取の試みに抵抗することを目的とした救済アプローチを設計することが不可欠であ る。

    いつ、どのような形で危害が発生しているかを理解することは、組織が十分な情報に基づいた行動をとるための重要な第一歩である。継続的な評価および改善プログラムは、潜在的な危害の事例およびパターンを特定する上で重 要な役割を果たすことができる。さらに、ID 管理をサポートするために確立された業務プロセス以外でも、問題の裁定および救済への 包括的なアプローチの一部として活用できる業務プロセスが存在する可能性がある。これらの活動だけでなく、ID 管理システムの利用者が懸念を表明し、救済への道を確保できるよう にするための追加的な慣行も実施できる。これらの実践の要件には以下が含まれる：

    • RP と CSP は、文書化され、アクセス可能で、追跡可能で、すべての人が利用可能であり、一般 向けのウェブサイトでその手順を簡単に見つけることができる問題処理プロセスを通じて、人々が苦情を伝え、救済 を求めることができるようにしなければならない。
    • RP および CSP は、この問題処理プロセスを実施するために、文書化された役割と責任を含むガバナン ス・モデルを制定しなければならない（SHALL）。
    • 争点処理プロセスは、以下を含む専用機能として実施されなければならない：
      • 争点に関連する証拠を公平にレビューするための手順；
      • 問題に関連する証拠を公平にレビューするための手順。
      • 問題を迅速に解決し、是正措置を決定するための手順。
    • RP および CSP は、チャットボットなどのアルゴリズムによるサポート・メカニズムが生成したissue 裁定出力に介入し、これを上書きするために、人間のサポート要員を利用できるようにしなければならない（SHALL）。
    • RP および CSP は、デジタル ID 管理システムの問題処理手順、救済手段、およびサービスへのアクセス を得るために利用可能な代替手段について、サポート要員を教育するものとする。
    • RP および CSP は、エンド・ユーザが直面する主な障害や一般的に表明される苦情を報告するためのサ ポート機能を提供する、要員および技術に関するプロセスを実装しなければならない（SHALL）。このプロセスにより、トレース（誰が/何を報告したかなど）および追跡（処置の進捗/状況など）が可能になるものとする。
    • RP および CSP は、問題処理プロセスから得られた知見を、継続的な評価および改善活動に組み込まなければならない（SHALL）。

    組織は、これらおよびその他の新たな救済方法を検討することが奨励される。支援技術も含め、新しい救済方法を採用する前に、組織は、意図しない結果、特に救済に関連する目標に反したり、矛盾したりするような結果の発生を避けるために、対象集団でその方法をテストすべきである（SHOULD）。

    サイバーセキュリティ、不正行為、アイデンティティ・プログラムの完全性

    アイデンティティ・ソリューションは孤立した状態で運用されるべきではない。ID 機能を、サイバーセキュリティ、プライバシー、脅威インテリジェンス、詐欺検出、およびプログラ ム整合性を担当するチームと緊密に連携させることで、ビジネス能力をより完全に保護することができ ると同時に、ID ソリューション能力を常に向上させることができる。たとえば、プログラム・インテグリティ・チームが収集した支払詐欺データは、危殆化した加入者ア カウントの指標や、ID 証明の実装における潜在的な弱点を提供する可能性がある。同様に、脅威インテリジェンス・チームは、ID プルーフィング、認証、およびフェデレーショ ン・プロセスに影響を与える可能性のある新しい TTP を知る可能性がある。組織は、重要な内部セキュリティおよび不正行為の関係者間で情報を交換するための一貫したメカニズ ムを確立しなければならない（SHALL）。組織は、オンライン・サービスの保護計画の一部である外部の利害関係者および ID サービスについても同じことを行うべきである（SHOULD）。

    サポートする ID サービス・プロバイダ（CSP など）が組織の外部にある場合、セキュリティ、 詐欺、およびその他の RP 機能に関連するデータの交換は、規制またはポリシーによって複雑 になる可能性がある。しかし、効果的な情報共有を可能にするために必要な仕組みとガイドラインを確立することは、契約上および法律上の仕組みにおいて考慮されるべきである（SHOULD）。収集、送信、共有されるデータはすべて、最小化され、生成エンティティによる詳細なプライバシ ーおよび法的評価の対象とならなければならない（SHALL）。

    このセクションは、ID 機能にとってより良い結果を達成するために、さまざまな組織機能チームと の調整および統合に取り組むことを意図している。このような調整は、リスク管理プロセスおよび運用ライフサイクル全体を通して実行されるのが 理想的である。付属書[SP800-63A]、[SP800-63B]、および[SP800-63C]は、各 ID 機能に関連する具体的な詐欺行為軽減要件を提供する。

    アイデンティティ・システムにおける人工知能（AI）と機械学習（ML）

    アイデンティティ・ソリューションは、バイオメトリクス・マッチング・システムの性能向上、認証の文書化、不正行為の検出、さらにはユーザー支援（チャットボットなど）など、さまざまな目的でAIやMLを使用してきたし、今後も使用していくだろう。AIやMLの潜在的な用途は多岐にわたる。また、格差のある結果、偏った出力、既存の不平等やアクセス問題の悪化など、明確なリスクや潜在的な問題も発生する。

    以下の要件は、ID システムでの使用方法に関係なく、AI および ML のすべての使用に適用される：

    • AI および ML のすべての使用は、文書化され、これらのシステムに依存する組織に通知されるものとする。CSP、IdP、または検証者による AI および ML を活用する統合技術の使用は、これらのシス テムからの情報に基づいてアクセス決定を行うすべての RP に開示されるものとする。
    • AI及びMLを使用するすべての組織は、その技術を使用するすべての主体に対し、モデルの訓練に使用した方法及び技術、訓練に使用したデータセットの説明、モデルの更新頻度に関する情報、並びにそのアルゴリズムについて完了したすべてのテストの結果に関する情報を提供しなければならない（SHALL）。
    • AI及びMLシステムを使用する、又はこれらのシステムを使用するサービスに依存するすべての組織は、AI及びMLの使用によってもたらされる可能性のあるリスクを評価するために、NIST AIリスク管理フレームワーク（[NISTAIRMF]）を実施しなければならない（SHALL）。
      1. AIおよびMLを使用するすべての組織は、 [SP1270]「人工知能におけるバイアスの管理基準に向けて」を参照するものとする。

    NISTは、多くの場を通じて、安全で信頼できるAIの実装を促進する取り組みを進めている。特に、NIST[US-AI-Safety-Inst]に所在する米国AI安全研究所は、安全性に焦点を当てたリソース、ガイダンス、ツールのポートフォリオを作成しており、組織がAIシステムを評価、導入、管理する方法を改善することができます。組織は、米国AI安全研究所の取り組みをフォローし、そのリソースを活用することが推奨される。

    1. プラクティスステートメントとその内容に関する更なる情報は、SP800-63Aの3.1節に記載されている。

    2. 救済とは一般的に、損害が発生した後に行われる救済を指す。

    用語集

    このセクションは参考になる。

    デジタル ID の領域ではさまざまな用語が使用される。多くの定義は SP 800-63 の以前のバージョンと一貫しているが、この改訂で変更されたものもある。これらの用語の多くには単一の一貫した定義がないため、ここでの用語の定義方法には注意が必 要である。

    アカウントリンク
    複数の連合識別子を単一のRP 加入者アカウントに関連付けること、またはそれらの関連付けを 管理すること。
    アカウント回復
    加入者アカウントとそれに関連する情報および権限の所有権を回復する能力。
    アカウント解決
    RP 加入者アカウントと、フェデレーション・トランザクション以前にRPがすでに保有していた情 報との関連付け。
    活性化
    認証に使用できるようにするために、多要素認証機能に起動要素を入力するプロセス。
    活性化因子
    多要素認証による 認証を成功させるために使用される追加認証要素。
    起動の秘密
    多要素認証の 起動要素としてローカルで使用されるパスワード。
    許可リスト
    ポリシー決定ごとに、許可される特定の要素の文書化されたリスト。フェデレーションのコンテキストでは、加入者の介入なしにIdPへの接続を許可されたRPのリストを指すために最も一般的に使用される。この概念は歴史的にホワイトリストとして知られてきた。
    申請者
    身分証明および登録のプロセスを受ける対象。
    \クリアページ
    応募者レファレンス
    申請者の身元、申請者に関連する特定の属性、または個人の状況に関連する条件（緊急事態、ホームレス状態など）を保証できる申請者の代理人。
    公認暗号
    暗号化アルゴリズム、ハッシュ関数、ランダム・ビット・ジェネレータ、または類似の技法で、連邦情報処理標準（FIPS）承認またはNIST推奨のもの。承認されたアルゴリズムおよび技法は、FIPS または NIST 勧告に規定または採用されている。
    アサーション
    サブスクライバの認証イベントに関する情報を含む、IDPからRPへのステートメント。アサーションには、サブスクライバの ID属性を含めることもできる。
    アサーションリファレンス
    アサーションとともに作成されるデータ・オブジェクトで、RPが認証された保護チャネ ル上でアサーションを取得するために使用する。
    アサーション・プレゼンテーション
    アサーションが RP に送信される方法。
    非対称キー
    公開鍵と秘密鍵で構成される2つの関連鍵で、暗号化と復号化、署名の検証と生成など、補完的な処理を実行するために使用される。
    認証
    通常、認証者がバインドされるときにCSP に伝達される情報で、接続されている認証者または認証操作に関与するエンドポイントの特徴を記述する。
    属性
    誰かまたは何かに帰属する品質または特性。ID 属性は、加入者の ID に関する属性である。
    属性バンドル
    CSP からの属性値および 派生属性値のパッケージ。パッケージは、CSP またはIdP との相互作用から独立してバンドルの検証を可能にするために、必要な暗号化保護を持っている。属性バンドルは、サブスクライバが管理するウォレットでよく使用されます。
    属性プロバイダ
    サブスクライバの属性へのアクセスを提供するID APIのプロバイダ。このAPIは、サブスクライバがRP に存在することを必ずしもアサートしない。
    属性検証
    一連の属性が正確であり、現実の ID に関連付けられていることを確認するプロセスまたは行為。検証を参照。
    属性値
    サブスクライバの ID 属性を、形式に関係なくアサートする完全なステートメント。たとえば、属性"birthday "の値は、"12/1/1980 "または "1980 年 12 月 1 日 "である。
    視聴者制限
    特定の対象者にメッセージを制限することで、受信者が別の受信者向けのメッセージを知らずに処理することを防ぐ。フェデレーション・プロトコルでは、RP が異なる RP 向けに生成されたアサーションを受け入れないように、アサーションは特定のRPに聴衆制限される。
    認証する

    認証済み保護チャネル

    認証プロトコルを使用して、請求者が 1 つまたは複数の認証子を所有し管理しているこ とを検証することにより、請求者の身元を検証するエンティティ。これを行うには、検証者は、認証子とサブスクライバ・アカウントのバインディングを確認し、 サブスクライバ・アカウントがアクティブであることを確認する必要がある。
    ベリファイアなりすまし
    フィッシングを参照。
    ゼロ化
    値ゼロのビットのみで構成されるデータでメモリロケーションを上書きし、データを破壊して回復不可能にする。これは、データそのものではなく、ファイルシステム内のデータへの参照を破壊するだけの削除方法とよく対比される。
    ゼロ知識パスワードプロトコル
    パスワードベースの認証プロトコルで、請求者がベリファイアに パスワードを明かす ことなくベリファイアを 認証できるようにする。このようなプロトコルの例として、EKE、SPEKE、SRPがある。

    変更履歴

    SP 800-63-1

    NIST SP 800-63-1 は、現在の認証子（当時は「トークン」と呼ばれた）技術を反映するために NIST SP 800-63 を更新し、ここで使用されるデジタル ID アーキテクチャ・モデルの理解を深めるために再構成した。デジタル ID モデルに実装される場合、CSP、認証情報の伝送に使用されるプロトコル、およびアサーションに ついて、追加の（最低限の）技術要件が規定された。

    SP 800-63-2

    NIST SP 800-63-2 は、SP 800-63-1 の限定的な更新であり、実質的な変更は第 5 節「登録および発行プロセ ス」のみであった。改訂された草案における実質的な変更は、ID 証明プロセスにおける専門家クレデンシャルの 使用を容易にし、レベル 3 リモート登録のクレデンシャルを発行するために記録された住所に郵便を 送る必要性を減らすことを意図したものである。第 5 条のその他の変更は、細かい説明と明確化である。

    SP 800-63-3

    NIST SP 800-63-3 は、SP 800-63-2 の大幅な更新および再構築である。SP 800-63-3 は、デジタル認証保証の各コンポーネント（AAL、IAL、および FAL）を導入し、認証強度の独立した扱いと個人の主張する ID への信頼（強力な仮名認証など）のニーズの高まりをサポートする。リスク評価方法論および IAL、AAL、および FAL へのその適用が、このガイドラインに含まれ ている。また、SP 800-63 でカバーされるデジタル ID ガイダンス全体が、認証について説明する 1 つの文書から、SP 800-63-3 を最上位文書とする 4 つの文書群（上記の個々の構成要素に個別に対応）に移行している。

    800-63-3で更新された他の分野は以下の通り：

    • IDプルーフおよびフェデレーションを含む範囲を適切に表現し、将来の改訂でデバイス ID またはマシン間認証を含む範囲の拡張をサポートするために、「デジタル ID ガイドライン」に改名した。
    • アサーション技術におけるトークンという言葉の矛盾した使用を避けるため、トークンの代わりにオーセンティケータを使用するなど、用語を変更した。
    • セキュリティ技術と脅威の両方の進歩を反映するため、認証およびアサーション要件を更新した。
    • 検証者による長期秘密の保管に関する要件の追加。
    • アイデンティティ証明モデルの再構築。
    • リモートID証明に関する要件を更新した。
    • 独立したチャンネルやデバイスを「持っているもの」として使用することを明確にした。
    • 事前登録されたナレッジトークン（認証子）は、（しばしば非常に弱い）パスワードの特殊なケースであるという認識のもと、削除された。
    • オーセンティケータの紛失または盗難時のアカウント回復に関する要件を追加。
    • 帯域外認証に有効なチャネルとして電子メールを削除した。
    • 再認証とセッション管理に関する説明の拡張。
    • アイデンティティ・フェデレーションに関する議論の拡大。フェデレーションのコンテキストにおけるアサーションの再構築。

    SP 800-63-4

    NIST SP 800-63-4は、SP 800-63-3からの大幅な更新と再編成を行った。800-63-4の更新内容は以下の通りです：

    • セキュリティとプライバシーへの配慮を拡大し、公平性とユーザビリティへの配慮を加えた。
    • デジタル・アイデンティティ・モデルを更新し、デジタル・ウォレットと属性バンドルへの注目と採用の高まりに対応する、ユーザー制御のウォレット・フェデレーション・モデルを追加した。
    • デジタル ID リスク管理プロセスを拡張し、保護されるオンラインサービス、ユーザグループ、 影響を受けるエンティティの定義を含める。
    • DIRM プロセスのコンテキスト、DIRM が対処するリスクの 2 つの側面、および意図する結果を 確立するための、より説明的な導入部。このコンテキスト設定ステップには、組織が提供し、ID システムで保護することを意図しているオンラ イン・サービスの定義と理解が含まれる。
    • デジタル ID リスク管理プロセスを拡張し、保護されるオンライン・サービス、ユーザ・グループ、 および影響を受けるエンティティの定義を含める。
    • デジタルアイデンティティ・リスク管理プロセスを更新し、初期ベースライン・コントロールの選択を調整するための追加評価を行う。
    • デジタル ID システムを継続的に評価するためのパフォーマンス測定基準を追加した。
    • 救済プロセスと要件に関する新しいサブセクションを追加。
    • デジタル ID サービスにおける人工知能の使用に対応する新しい人工知能サブセクションを追加。
