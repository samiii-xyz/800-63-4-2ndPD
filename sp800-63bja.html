    ABSTRACT

    このガイドラインは、ネットワークを介して政府情報システムと相互作用する主体の認証に重点を置 き、特定の請求者が、以前に認証された加入者であることを立証する。認証プロセスの結果は、認証を実行するシステムによってローカルに使用されるか、または連携 ID システムの他の場所でアサートされる。本文書は、3 つの認証者保証レベルごとに技術要件を定義する。このガイドラインは、この目的以外の標準の開発または使用を制約することを意図するものではない。本書は、NIST 特別刊行物（SP）800-63B に取って代わるものである。

    レビュアーの皆様へ

    2022 年 12 月、NIST は SP 800-63 改訂 4 版の初期公開草案（IPD）を公表した。119 日間のパブリックコメント期間中、著者は幅広い関係団体や個人から非常に優れた フィードバックを受けた。約 4,000 件の具体的なコメントからのインプットは、安全で、非公開で、公平で、アクセス可能な ID システムの実装を可能にする基礎的なリスク管理プロセスおよび要件を提供するという NIST の重要な目標を支援する形で、デジタル ID ガイドラインの改善を進めるのに役立った。この最初のフィードバックの波に基づいて、すべてのボリュームにわたっていくつかの実質的な 変更が加えられた。これらの変更には以下が含まれるが、これらに限定されない：

    1. リスク管理のための本文およびコンテキスト設定の更新。具体的には、著者は IPD で定義されたプロセスを修正し、組織が提供し、ID システムで保護する 可能性があるオンライン・サービスを定義し、理解するコンテキスト設定ステップを含めるようにし た。
    2. 推奨される継続的評価基準の追加IPD によって導入された継続的改善のセクションが拡張され、ID ソリューションのパフォーマンスを全体 的に評価するための推奨メトリクス一式が追加された。これらは、データ・ストリームの複雑さやソリューション展開のばらつきのために推奨されている。
    3. 拡張された不正要件および推奨。クレデンシャル・サービス・プロバイダおよび依拠当事者に対するプログラム的不正 管理要件は、不正チェックの実施から生じる可能性のある問題および課題に対応するようになった。
    4. ID 証明の管理を再構築した。証明を提供する手段に基づいて、各保証レベルの要件に新しい分類法と構造が設けられた：リモート無人、リモート出席（ビデオ・セッションなど）、オンサイト無人（キオスクな ど）、オンサイト出席（対面など）。
    5. 統合された同期可能なAuthenticator。2024 年 4 月、NIST は、同期可能なAuthenticatorに関する暫定ガイダンスを公表した。このガイダンスは、規範となるテキストとして SP 800-63B に統合され、改訂 4 版のボリュームセットの一部として、一般からのフィードバックのために提供されている。
    6. フェデレーション・モデルにユーザ管理ウォレットを追加。デジタル・ウォレットとクレデンシャル（SP 800-63C では「属性バンドル」と呼ばれる）は、注目と採用 が高まっている。その核心は、フェデレーショ ンされた IdP のように機能し、対象者に関する署名されたアサーションを生成することである。このプレゼンテー ションおよび新たなコンテキストに関する具体的な要件は、SP 800-63C-4 に示されている。

    過去数年間のオンライン・サービスの急速な普及により、信頼性が高く、公平で、安全で、プライ バシを保護するデジタル ID ソリューションの必要性が高まっている。 NIST 特別刊行物 SP 800-63「デジタル ID ガイドライン」の改訂 4 は、オンライン・リスクの現実世界での影響を含め、このスイートの最後の大改訂が 2017 年に発行されて以降に現れた変化するデジタル状況に対応することを意図している。このガイドラインは、セキュリティとプライバシーの要件だけでなく、公平性の育成とデジ タル ID ソリューションおよび技術の使いやすさへの配慮を含め、ID 証明、認証、およびフェデレーショ ンのためのデジタル ID 管理保証レベルを満たすためのプロセスと技術要件を提示している。

    序文

    本書およびその付属書である[SP800-63]、[SP800-63A]、および[SP800-63C]は、組織がデジタル ID サービスを実装するための技術ガイドラインを提供する。

    この文書、SP 800-63B は、3 つの認証保証レベル（Authentication Assurance Levels：AALs）ごとに、リモート・ユーザ認証に関するクレデンシャル・サービス・プロバイダ（CSPs）に対する要件を規定している。

    はじめに

    このセクションは参考になる。

    認証とは、デジタル・サービスにアクセスしようとする対象者が認証に使用される秘密を管理して いることを証明することによって、デジタル ID を主張するために使用される 1 つまたは複数の Authenticatorの妥当性を決定するプロセスである。再訪問がサービスに適用される場合、認証に成功すると、現在サービスにアクセスしている対象者 が、以前にサービスにアクセスした者と同じであるという合理的なリスクベースの保証が提供 される。ワンタイム・サービス（サブスクライバがサービスにアクセスするのが 1 回だけの場合）では、永続的なデジタル認証をサポートするAuthenticatorの発行は必ずしも必要ではない。

    請求者の認証は、クレデンシャル・サービス・プロバイダ（CSP）によって維持される加入者ア カウントに記録される加入者のオンライン活動と加入者を関連付けるプロセスの中心である。認証は、請求者が所定のサブスクライバ・アカウントに関連付けられた 1 つ以上のAuthenticator（SP 800-63 の旧版ではトークンと呼ばれる）を管理していることを検証することによって実行される。認証プロセスは、CSP または連携認証では ID プロバイダ（IdP）の役割であるベリファイアによっ て実施される。認証に成功すると、ベリファイアは依拠当事者（RP）に識別子をアサートする。オプションとして、ベリファイアは RP に追加の属性をアサートすることができる。

    本文書は、さまざまな認証保証レベル（Authentication Assurance Levels：AALs）で使用され るAuthenticatorの選択など、認証プロセスのタイプに関する推奨事項を規定する。また、初期発行、保守、およびAuthenticatorの紛失または盗難時の無効化など、Authenticatorの有効期間 中に発生する可能性のあるイベントについても推奨する。

    この技術ガイドラインは、ネットワーク上のシステムに対するサブジェクトのデジタル認証 に適用される。また、認証プロトコルに参加するベリファイアおよび RP は、認証するサービスの ID を保証するために、請求者に対して認証される必要がある。物理的なアクセス（建物へのアクセスなど）のための人の認証には対応していない。ただし、[SP800-116]に記載されているように、デジタルアクセスに使用されるクレデンシャルの一部は、物理アクセス認証にも使用される場合がある。

    AAL は、認証トランザクションの強度を序列カテゴリとして特徴付ける。より強力な認証（すなわち、より高い AAL）には、悪意のあるアクターがより優れた能力を有し、認証プロセスを成功裏に破壊するために、より大きなリソースを費やす必要がある。より高い AAL での認証は、攻撃のリスクを効果的に低減できる。各 AAL の技術要件の概略を以下に示す。具体的な規範要件については、本文書の第 2 節および第 3 節を参照のこと。

    認証保証レベル 1： AAL1 は、請求者が、認証されるサブスクライバ・アカウントにバインドされ たAuthenticatorを管理しているという基本的な信頼性を提供する。AAL1 では、利用可能な幅広い認証技術を使用した 1 要素認証のみが要求される。ただし、AAL1 で評価されるアプリケーションは、多要素認証オプショ ンを提供することが推奨される。認証を成功させるには、請求者がAuthenticatorを所有し管理していることを証明する必要がある。

    認証保証レベル 2： AAL2 は、請求者が、認証されるサブスクライバ・アカウントにバインドされる 1 つ以上の認証要素を管理しているという高い信頼性を提供する。2 つの異なる認証要素を所有し、管理していることの証明が必要である。AAL2 で評価されるアプリケーションは、耐フィッシング性の認証オプションを提供する必要がある。

    認証保証レベル 3： AAL3 は、請求者が認証されるサブスクライバ・アカウントにバインドされる 1 つ以上の Authenticatorを管理しているという非常に高い信頼性を提供する。AAL3 における認証は、公開鍵暗号プロトコルの使用による鍵の所有の証明に基づく。AAL3認証には、エクスポート不可能な秘密鍵を持つハードウェアベースの認証機と、 フィッシング耐性のある認証機が必要である（セクション3.2.5参照）。AAL3で認証するために、請求者は、2つの異なる認証要素の所有と管理を証明する必要が ある。

    セッションが所定のAALで認証され、より高いAALが必要な場合、認証プロセスは、 セッションのAALを上げるためにステップアップ認証を提供することもできる。

    表記

    本ガイドラインでは、本文中において以下のような表記規則を用いている：

    • 大文字で示された特定の用語は、規範となる要求事項を表す。同じ用語が大文字で表されていない場合，その用語は規範的要求事項を表していない。
      • SHALL"及び "SHALL NOT"という用語は，その出版物に適合するために厳密に従うべき要求事項を示し，そこからの逸脱は許されない。
      • SHOULD"及び "SHOULD NOT"という用語は，いくつかの可能性の中で，他の可能性に言及したり除外したりすることなく，特に適切なものとして推奨されること，ある行動方針が望ましいが必ずしも必要ではないこと，又は（否定形では）ある可能性や行動方針が推奨されないが禁止されないことを示す。
      • MAY"と "NEED NOT"という用語は、出版物の範囲内で許容される行動方針を示す。
      • CAN"と "CANNOT"という用語は、物質的、物理的、または因果的な可能性や能力を示し、否定形では、その可能性や能力がないことを示す。

    認証保証レベル

    このセクションは規範的なものである。

    所定の AAL の要件を満たし、サブスクライバとして認識されるために、請求者は、 [SP800-63C]に記載されているように、RP または IdP に対して、そのレベルの要件と同等以上の強度を持つプロセスで認証を行わなければならない（SHALL）。認証プロセスは、加入者がその RP に対して認証するたびに、加入者を一意に識別する 識別子を生成する。識別子は仮名であってもよい[MAY]。サブスクライバを一意のサブジェクトとして識別する他の属性も提供してもよい[MAY]。

    各AALにおける認証者および検証者に対する詳細な規範要件は、セクション3に記載されている。最適なAALの選択方法の詳細については、[SP800-63]セクション3を参照のこと。

    身元証明中およびその後に収集された個人情報（[SP800-63A]に記載）は、加入者アカウントを通じてデジ タル ID サービスにより加入者が利用できるようにしてもよい。連邦機関が個人識別情報（PII）またはその他の個人情報を公開またはオンラインで利用可 能にするには、[EO13681]に従って多要素認証が必要である。したがって、連邦機関は、PII またはその他の個人情報をオンラインで利用可能にする場合、最低 AAL2 を選択しなければならない（SHALL）。

    すべての AAL において、誤認証のリスクを低減するために事前認証チェックを使用してもよい （MAY）。たとえば、予期しないジオロケーションまたは IP アドレス・ブロック（クラウド・サービスなど）からの認証は、リスク・ベースの追加制御の使用を促す場合がある。使用される場合、CSP または検証者は、事前認証チェックの有効性を評価し、潜在的なユーザ集団への異 なる影響を特定および軽減するものとする。CSP または検証者は、認証前チェックを認証プライバシー・リスク評価に含めるものとする。事前認証チェックは、トランザクションの AAL に影響を与えたり変更したり、認証要素の代用となる ことはない。

    本文書全体を通じて、[FIPS140]の要件は、FIPS 140 の最新版によって満たされる。有効である限り、従来の FIPS 140 認証も使用することができる。

    認証保証レベル1

    AAL1 は、サブスクライバ・アカウントにバインドされたAuthenticatorを請求者が管理しているとい う基本的な信頼性を提供する。AAL1 は、利用可能な幅広い認証技術を使用して、1 要素認証または多要素認証のいずれかを要 求する。ベリファイアは、AAL1 で多要素認証オプションを利用できるようにし、その使用を奨励す べきである（SHOULD）。認証に成功するには、安全な認証プロトコルを介して、請求者がAuthenticatorを所有し管理している ことを証明する必要がある。

    許可される認証タイプ

    AAL1認証は、以下の認証タイプのいずれかを使用しなければならない[SHALL ]：

    • パスワード（第3.1.1項）：加入者が通常選択する、記憶可能な秘密鍵。
    • ルックアップ秘密(Sec. 3.1.2)：加入者が保持するリスト中のプロンプト値を検索することにより、請求者が決定する秘密。
    • 帯域外装置（第3.1.3項）：加入者と別の通信チャネルを通じて送受信される秘密。
    • 単一要素ワンタイムパスワード（OTP）（第 3.1.4 項）：加入者が保持するデバイスまたはアプリケーションから取得されるワンタイムシークレット。
    • 多要素 OTP（第 3.1.5 項）：加入者が保持するデバイスまたはアプリケーションから取得されるワンタイムシークレット。
    • 一要素暗号認証（Sec. 3.1.6）：サブスクライバが保持する暗号鍵の認証プロトコルによる所有および管理の証明。
    • 多要素暗号認証（Sec. 3.1.7）：サブスクライバが保持する暗号鍵の認証プロトコルによる所有および管理の証明。

    Authenticatorと検証子の要件

    AAL1で使用されるAuthenticatorは、 承認された暗号を使用しなければならない（SHALL）。言い換えれば、承認されたアルゴリズムを使用しなければならないが、実装は[FIPS140]に基づいて検証される必要はない。

    請求者と検証者の間の通信は、1 つ以上の認証された保護されたチャネルを介して行われなけれ ばならない（SHALL）。

    連邦機関によって、または連邦機関のために AAL1 で運用される検証機で使用される暗号は、[FIPS140]レベル 1 の要件を満たすように検証されなければならない。

    再認証

    本ガイドラインでは、2種類のタイムアウトを規定しており、その詳細については第5.2節で説明する：

    1. 全体的なタイムアウトは、認証または前回の再認証の後、認証されたセッションの期間を指定された期間に制限する。
    2. 非活動時タイムアウトは、指定された期間サブスクライバからの活動がないセッションを終了させる。

    セクション5.2に記載されているように、加入者セッションの定期的な再認証を実 行しなければならない[SHALL]。明確な再認証全体のタイムアウトを設定しなければならない[SHALL]が、これは AAL1では30日以内であるべきである[SHOULD]。非活動タイムアウトを適用してもよい[MAY]が、AAL1では必須ではない。

    認証保証レベル2

    AAL2 は、サブスクライバ・アカウントにバインドされる 1 つまたは複数の認証要素を、請求者 が管理しているという高い信頼性を提供する。安全な認証プロトコルを使用することにより、2 つの異なる認証要素を所有し管理しているこ とを証明することが要求される。承認された暗号化技術が必要である。

    許可される認証タイプ

    AAL2 では、認証は、多要素認証機能または 2 つの単一要素認証機能の組み合わせを使用しなけれ ばならない（SHALL）。多要素認証機能は、1 つの認証イベントを実行するために 2 つの要素を必要とする。たとえば、デバ イスをアクティブにするために必要な生体センサーが統合された暗号化されたセキュアデバ イスなどである。Authenticator要件は第 3 節で規定される。

    多要素認証機能が使用される場合、以下のいずれかを使用してもよい（MAY）：

    • 多要素帯域外認証機能（3.1.3.4 節）
    • 多要素OTP（第3.1.5項）
    • 多要素暗号認証（第3.1.7項）

    2つの単要素Authenticatorの組み合わせが使用される場合、その組み合わせには、パスワード（第 3.1.1 項）と、以下のリストから1つの物理的Authenticator（すなわち、「あなたが持っているもの」）を含めなければならない（SHALL）：

    • ルックアップシークレット（第3.1.2項）
    • 帯域外装置（第3.1.3項）
    • 一要素OTP（第3.1.4項）
    • 一要素暗号認証（第3.1.6項）

    バイオメトリクスの特性は、それ自体ではAuthenticatorとして認識されない。バイオメトリクス認証が第 3.2.3 節の要件を満たす場合、バイオメトリクスとともに物理的な Authenticatorも認証される。このとき、物理的なAuthenticatorは、"あなたが持っているもの "として機能し、バイオメトリッ クの照合は、"あなたがいるもの "として機能する。バイオメトリクスの比較が多要素認証機能のアクティブ化要素として使用される場合、認証機 能自体が物理的認証機能として機能する。

    Authenticatorと検証子の要件

    AAL2 で使用されるAuthenticatorは、承認された暗号を使用しなければならない（SHALL）。連邦機関によって調達される暗号化Authenticatorは、[FIPS140]レベル1の要件を満たすように検証されなけれ ばならない[SHALL]。AAL2で使用される少なくとも1つのAuthenticatorは、 セクション3.2.7に記載されているように、リプレイ耐性を持たなければならない[SHALL]。AAL2における認証は、Sec. 3.2.8で説明するように、少なくとも1つの認証者から認証の意図を示すべきである(SHOULD)。

    請求者と検証者の間の通信は、1 つ以上の認証された保護されたチャネルを介して行われなけれ ばならない（SHALL）。

    連邦機関によって、または連邦機関のために AAL2 で運用される検証機で使用される暗号は、[FIPS140]レベル 1 の要件を満たすように検証されなければならない。

    AAL2 における認証にバイオメトリクス要素が使用される場合、Sec.3.2.3に記載される性能要件を満たさなければならず（SHALL）、検証者は、バイオメトリクスセンサおよびその後の処理がこ れらの要件を満たしていることを判断しなければならない（SHALL） 。

    ベリファイアは、 第 3.2.5 節に記載されるとおり、AAL2 において少なくとも 1 つのフィッシング耐 性認証オプションを提供しなければならない。連邦機関は、職員、請負業者、パートナーが連邦情報システムにアクセスする際に、フィッシング耐 性認証を使用することを義務づけるものとする。フィッシングは重大な脅威ベクタであるため、検証者は、どのような場合でも、AAL2 での耐フィッシング認証の利用を奨励すべきである。

    再認証

    加入者セッションの定期的な再認証は、セクション5.2に記載されているように実 行されなければならない[SHALL]。明確な再認証全体のタイムアウトを設定しなければならない[SHALL]が、これは AAL2では24時間以下にすべきである[SHOULD]。非アクティブ時のタイムアウトは1時間以下であるべきである[SHOULD]。非アクティブ時のタイムアウトが発生したが、全体的なタイムアウトがまだ発生し ていない場合、ベリファイアは、セッションシークレットと組み合わせて、成功したパ スワードまたはバイオメトリック比較のみを使用してサブスクライバを再認証するこ とを許可してもよい[MAY]。

    認証保証レベル3

    AAL3 は、請求者が加入者アカウントにバインドされたAuthenticatorを管理しているという非常に高い信頼性を提 供する。AAL3 での認証は、暗号プロトコルの使用による鍵の所有の証明と、アクティブ化要素またはパス ワードのいずれかに基づいている。AAL3 認証では、フィッシング耐性を提供するハードウェアベースのAuthenticatorの使用が必要である。承認された暗号技術が必要である。

    一般要件

    以下の要件が、すべての AAL における認証に適用される。

    セキュリティ・コントロール

    CSP は、これらのガイドラインが保護に使用される情報システム、アプリケーション、およびオンライン・ サービスに対して、[SP800-53]または組織が選択した同等の連邦標準 (例えば、[FEDRAMP]) または業界標準に定義され ている中程度の基本セキュリティ管理から、適切に調整されたセキュリティ管理を採用するものとする。CSP は、適切なシステムの最低保証関連管理が満たされていることを保証するものとする。

    記録保存方針

    CSP は、適用される可能性のある米国国立公文書記録管理局（NARA）の記録保持スケジュールを含め、適用され る法律、規制、およびポリシーに従って、それぞれの記録保持ポリシーを遵守するものとする。強制的な要件がない場合に CSP が記録の保持を選択する場合、CSP は、プライバシーおよびセ キュリティ・リスクの評価を含むリスク管理プロセスを実施して、記録の保持期間を決定し、その保持方 針をサブスクライバに通知しなければならない ものとする。

    プライバシー要件

    CSP は、[SP800-53]または同等の業界標準に定義されている、適切に調整されたプライバシー管理を採用するものとする。

    CSP が、ID サービス（すなわち、ID 証明、認証、または属性アサーション）、関連する不正行為の 軽減、または法律もしくは法的手続きへの準拠以外の目的で属性を処理する場合、CSP は、追加的な 処理から生じるプライバシー・リスクに見合った予測可能性および管理可能性を維持するための対策を 実施しなければならない（SHALL）。このような措置の例には、明確な通知の提供、加入者の同意の取得、属性の選択的な使用または開示の可 能化などが含まれる。CSP が同意手段を使用する場合、CSP は、追加処理に対する同意を ID サービスの条件にしてはならない。

    CSP が省庁であるか民間プロバイダであるかにかかわらず、認証サービスを提供または使用す る連邦省庁には、以下の要件が適用される：

    1. 当該機関は、プライバシー上級機関職員（SAOP）と協議し、認証機を発行または維持するための PII の収集が1974 年プライバシー法 （ Privacy Act）の要件（第 7.4 項を参照）の引き金となるかどうかを判断するための分析を実施しなければならない。
    2. 当該機関は、該当する場合、そのようなコレクションを対象とする記録システム通知（SORN）を発行しなければならない（SHALL）。
    3. 当該機関は、SAOP と協議し、認証機能を発行または維持するために PII を収集することが2002 年電子政府法（E-Government Act of 2002 [E-Gov]）の要件を満たすかどうかを判断するための分析を実施しなければならない（SHALL）。
    4. 当該機関は、該当する場合、そのような収集を対象とするプライバシー影響評価（PIA）を公表しなければならない（SHALL）。

    救済要件

    CSP および検証者は、SP 800-63 のセクション 5.6 に記載されているように、サブスクライバの苦情およびサブスクライバ認証プロセスで発生した問題を解決するためのメカニズムを提供しなければならない。これらのメカニズムは、サブスクライバが容易に見つけ、使用できるものとする。CSP は、苦情または問題の解決に有効なメカニズムを評価するものとする。

    パスワード

    パスワード（パスフレーズ、または数値の場合はPINと呼ばれることもある）は、加入者 が選択し、記憶または記録することを意図した秘密の値である。パスワードは、攻撃者が正しい秘密の値を推測したり、そうでなければ発見したりすることが現実的でないような、十分な複雑さと機密性を備えていなければならない。パスワードは「あなたが知っているもの」である。

    このセクションの要件は、独立した認証要素として使用され、認証された保護されたチャネルを介して CSP の検証者に送信される、一元的に検証されたパスワードに適用される。多要素認証機能のアクティブ化要素としてローカルで使用されるパスワードは、アクティブ化秘密と呼ばれ、セ クション 3.2.10 で説明される。

    パスワードはフィッシングに強いものではない。

    パスワード認証

    パスワードは、サブスクライバが選択するか、または CSP がランダムに割り当てるものとする。

    選択されたパスワードが、一般的に使用される、予想される、または危殆化した値のブロック・リスト（第 3.1.1.2 節を参照）にあるために CSP がそのパスワードを許可しない場合、サブスクライバは別のパスワードを選択するよう要求されるものとする（SHALL）。パスワードに対する他の複雑さ要件を課してはならない[SHALL NOT]。その根拠は、付録A「パスワードの強度」に示されている。

    パスワード検証装置

    パスワードには以下の要件が適用される：

    1. 検証機関と CSP は、パスワードの長さが最低 8 文字であることを義務付けるものとし（SHALL）、パスワードの長さが最低 15 文字であることを義務付けるべきである（SHOULD）。
    2. 検証機関と CSP は、少なくとも 64 文字の最大パスワード長を許可すべきである（SHOULD）。
    3. ベリファイアおよび CSP は、パスワード内のすべての印刷 ASCII[RFC20]文字およびスペース文字を受 け入れるべきである（SHOULD）。
    4. 検証機関と CSP は、パスワード内の Unicode[ISO/ISC 10646]文字を受け入れるべきである（SHOULD）。パスワードの長さを評価する場合、Unicode の各コード・ポイントは 1 文字としてカウントされるものとする。
    5. 検証機関および CSP は、パスワードに他の構成規則（異なる文字タイプの混在を要求するなど）を課してはなら ない。
    6. ベリファイアおよび CSP は、パスワードの定期的な変更をユーザに要求してはならない。ただし、ベリファイアは、認証者の危殆化の証拠がある場合、変更を強制するものとする。
    7. ベリファイアおよび CSP は、サブスクライバが認証されていない請求者からアクセス可能なヒントを保 存することを許可してはならない。
    8. ベリファイアおよび CSP は、パスワードを選択する際に、知識ベース認証（KBA）（例えば、 「最初のペットの名前は何ですか」）またはセキュリティ質問を使用するよう、サブスクライバに促してはならない。
    9. ベリファイアは、提出されたパスワード全体を検証しなければならない（すなわち、パスワー ドを切り詰めない）。

    Unicode文字がパスワードで受け入れられる場合、検証者はUnicode Normalization Forms [UAX15]の第12.1節で定義されているNFKC正規化またはNFKD正規化のいずれかを使用して、安定化文字列の正規化処理を適用すべきである[SHOULD]。この処理は、パスワードを表すバイト列をハッシュする前に適用される。Unicode文字を含むパスワードを選択する加入者は、エンドポイントによっては一部の文字が 異なって表現される可能性があり、それが認証を成功させる能力に影響することを通知されるべきで ある[SHOULD]。

    パスワードの設定または変更の要求を処理するとき、ベリファイアは、一般的に使用され、予期 され、または危殆化した既知のパスワードを含むブロックリストと、想定される秘密を比較しなけ ればならない[SHALL]。比較の対象となるのは、パスワード全体であって、その中に含まれる可能性の ある部分文字列や単語ではないものとする。例えば、リストには次のものを含めてもよいが、これに限定されない：

    • 過去の違反コーパスから入手したパスワード
    • 辞書用語
    • サービス名、ユーザー名、およびその派生語など、コンテキスト固有の単語

    選択したパスワードがブロックリストにある場合、CSP または検証者はサブスクライバに別の秘 密を選択するよう要求し、拒否の理由を提示しなければならない（ SHALL ）。ブロックリストは、ブルートフォース攻撃を防御するために使用され、以下に説明するように、失敗した試行はレ ート制限されるため、ブロックリストは、試行制限に達する前に攻撃者が推測する可能性が高いパスワードをサブスク ライバが選択することを防ぐために十分なサイズであるべきである（SHOULD）。

    ブロックリストはオンライン攻撃を防御するために使用されるため、過度 に大きなブロックリストは、セキュリティ上のメリットはほとんどない。

    検証者は、利用者が強力なパスワードを選択できるように、利用者にガイダンスを提供しな ければならない[SHALL]。これは、ブロックリストに掲載されたパスワードが拒否された後に、掲載された弱いパスワー ドの些細な変更を阻止するために特に重要である[Blocklists]。

    ベリファイアは、 第 3.2.2 節に記載されているように、サブスクライバアカウントで行える認証試行 の失敗回数を効果的に制限する、レート制限メカニズムを実装しなければならない 。

    ベリファイアは、パスワード・マネージャの使用を許可しなければならない（SHALL）。ベリファイアは、パスワードの入力を容易にするために、請求者が「ペースト」機能を使用す ることを認めるべきである（SHOULD）。パスワードマネージャは、特にパスワードジェネレータ[Managers]を含む場合、ユー ザーがより強力なパスワードを選択する可能性を高めることが示されている。

    請求者がパスワードを正しく入力できるように、検証者は、入力中および検証者に提出されるま で、一連のドットやアスタリスクではなく、秘密を表示するオプションを提供すべきである。これにより、請求者が画面を観察されにくい場所にいる場合、入力を確認することができる。ベリファイアはまた、正しい入力を確認するために、各文字が入力された後に短時間、請求 者のデバイスが個々の入力された文字を表示することを許可してもよい。これはモバイル機器では一般的である。

    検証者は、検証前に先頭および末尾の空白文字を削除したり、先頭文字の大文字小文字が異なるパス ワードの検証を許可するなど、ミスタイプを許容してもよい。

    検証機関と CSP は、パスワードを要求するときに、承認済みの暗号化と認証済みの保護されたチャネ ルを使用しなければならない（SHALL）。

    検証者は、オフライン攻撃に耐性のある形式でパスワードを保存しなければならない（SHALL）。パスワードは、適切なパスワード・ハッシュ・スキームを使用して、ソルトとハッシュを付 けなければならない（SHALL）。パスワードハッシュ方式は、パスワード、ソルト、コスト要素を入力とし、パスワードハッシュを生成する。その目的は、ハッシュ化されたパスワードファイルを入手した攻撃者にとって、 パスワードを推測するたびにコストがかかるようにすることである。選択されたコスト係数は、検証機の性能に悪影響を与えない範囲で、実用的な高さであるべき である。コスト係数は、計算機性能の向上を考慮し、時間の経過とともに増加させるべきである（SHOULD）。SP800-132]の最新版に掲載されている承認済みパスワードハッシュ方式、またはパスワード ハッシュ方式に関するNISTガイドラインの更新版を使用すべきである[SHOULD]。ソルトとバージョニング情報を除くパスワード検証者の選択された出力長は、基礎となる パスワードハッシュ方式の出力長と同じであるべきである[SHOULD]。

    ソルトは少なくとも32ビット長でなければならず(SHALL)、保存されたハッシュ間の ソルト値の衝突を最小化するように選択されなければならない(SHALL)。各パスワードに対して、ソルト値と結果のハッシュの両方を保存しなければ ならない[SHALL]。新しいアルゴリズムとワークファクターへの移行を可能にするため、ワークファクターを 含む使用されたパスワードハッシュスキームへの参照を各パスワードに対して保存すべきである[SHOULD]。例えば、パスワードベースの鍵導出関数2(PBKDF2)[SP800-132]の場合、コストファクターは反復回数である。

    さらに、ベリファイアは、ベリファイアのみが知っている秘密鍵を使用して、鍵付きハッシ ングまたは暗号化操作の追加反復を実行すべきである(SHOULD)。使用する場合、この鍵値は、3.2.12 節で説明するように、承認されたランダムビット生成機で生成しなけれ ばならない（SHALL）。秘密鍵の値は、ハッシュ化されたパスワードとは別に保管しなければならない（SHALL）。秘密鍵は、ハードウェアセキュリティモジュールや信頼された実行環境(Trusted Execution Environment: TEE)のようなハードウェアで保護された領域内に保存され、使用されるべきである(SHOULD)。この追加的な反復により、ハッシュ化されたパスワードに対するブルートフォース攻撃は、 秘密鍵の値が秘密のままである限り、現実的ではない。

    ルックアップの秘密

    ルックアップ秘密Authenticatorは、請求者と CSP 間で共有される一連の秘密を格納する物理的または電子的な 記録である。請求者はAuthenticatorを使用して、検証者からのプロンプトに応答するために必要な適切な秘密を検索する。たとえば、ベリファイアは、カードに印刷された数値または文字列の特定のサブセットを表形式で提供するよう請求者に求めることができる。ルックアップ秘密の典型的な用途は、別の認証器が紛失または故障した場合に使用するため、加入者 が保存する 1 回限りの保存リカバリコード（第 4.2.1.1 節参照）である。ルックアップシークレットとは、「あなたが持っているもの」である。

    ルックアップ・シークレットはフィッシングに強くない。

    ルックアップ・シークレット認証

    ルックアップ・シークレットのAuthenticatorを作成する CSP は、 Sec.3.2.12 に記載されるように、承認されたランダム・ビッ ト・ジェネレータを使用してシークレット・リストを生成し、Authenticatorリストをサブスクライバに安全に配 布しなければならない（例えば、対面セッション、AAL2 以上でサブスクライバによって認証されたセッション経由、ま たは連絡先住所への郵便など）。ルックアップシークレットは、少なくとも6桁(または同等の長さ)でなければならない[SHALL]。長さによっては、セクション3.1.2.2に記載される追加要件も適用される。

    ルックアップ・シークレットは、CSP が直接配布しても、利用者の連絡先住所に郵送しても、オンライン配布しても よい。オンラインで配布される場合、ルックアップ秘密は、第 4.1.2 節の登録後のバインディング要件に従い、安全なチャネ ルを介して配布されるものとする。

    ルックアップ・シークレット・ベリファイア

    ルックアップ秘密の検証者は、認証者の次の秘密または特定の(例えば、番号の付された)秘密を要求 するよう、請求者に促さなければならない。ルックアップ秘密鍵の認証者からの秘密鍵は、一度だけ正常に使用できるものとする。ルックアップ秘密鍵がグリッドカードに由来する場合、各グリッドセルは一度だけ 使用されるべきである[SHOULD]。そうでない場合、非常に長い秘密のリストが必要となる。

    検証者は、ルックアップ秘密をオフライン攻撃に耐性のある形式で保存しなければならない。すべてのルックアップ秘密は、承認されたハッシュ関数を使用して、ハッシュ化された形式で格納されなけれ ばならない（SHALL）。

    ルックアップシークレットは、第3.1.2.1項に規定されるとおり、少なくとも10進数6桁（または同等 の長さ）でなければならない（SHALL）。指定された長さより短いルックアップシークレットには、以下の追加検証要件がある：

    • SP800-131A]の最新改訂版で規定されている最小セキュリティ強度(本書の日付時点で 112ビット)よりも短いルックアップ秘密鍵は、第3.1.1.2項に記載されているように、適切なパスワー ドハッシュ方式を使用してソルト化し、ハッシュ化した形で保存しなければならない。ソルト値は少なくとも32ビット長でなければならず(SHALL)、保存されたハッシュ間の ソルト値の衝突を最小化するために任意に選択されなければならない(SHALL)。各ルックアップ秘密鍵に対して、ソルト値と結果のハッシュの両方を保存しなければ ならない[SHALL]。ルックアップ秘密鍵はランダムビット生成器を使用して生成されるため、 パスワードハッシュ方式のワークファクターは小さくてもよい[MAY]。

    • ベリファイアは、セクション3.2.2に記載されるとおり、加入者アカウントで行える認証試行 の失敗回数を効果的に制限する、レート制限メカニズムを実装しなければならない（SHALL） 。

    検証者は、ルックアップ秘密を要求するときに、承認された暗号化と認証された保護チャネ ルを使用しなければならない[SHALL]。

    帯域外デバイス

    帯域外Authenticatorは、一意にアドレス指定可能な物理的デバイスであり、セカンダリ・チャネ ルと呼ばれる別個の通信チャネルを介して検証者と安全に通信できる。このデバイスは、請求者が所有・管理し、認証のためのプライマリ・チャネルとは別の、 このセカンダリ・チャネルを介したプライベート通信をサポートする。帯域外Authenticatorは、"持っているもの "である。

    帯域外認証は、ベリファイアが生成した短期秘密を使用する。この秘密は、プライマリ・チャネルおよびセカンダリ・チャネル上の認証動作を安全にバインドし、帯域外デバイスの請求者の制御を確立する。

    帯域外認証はフィッシングに強くない。

    帯域外認証機能は、以下のいずれかの方法で動作する：

    • 請求者は、セカンダリ・チャネルを介して帯域外デバイスが受信した秘密を、プライマリ・チャネルを使用して検証者に転送する。たとえば、請求者は秘密（通常は6桁のコード）をモバイル・デバイスで受信し、認証セッションに入力することができる。この方法を図 2 に示す。

    • 請求者は、セカンダリ・チャネルを介して検証者に送信するために、プライマリ・チャネ ルを介して受信した秘密を帯域外デバイスに転送する。例えば、請求者は認証セッションで秘密を表示し、それをモバイル・デバイスのアプリに入力するか、バーコードやQRコードなどの技術を使用して転送を実行することができる。この方法を図3に示す。

    帯域外認証の第3の方法は、プライマリチャネルとセカンダリチャネルから受信した秘密を比較し、セカンダリチャネルで承認を要求する。この方法は、要求されたように実際に秘密を比較することなく、加入者が認証要 求を承認する可能性が高くなるため、もはや容認できないと考えられている。これは、攻撃者(請求者)が加入者に多くの帯域外認証要求を生成し、加入者は迷惑を除 くために1つを承認するかもしれない、「認証疲れ」攻撃で観察されている。このため、ベリファイアからプッシュ通知を受信し、単にトランザクションを承認するよう 請求者に求める認証者は（たとえ認証に関する追加情報を提供したとしても）、本節の要 件を満たしていない。

    帯域外認証装置

    帯域外認証機能は、帯域外秘密または認証要求を取得するために、ベリファイアと別のチャネ ルを確立しなければならない（SHALL）。このチャネルは、デバイスが請求者の許可なく一方のチャネルから他方のチャネルに 情報を漏らさないことを条件として、プライマリ通信チャネルに対して帯域外であるとみなさ れる（同じデバイス上で終端する場合であっても）。

    帯域外デバイスは、検証者によって一意にアドレス指定可能であるべきである（SHOULD）。セカンダリチャネルを介した通信は、公衆交換電話網（PSTN）を介して送信されない限り、 承認された暗号化を使用しなければならない（SHALL）。帯域外認証に PSTN を使用する場合に固有のその他のAuthenticator要件については、第 3.1.3.3 節を参照のこと。

    電子メールを帯域外認証に使用してはならない（SHALL NOT）：

    • パスワードのみでアクセス可能
    • 転送中または中間メールサーバーでの傍受
    • DNSスプーフィングなどによるリルート攻撃

    帯域外認証局は、ベリファイアと通信する際、以下のいずれかの方法で自身を一意に認証しなけれ ばならない：

    • 承認された暗号を使用し、検証者に対して相互に認証された保護されたチャネル（ク ライアント認証TLSなど）を確立する。帯域外認証機関とベリファイア間の通信は、それぞれが認証する信頼できる仲介サービスを使用してもよい[MAY]。鍵は、Sec. 4.1に記載されているように、認証者のバインディング中に、相互に認証されたセッ ションでプロビジョニングされなければならない[SHALL]。

    • 加入者を一意に識別するSIMカードまたは同等の秘密を使用して、公衆携帯 電話ネットワークに認証する。この方法は、ベリファイアからPSTN（SMSまたは音声）または暗号化されたインスタントメッ セージングサービスを介して帯域外デバイスに秘密が送信される場合にのみ使用されるものとする[SHALL]。

    • 検証者が電話をかけ、帯域外秘話を口述できるPSTNへの有線接続を使用する。本定義では、「有線接続」には、他の有線メディアやアナログ電話アダプタ経由のファイバーを通じて PSTN サービスを提供するケーブルプロバイダなどのサービスが含まれる。

    単一ファクタの帯域外認証装置では、ベリファイアから帯域外デバイスに秘密が送信された場 合、デバイスは、所有者によってロックされている間は認証秘密を表示すべきではない （すなわち、デバイスは、表示するために PIN、パスコード、またはバイオメトリック特性の提示 と検証を要求すべきである）。ただし、認証者は、ロックされたデバイスで認証秘密を受信したことを示すべきである（SHOULD）。

    帯域外認証機関が、請求者がプライマリ通信チャネルに転送する秘密を提示するのでは なく、セカンダリ通信チャネル上で承認を要求する場合、プライマリチャネルからの秘密の 転送を受け入れ、それをセカンダリチャネル上でベリファイアに送信し、承認を認証トランザ クションに関連付けなければならない（SHALL）。請求者は、手動で、またはバーコードや QR コードなどの表現による支援を受けて、転送を実行してもよい（MAY）。

    帯域外検証装置

    PSTN特有の追加検証要件については、第3.1.3.3項を参照のこと。

    ベリファイアは、帯域外認証機関と認証済み保護チャネルが確立されるのを待ち、その 識別鍵を検証する。ベリファイアは、識別鍵自体を保存してはならない(SHALL NOT)が、 認証者を一意に識別するために、検証方法(例えば、承認されたハッシュ関数や識別鍵の所有証明)を使用しなければならない(SHALL)。認証されると、ベリファイアは認証シークレットを認証者に送信する。帯域外認証者との接続は、手動で開始してもよいし、プッシュ通知などのメカニズムによって促 進してもよい（MAY）。

    ベリファイアは、 第 3.2.12 節に記載されるとおり、承認されたランダムビットジェネレータを使用 して、少なくとも 10 進数 6 桁（またはそれと同等）の長さのランダムな認証秘密を生成しなければなら ない（SHALL）。認証シークレットが 64 ビット未満の場合、ベリファイアは、第 3.2.2 節に記載されるとおり、 サブスクライバ・アカウントで連続して試行できる認証失敗の総数を効果的に制限するレ ート制限メカニズムを実装しなければならない（SHALL）。新しい認証秘密を生成しても、認証失敗回数をリセットしてはならない（SHALL NOT）。

    加入者デバイスにプッシュ通知を送信する帯域外検証機は、最後に認証が成功してから送信されるプッシュ通知 のレートまたは総数について、合理的な制限を実装すべきである（SHOULD）。

    公衆交換電話網を使った認証

    帯域外検証のためのPSTNの使用は、本セクションおよびSec.3.2.9に記載されるとおり制限される。事前に登録された電話番号の設定または変更は、新規Authenticatorのバインディングとみなされ、Sec.4.1.2 に記載されている場合にのみ発生するものとする。

    加入者によっては、電話カバー範囲が限定されている地域（特に携帯電話サービスがない地域） では、帯域外認証秘密を配信するために PSTN を使用できない場合がある。したがって、ベリファイアは、すべての加入者が代替の認証タイプを利用できるようにしなければならず[SHALL]、加入者が管理する 1 つ以上のデバイスをバインドする前に、 PSTN 帯域外認証のこの制限を加入者に思い出させるべきである[SHOULD]。

    検証者は、帯域外認証シークレットの配信に PSTN を使用する前に、リスク指標（デバイスの交換、 SIM の変更、番号ポーティング、その他の異常動作など）を考慮すべきである。

    NIST は、第 3.2.9 節の認証者の制限と同様、脅威状況の進化および PSTN の技術的運用に基づ いて、PSTN を使用する帯域外認証の制限状況を調整することができる。

    多要素帯域外認証装置

    多要素帯域外認証機能は、1 要素帯域外認証機能（第 3.1.3.1 節参照）と同様に動作する。ただし、多要素帯域外認証機能は、請求者が認証トランザクションを完了する前（すなわち、 使用される認証フローに応じて認証シークレットにアクセスまたは入力する前）に、起動要素（パスワー ドまたはバイオメトリック特性）の提示および検証を必要とする。オーセンティケータを使用するたびに、アクティベーション要素の提示を要求するものとする。

    認証者起動秘密は、 第 3.2.10 節の要件を満たさなければならない。バイオメトリクス起動要素は、連続認証失敗回数の制限を含め、第 3.2.3 項の要件を満たさなければな らない（SHALL）。起動に使用されたパスワードまたはバイオメトリック・サンプル、およびバイオメトリック・サンプルに由来するバイオメトリック・データ（信号処理によって生成されたプローブなど）は、認証操作の直後にゼロ化（消去）されなければならない（SHALL）。

    シングルファクターOTP

    シングルファクタ OTP は、ワンタイムパスワード（OTP）を生成する。このカテゴリには、ハードウェア・デバイスと、携帯電話などのデバイスにインス トールされるソフトウェア・ベースの OTP ジェネレータが含まれる。これらの認証機には、OTP を生成するためのシードとして使用される秘密鍵が埋め込まれており、第 2 要素によるアクティベーションを必要としない。OTP は、認証機に表示され、検証者に送信するために手動で入力され、これによって認証機の 所有と管理が証明される。一要素 OTP 認証器は、あなたが持っているものです。

    シングルファクタ OTP は、秘密鍵が暗号化され、認証者と検証者によって独立に生成され、 検証者によって比較される点を除き、ルックアップ秘密認証に似ている。秘密鍵は、時間ベースまたは認証者と検証者のカウンタから計算される nonce に基づいて計算される。

    OTP 認証はフィッシング耐性がない。[FIPS140] によるOTP 認証者および検証者の検証は必須ではない。

    一要素OTP認証機能

    1 要素 OTP 認証および検証には、2 つの永続的な値が含まれる：1) 認証者の有効期間中存続する対称鍵、2) 認証者が使用されるたびに変更されるか、リアルタイムクロックに基 づく非信号。

    秘密鍵およびそのアルゴリズムは、[SP800-131A]の最新リビジョンで規定される最小セキュリティ強度(本書発行日現在112ビット)を少なくとも提供しなければならない(SHALL)。nonceは、その有効期間中、認証者の各操作に対して一意であることを保証するために、十分な長 さを持たなければならない。加入者は、ソフトウェアベースのOTP認証機能が存在するデバイスを変更する必要がある場合、セクション4.1.2に記載されているように、新しいデバイス上の認証機能アプリケーションを 加入者アカウントにバインドし、使用されなくなる認証機能アプリケーションを無効にすべきである。

    認証機関の出力は、鍵およびノンスを安全に結合するために、承認されたブロック暗号またはハッ シュ関数を使用して得られる。ベリファイアと協調して、認証機関は、その出力を、最小 6 桁の 10 進数またはそれと同等 の数に切り捨ててもよい（MAY）。

    Authenticator出力の生成に使用されるnonceがリアルタイムクロックに基づく場合、nonceは少 なくとも2分に1回変更されなければならない[SHALL]。

    シングルファクターOTPベリファイア

    一要素 OTP 検証機は、認証機で使用される OTP 生成プロセスを効果的に複製する。そのため、認証機で使用される共通鍵は、検証機にも存在し、アクセスを必要とするソフ トウェアコンポーネントのみに鍵へのアクセスを制限するアクセス制御により、不正な開示から強 力に保護されなければならない（SHALL）。

    一要素 OTP 認証機能をサブスクライバのアカウントにバインドする場合、ベリファイアまたは関連 CSP は、鍵の生成および交換、または認証機能の出力を複製するために必要な秘密を取得するために、鍵の確立に承認された暗号技術を使用しなけれ ばならない（SHALL）。

    ベリファイアは、OTPを収集する際、承認された暗号化および認証された保護されたチャネ ルを使用しなければならない（SHALL）。ベリファイアは、 Sec. 3.2.7 に記載されるように、リプレイ耐性を提供するために、有効な OTP を一度だけ受け付けるものとする。OTP の重複使用により請求者の認証が拒否された場合、攻撃者が事前に認証できていれば、ベリファイア は請求者に警告してもよい[MAY]。ベリファイアは、OTPの重複使用が試みられたことを、既存セッションの加入者にも警告してもよい[MAY]。

    ベリファイアは、Sec. 3.2.2に記載されているように、Authenticator出力が64ビット未満の場合、加入者ア カウントで試行できる認証失敗回数を効果的に制限するレート制限メカニズムを実装す べきである[SHOULD]か、または実装しなければならない[SHALL] 。

    多要素OTP

    多要素 OTP は、起動要素の入力後に認証用のワンタイムパスワードを生成する。これには、ハードウェア・デバイスや、携帯電話などにインストールされるソ フトウェアベースの OTP ジェネレータが含まれる。第 2 の認証要素は、一体型入力パッド、一体型バイオメトリクス（例えば、指紋）リーダ、 または直接コンピュータインタフェース（例えば、USB ポート）を通じて提供される。OTP は認証機に表示され、検証機に送信するために手動で入力される。多要素 OTP 認証は、「あなたが知っている何か」または「あなたがいる何か」によって起動される 「あなたが持っている何か」である。

    OTP 認証はフィッシング耐性がない。[FIPS140] によるOTP 認証者および検証者の検証は必須ではない。

    多要素 OTP 検証機は、認証者が使用する OTP 生成プロセスを、2 つ目の認証要素を必要とせずに、効果 的に複製する。そのため、認証者が使用する共通鍵は、アクセスを必要とするソフトウェア・コンポーネントのみに 鍵へのアクセスを制限するアクセス制御により、不正な開示から強力に保護されなければならない。

    一要素暗号認証

    一要素暗号認証は、認証プロトコルを介して暗号鍵の所有と制御を証明することで達成される。要求される認証の強度に応じて、秘密鍵または対称鍵は、認証されるエンドポイントにアクセス可能な方法、または鍵がエクスポートできない、直接接続された別のプロセッサまたはデバイスに格納される。Authenticatorの出力は、使用される特定の暗号化プロトコルに大きく依存するが、一般的に、ある種の 署名されたメッセージである。一要素暗号化Authenticatorは、"持っているもの "である。

    暗号化認証は、セクション3.2.5の追加要件を満たす場合、フィッシング耐性がある。

    単一要素暗号認証

    一要素暗号認証方式は、1つ以上の秘密鍵または共通鍵をカプセル化する。鍵は、認証者が利用可能な適切なストレージ(例えば、キーチェー ン・ストレージ)に格納されるべきである(SHOULD)。また、鍵を非エクスポートとする場合、 ハードウェアで保護された隔離された実行環境、またはユーザー・エンドポイントのセントラル・ プロセッシング・ユニットとのインターフェースが制御された別個のプロセッサに格納されなければなら ない(SHALL)。認証されるエンドポイントにアクセス可能な場合、秘密鍵または共通鍵は、アクセスを必要とするソフ トウェア・コンポーネントのみに鍵へのアクセスを制限するアクセス制御を使用することにより、不正な 開示から強力に保護されなければならない（SHALL）。

    外部（すなわち、非組込み）暗号化Authenticatorは、 第 3.2.11 節の接続Authenticatorに対する要件を満たさなけれ ばならない（SHALL）。

    セクション 2.3.2 で要求されるとおり、AAL3 で使用される一要素暗号認証方式は、セクション 3.2.8 の認証意図要件を満たさなければならない。

    単一要素暗号検証機

    一要素暗号検証者は、チャレンジ・ノンスを生成し、それを対応する認証者に送信し、 認証者の出力を使用して認証者の所有を検証する。オーセンティケータ出力は、使用する暗号化オーセンティケータやプロトコルに大きく依存するが、一般的には何らかの署名付きメッセージである。

    検証者は、各認証者に対応する対称または非対称の公開暗号鍵を持つ。いずれのタイプの鍵も改ざんから保護されなければならないが、対称鍵はさらに、鍵へのア クセスを必要とするソフトウェア・コンポーネントのみに制限するアクセス制御によって、不正な 開示から保護されなければならない。

    秘密鍵または共通鍵およびそのアルゴリズムは、少なくとも[SP800-131A]の最新リビジョンで規定され る最小セキュリティ強度(本書の日付時点で112ビット)を提供しなければならない。チャレンジノンスは、少なくとも64ビットの長さでなければならず、認証者の有効期間中一意であるか、 または統計的に一意でなければならない(すなわち、セクション3.2.12に記載されているように、 承認されたランダムビットジェネレータを使用して生成される)。検証操作は、承認された暗号技術を使用しなければならない（SHALL）。

    多要素暗号認証

    多要素暗号認証方式は、起動要素（すなわち、パスワードまたはバイオメトリック特性）の提示および 検証によってのみアクセス可能な、1 つまたは複数の秘密鍵または共通鍵をカプセル化する。鍵は、認証者が利用可能な適切なストレージ(例えば、キーチェーンストレージ)に格納されるべきであ る(SHOULD)。また、鍵が非エクスポートである場合、ハードウェアで保護された隔離された実行 環境、またはユーザーエンドポイントの中央処理装置への制御されたインターフェースを持つ別個のプ ロセッサに格納されなければならない(SHALL)。認証されるエンドポイントにアクセス可能な場合、鍵は、アクセスを必要とするソフトウェア・ コンポーネントのみに鍵へのアクセスを制限するアクセス制御を使用することにより、不正な開示から強 力に保護されなければならない（SHALL）。

    加入者管理型ウォレットでの利用

    多要素暗号化認証の特殊なケースとして、[SP800-63C]のSec.請求者が最初にアクティブ化要素を使用してウォレットのロックを解除した後、認証プロセスは[SP800-63C]に詳述されているように、フェデレーション・プロトコルを使用する。フェデレーション・プロトコルのアサーション・コンテンツおよび提示要件は、暗号化認証機 関に要求されるセキュリティ特性を提供する。そのため、サブスクライバが制御するウォレットは、アクティベーション要素、ウォレットが生成するアサーションの提示と検証を通じて、多要素認証機関とみなすことができる。

    秘密鍵へのアクセスには、起動要素が必要であるものとする。認証機関活性化秘密鍵は、 第 3.2.10 節の要件を満たさなければならない。バイオメトリクス起動要素は、連続認証失敗回数の制限を含め、第 3.2.3 項の要件を満たさなければな らない。アクティブ化に使用されたパスワードまたはバイオメトリクス・サンプル、およびバイオメトリクス・サンプルから派生したバイオメトリクス・データは、認証トランザクションの直後にゼロ化（消去）されなければならない（SHALL）。

    サブスクライバが管理するウォレットを使用する認証プロセスは、[SP800-63C]のセクション 5 に詳述されているフェデレーション・プロセスと共に使用しなければならない（SHALL）。サブスクライバが管理するウォレットによって生成される署名付き視聴者制限付きアサーションは、偽者 RP に提示されたアサーションが正当な RP によって使用されることを防ぐため、フィッシング耐性があるとみなされる。ウォレットからの有効な署名または視聴者制限を欠くアサーションは、フィッシング耐性があるとはみなされないものとする。

    同期可能な認証機能

    多要素暗号化認証方式には、加入者が認証秘密を、通常は同期ファブリックを介して、 追加のデバイスにコピー（クローン）できるものもある。これは、認証に追加のデバイスを使用したい加入者の負担を軽減する。同期可能な認証機関と同期ファブリックに関する具体的な要件は、付録Bに記載されている。

    一般的なAuthenticatorの要件

    以下の要件は、すべてのタイプの認証に適用される。

    物理的認証機能

    CSP は、Authenticatorを盗難または紛失から適切に保護するための利用者の指示を提供するものとする。CSP は、Authenticatorの紛失、盗難、または危殆化が疑われるとサブスクライバから通知があった場合、直ちに Authenticatorを無効にする仕組み1 を提供するものとする。

    物理的Authenticatorの所有および管理は、Authenticatorに関連付けられた秘密の所有の証明に基づく。埋め込まれた秘密（通常、証明書および関連する秘密鍵）がエンドポイントにある場合、その 「デバイス ID」は物理的Authenticatorとみなすことができる。ただし、これには、認証される AAL に適した安全な認証プロトコルが必要である。ブラウザクッキーは、AAL1 または第 5.1.1 節に記述されているようなセッション維持（認証ではない） のための短期秘密としての場合を除き、この要件を満たさない。

    レート制限（スロットリング）

    第 3.1 節のAuthenticatorタイプの記述で要求される場合、ベリファイアは、オンライン推測攻撃 から保護するための制御を実装しなければならない（SHALL）。所定のAuthenticatorに関する記述に別段の定めがない限り、ベリファイアは、1 つの加入者アカウ ントで 1 つ以上の特定のAuthenticatorを使用して連続して認証に失敗した回数を 100 回以下に制限しなけ ればならない。

    100 回という試行回数の制限は上限である。100 回という制限は、正しい推測の可能性（例えば、6 桁の 10 進数 OTP 認証出力に対する 100 回の試行）と、制限を超えた場合のアカウント回復の潜在的な必要性とのバランスを取るために選択された。

    攻撃者がレート制限のために正当な請求者を締め出す可能性を減らすために、 追加の技術を使用してもよい(MAY)。これには以下が含まれる：

    • 認証を試みる前に、ボット検知と緩和のためのチャレンジを完了するよう請求者に要求する。

    • 試行失敗後、加入者アカウントが連続試行失敗の最大許容量に近づくにつれて増加する時間（例えば、30秒から1時間まで）、待機することを請求者に要求すること。

    • 加入者が以前に認証に成功したIPアドレスからの認証要求のみを受け付ける。

    • その他のリスクベースまたは適応型認証技術を活用し、典型的な規範の範囲内または範囲外の ユーザ行動を識別する（例えば、請求者の IP アドレス、地理的位置、リクエストパターンの タイミング、またはブラウザのメタデータを使用する）。

    サブスクライバが認証に成功した場合、ベリファイアは、認証成功時に使用された Authenticatorに対する以前の失敗試行を無視するべきである[SHOULD]。

    所定のAALで認証に成功した後、ベリファイアは、過剰な再試行によってロックアウトさ れたAuthenticatorの再試行カウントをリセットすべきである[SHOULD]。これが提供される場合、リセッ トされるAuthenticatorの最大AALは、リセット元のセッションのAALを超えてはならない[SHALL]。サブスクライバが要求されたAALで認証できない場合、セクション4.2のアカウント回復 手順を使用しなければならない[SHALL]。

    バイオメトリクスの利用

    認証におけるバイオメトリクス（すなわち、自分である何か）の使用には、身体的特徴（例 えば、指紋、虹彩、顔の特徴）の測定と行動的特徴（例えば、タイピングのケイデンス）の測定の 両方が含まれる。どちらのクラスもバイオメトリクス・モダリティとみなされるが、モダリティは、第 3.2.8 節で説明するように、認証の意図を確立する程度において異なる場合がある。

    さまざまな理由から、本文書は、認証のためのバイオメトリクスの限定的な使用のみをサポ ートする。これらの理由には、以下が含まれる：

    • バイオメトリクス誤照合率（FMR）は、それだけでは加入者の認証に十分な信頼性を与えない。さらに、FMR はスプーフィング攻撃を考慮していない。
    • バイオメトリクス比較は確率的であるのに対し、他の認証要素は決定論的である。
    • バイオメトリック・テンプレート保護スキームは、他の認証要素（例えば、PKI 証明書やパスワー ド）に匹敵するバイオメトリック特性を失効させる方法を提供する。しかし、このようなソリューションの利用可能性は限られている。
    • バイオメトリクスの特性は秘密ではない。それらはしばしばオンラインで、あるいは顔画像の場合は本人が知っているかどうかにかかわらず、写真を撮ることで入手することができる。潜在指紋は、誰かが触れた物体から採取することができ、虹彩パターンは高解像度画像でキャプチャすることができる。プレゼンテーション攻撃検出（PAD）技術はこの種の攻撃のリスクを軽減することができるが、PAD が CSP と加入者のニーズに従って動作していることを保証するには、センサーまたはバイオメトリック処理に対するさらなる信頼が必要である。

    したがって、認証のためのバイオメトリクスの限定的な使用は、以下の要件およびガ イドラインによってサポートされる。

    バイオメトリクスは、物理的Authenticator（すなわち、「持っているもの」）を使用する多要素認証の一部 としてのみ使用しなければならない（SHALL）。バイオメトリクスの特性は、認証操作ごとに提示され、比較されなければならない（SHALL）。バイオメトリクス以外の代替認証オプションは、常に加入者に提供されなければならない（SHALL）。バイオメトリクス・データは、機微な PII として扱われ、保護されなければならない。

    バイオメトリックシステムは、10000 分の 1 以上の FMR[ISO/IEC2382-37]で動作しなければならない。この FMR は、[ISO/IEC30107-1]で定義されているような適合攻撃（すなわち、労力ゼロの 偽者の試み）の条件下で達成されなければならない（SHALL）。バイオメトリクス・システムは、5%未満の偽の非一致率（FNMR）を示すべきである（SHOULD）。バイオメトリック性能は、[ISO/IEC19795-1]に従ってテストされなければならない（SHALL）。

    バイオメトリクス認証技術は、異なる人口統計学的タイプ（例えば、人種的背景、性別、民族 性）の加入者に対して、同様の性能を提供しなければならない（SHALL）。

    バイオメトリックシステムは、PAD を実装するべきである（SHOULD）。バイオメトリックシステムを配備するためのテストは、0.15 未満の偽者攻撃提示受理率（IAPAR）を実証す べきである（SHOULD）。提示攻撃耐性は、[ISO/IEC30107-3]の第 13 節に従ってテストされなければならない。PADの判定は、請求者のデバイス上でローカルに行なっても、中央検証者が行なっても よい[MAY]。

    バイオメトリクス・システムは、連続 5 回までの認証の失敗、または PAD が実装され上記の要 件を満たしている場合は連続 10 回までの認証の失敗を許容しなければならない。この制限に達すると、バイオメトリッ ク認証システムは、後続の各試行までに少なくとも 30 秒の遅延を課し、全体として連続 50 回以内の認証失敗を許容しなければならない。全体的な制限に達すると、バイオメトリクス・システムは、バイオメトリクス・ユーザ認証を 無効にし、そのような代替方法がすでに利用可能な場合は、別の要因（例えば、別のバイオメトリ ック・モダリティまたは必須要因でない場合は起動秘密）を提供しなければならない（SHALL）。これらの限界は上限であり、機関はリスクに基づいて下限を課す決定を行うことができる。

    ベリファイアは、センサーと関連するエンドポイントの性能と完全性を判断すべきである（SHOULD）。この判定を行うための許容可能な方法には、以下が含まれるが、これらに限定されない：

    • センサー認証によって決定される、既知のセンサーの使用
    • バイオメトリクスの性能基準に対するファースト・テストまたはサード・パーティ・テスト
    • 第3.2.4節に記述されているように、署名されたメタデータの実行時問い合わせ（例えば、認証）。

    バイオメトリクス比較は、請求者が使用するデバイス上でローカルに実行することも、中央検証機 で実行することもできる。中央検証機関では、より大規模な攻撃を受ける可能性が高いため、比較はローカルで行うべ きである（SHOULD）。

    認証機能起動のためのバイオメトリック因子の提示は、ホスト機器（スマートフォンなど）のロック解 除とは別の操作でなければならない（SHALL）。ただし、ホスト・デバイスのロック解除に使用されたのと同じアクティベーショ ン係数を、認証操作に使用してもよい。機関は、（モバイル・デバイス管理などを通じて）CSP によって、または CSP に代わって管理され、機関 が決定した短い非アクティブ・タイムアウトを持つように制約された認証機関と、上記の要件を満たすバイオメトリッ ク・システムについては、この要件を引き下げてもよい。

    比較が中央で行われる場合：

    • センサーまたはエンドポイントは、請求者からバイオメトリクス・サンプルを採取する前に、 認証されなければならない（SHALL）。検証者は、中央保存されたバイオメトリック・テンプレートの使用を、特定のセンサーまたはセ ンサー・クラス（例えば、センサー・メーカーまたはモデル）に制限してもよい。
    • 機微な PII に対する適切な管理（暗号化やアクセス管理など）を実施しなければならない。
    • センサー（またはセンサーの交換に抵抗するセンサーを含むエンドポイント）と検証者の間に、 認証された保護されたチャネルを確立しなければならない（SHALL）。すべてのバイオメトリクス情報の送信は、その認証された保護されたチャネルを介して行わなければならない（SHALL）。

    認証プロセスで収集されたバイオメトリックサンプルは、比較アルゴリズムのトレーニング（た とえば、加入者の特性の変化に対応するためのテンプレートの更新）、または加入者の同意があ れば、その他の研究目的に使用することができる。バイオメトリクス・サンプルおよびバイオメトリクス・サンプルから派生したバイオメトリクス・デ ータは、トレーニングまたは研究データが派生した後、直ちにゼロ化（消去）されなければなら ない（SHALL）。

    認証

    CSP は、署名された証明書を含むなど、認証者の特性を評価するための信頼できる根拠を持つ 必要がある。認証とは、通常、認証者がバインドされるときに、接続されている認証者または認証操作に関与するエンドポイ ントに関して CSP に伝達される情報である。認証によって伝達される情報には、以下を含んでもよいが、これらに限定されない：

    • 認証機関およびエンドポイントの出所（製造業者または供給業者の認証など）、健全性、完全性
    • 認証機関のセキュリティ機能
    • バイオメトリックセンサーのセキュリティと性能特性
    • センサーのモダリティ

    証明書は、[SP800-131A]の最新リビジョン（本書の日付時点で112ビット）で規定される最小限のセキュリ ティ強度を提供するデジタル署名を使用して署名されなければならない（SHALL）。

    連邦エンタープライズ・システム2のベリファイアは、認証者の能力と出所を検証するために、 認証機能を使用すべきである（SHOULD）。その他の用途では、認証情報は、ベリファイアのリスク・ベースの認証判断の一部として使用してもよい（MAY）。

    フィッシング（検証者なりすまし）への耐性

    フィッシング攻撃は、以前 SP 800-63B で「検証者のなりすまし」と呼ばれていたが、詐欺的な検証者や RP による試みであり、不用心な請求者を騙して偽者に認証者を提示させるものである。SP 800-63 の以前のバージョンでは、フィッシング攻撃に耐性のあるプロトコルは「強力な MitM 耐性」とも呼ばれていた。

    フィッシングという用語は、さまざまな類似の攻撃を表すために広く使用されている。本文書では、フィッシング耐性とは、請求者の警戒心に頼ることなく、偽者ベリファイアへの認証秘 密および有効なAuthenticator出力の開示を防止する認証プロトコルの能力である。請求者がどのように偽者検証者に誘導されるかは関係ない。例えば、請求者が検索エンジンの最適化によってそこに誘導されたか、電子メールによって促されたかに関係なく、それはフィッシング攻撃であるとみなされる。

    承認された暗号アルゴリズムは、必要な場合、フィッシング耐性を確立するために使用されなけれ ばならない(SHALL)。この目的のために使用される鍵は、[SP800-131A]の最新リビジョンで規定されている最小セキュリティ強度(本書の日付現在112ビット)を少なくとも提供しなければならない(SHALL)。

    フィッシング耐性には、一要素または多要素の暗号化認証が必要である。Authenticator出力を手動で入力するAuthenticator(例えば、帯域外AuthenticatorやOTPAuthenticator)は、手動入力が認証される特定のセッションにAuthenticator出力をバインドしないため、フィッシング耐性がない。たとえば、偽者のベリファイアは、認証者出力をベリファイアに中継し、認証に成功する可能性がある。

    フィッシングに対抗する方法として、チャネル・バインディングと検証者名バインディングの2つが認められている。チャネル・バインディングはベリファイア証明書の誤発行や流用に脆弱でないため、ベリファイア名バインディングよりも安全であると考えられているが、どちらの方法もフィッシング耐性の要件を満たしている。

    チャンネル・バインディング

    チャネルバインディングを伴う認証プロトコルを使用して、ベリファイアと認証済み保護チャネ ルを確立しなければならない[SHALL]。プロトコルは、認証された保護されたチャネルを確立する際にネゴシエートされた チャネル識別子を、認証者の出力に強く不可逆的にバインドしなければならない(例えば、 公開鍵がベリファイアに知られている請求者が管理する秘密鍵を使用して、2つの 値を一緒に署名する)。ベリファイアは、フィッシングへの耐性を証明するために使用される署名またはその他の情報を 検証しなければならない(SHALL)。これにより、偽のベリファイアが、たとえ実際のベリファイアを代表する証明書を取得し ていても、別の認証済み保護チャネル上でその認証の中継に成功することを防ぐことができる。

    チャネルバインディングを使用するフィッシングに耐性のある認証プロトコルの例は、クライアント認証TLS[TLS]である。

    ベリファイア名バインディング

    ベリファイア名バインディングを伴う認証プロトコルは、ベリファイアと認証済み保護チャネ ルを確立するために使用されなければならない[SHALL]。プロトコルは、プロトコルの一部として認証されたベリファイア識別子に暗号 的にバインドされたAuthenticator出力を生成しなければならない[SHALL]。ドメインネームシステム(DNS)識別子の場合、ベリファイア識別子は、ベリファイア の認証済みホスト名か、そのホスト名に関連する公開サフィックス[PSL]の少なくとも 1レベル下の親ドメインのいずれかでなければならない[SHALL]。バインディングは、関連する認証者秘密を選択する、ベリファイア識別子を使用して 認証者秘密を導出する、認証者出力にベリファイア識別子で暗号署名する、 または同様の暗号的に安全な手段を使用して確立してもよい(MAY)。

    W3CのWebAuthn[WebAuthn]は、FIDO2仕様[FIDO2]を実装するオーセンティケータによって使用され、ベリファイア名のバインディングによってフィッシング耐性を提供する標準の例である。

    検証者とCSPのコミュニケーション

    検証者と CSP が別個のエンティティである場合（[SP800-63]の図 3 に点線で示されているように）、検証者と CSP の間の通信は、承認された暗号を使用して、相互に認証された安全なチャネル（例えば、クライアント認証された TLS 接続）を介して行われるものとする 。

    リプレイ・レジスタンス

    認証プロセスは、以前の認証メッセージを記録して再生することで認証を成功させることが現実的でない場合、リプレイ攻撃に抵抗する。リプレイ耐性は、認証された保護チャネルプロトコルのリプレイ耐性の性質に加えて、 保護チャネルに入る前に出力が盗まれる可能性があるためである。トランザクションの「新しさ」を証明するためにnoncesまたはchallengeを使用するプロトコルは、適切なnoncesまたはtimelinessデータを含まないため、古いプロトコルメッセージがリプレイされたときに検証者が容易に検出できるため、リプレイ攻撃に強い。

    リプレイ耐性Authenticatorの例には、OTPAuthenticator、暗号化Authenticator、ルックアップ秘密鍵が含まれる。

    対照的に、パスワードは、認証のたびに認証者の出力、つまり秘密そのものが提供されるため、リプレイ耐性があるとはみなされない。

    認証の意図

    認証プロセスは、各認証または再認証要求に対して明示的に応答することを要求する場合、 意図を示す。認証の意図の目的は、エンドポイント上のマルウェアなど、請求者が知らないうちに認証機 能（多要素暗号化認証機能など）が使用されることを困難にすることである。多要素暗号化認証機能は、認証機能の起動要素を再入力することで、認証意思を確立してもよ いが(MAY)、認証機能自体が認証意思を確立しなければならない(SHALL)。

    認証の意図は、いくつかの方法で立証してもよい（MAY）。請求者の介入を必要とする認証プロセスを使用して、意思を証明することができる（例： OTP 認証装置から出力された認証情報を入力する請求者）。各認証または再認証操作にユーザーのアクションを必要とする暗号化認証機も、意思の立証に 使用できる（例えば、ボタンを押す、または再挿入する）。

    バイオメトリクス特性の提示は、常に認証の意図を立証するわけではない。たとえば、携帯電話の前面カメラを使用して顔バイオメトリクスをキャプチャしても、デバイス が他の非認証目的に使用されている間に顔画像がキャプチャされることが合理的に予想されるため、 意図とは見なされない。このようなシナリオでは、認証の意図を確立するための明示的なメカニズム（ソ フトウェアまたは物理ボタンのタップなど）を提供しなければならない（SHALL）。

    制限されたAuthenticator

    脅威が進化するにつれて、認証者の攻撃に対する耐性は通常低下する。逆に、一部の認証機関の性能は、その基礎となる規格の変更により、特定の攻撃への抵抗能力が高まった場合など、向上する可能性がある。

    このようなAuthenticator性能の変化を考慮するため、NIST は、AuthenticatorタイプまたはAuthenticatorタ イプの特定のクラスまたはインスタンスに、さらなる制限を設けている。これらの認証方式は、多要素認証の安全性には劣るが、一部の政府対公共アプリケーションには、制限付き認証方式が依然として必要である。

    制限された認証方式を受け入れるには、実施組織がその認証方式に関連するリスクを評価、 理解、受容し、時間とともにリスクが増大する可能性が高いことを認識する必要がある。システムおよび関連データについて許容可能なリスク・レベルを決定し、過度のリスクを軽減する ための方法を定義し、それらの決定を検証者に伝えることは、RP の責任である。RP が、いずれかの当事者に対するリスクが許容できないと判断した場合、制限付き認証機 器を使用してはならず、代替の認証器タイプを使用しなければならない。

    さらに、認証エラーのリスクは通常、実施組織、認証の判断に依存する組織、およびサブスクライバなど、複数の 当事者によって負担される。組織が制限付き認証者を受け入れた場合、サブスクライバは追加のリスクにさらされる可能性があり、サブスクライバはそのリスクに対する理解やリスクを制御する能力が限られている可能性があるため、CSP は以下のすべてを実行しなければならない（SHALL）：

    1. 要求される AAL での認証に使用できる、制限のない少なくとも 1 つの代替認証手段を加入者に提 供する。

    2. 制限されたAuthenticatorのセキュリティリスクと制限のない代替手段の利用可能性について、利用 者に有意義な通知を提供すること。

    3. リスク評価において、加入者とRPに対する追加リスクに対処する。

    4. 将来、制限付き認証方式が受け入れられなくなる可能性に備えて移行計画を策定し、この移 行計画をデジタル ID アクセプタンスステートメントに記載する（[SP800-63]の 3.4.4 項を参照）。

    アクティベーションの秘密

    多要素認証機能のアクティブ化要素としてローカルで使用されるパスワードは、アクティブ化 秘密（activation secret）と呼ばれる。起動シークレットは、保存された認証キーへのアクセスを取得するために使用される。すべての場合において、アクティブ化シークレットは、認証機および関連するユーザ・エンドポイント内に残らなければならない（SHALL）。

    アクティベーションシークレットを使用する認証機関は、シークレットの長さを4文字以上とすることを義務付けなければならず(SHALL)、シークレットの長さを6文字以上とすることを義務付けるべきである(SHOULD)。アクティブ化秘密は、すべて数字(PINなど)でもよい[MAY]。英数字が許可される場合、すべての印字用ASCII[RFC20]文字とスペース文字が許可されるべきである (SHOULD)。Unicode[ISO/ISC 10646]文字も英数字の秘密鍵で許可されるべきである(SHOULD)。認証機関またはその管理ツールは、利用者が一般的に使用される起動秘話(例: 123456)を選択しないように、ブロックリストを実装するべきである(SHOULD)。

    認証機またはベリファイアは、認証機を使用した連続した起動失敗回数を10回以下に制限する再試行 制限メカニズムを実装しなければならない(SHALL)。不正な活性化秘密の入力が原因で、認証者が中央検証者に無効な出力を提供する場合、検証者は、 この再試行制限メカニズムを実装してもよい[MAY]。そうでない場合、再試行制限は認証局に実装されなければならない[SHALL]。試行回数が制限に達すると、認証器は無効化され、認証のために別の認証器が要求されなけれ ばならない[ SHALL ]。

    AAL3 で使用可能なAuthenticatorの場合、起動秘密の検証は、ハードウェアで保護された環境（セキュア・エレメント、TPM、TEE など）で実行しなければならない（SHALL）。AAL2 において、ハードウェアで保護された環境が使用されない場合、認証者は、認証鍵を復号 するために使用される鍵を導出するために、活性化秘密を使用しなければならない（SHALL）。

    活性化因子の提出は、ホストデバイス（スマートフォンなど）のロック解除とは別の操作としなけれ ばならない（SHALL）。ただし、ホスト・デバイスのロック解 除に使用されたのと同じアクティベーション係数を、認証操作に使用してもよい（MAY）。機関は、機関が決定した短い非アクティブ・タイムアウトと、本セクションの対応する要件を満たすデバイスのアクティベー ション・ファクタを持つように制約されている、（例えば、モバイル・デバイス管理によって）CSP によって、または CSP に代わって管理される認証機関については、この要件を引き下げてもよい（MAY）。

    接続された認証機能

    暗号化認証手段は、認証手段と認証されるエンドポイントとの間に、盗聴、インジェクション、リレ ー攻撃に対する耐性を提供する信頼できる接続を必要とする。この接続は、有線接続（USB またはスマートカードとの直接接続など）、無線技術、またはネットワーク 接続を含むこれらの技術のハイブリッドを使用して行われなければならない（SHALL）。

    承認された暗号は、暗号操作が必要とされるすべての場合に使用されなければならない（SHALL）。認証機関とエンドポイント間の認証データの通信はすべて、これらの機器間で直接行われるか、ま たは認証機関とエンドポイント間の認証済み保護チャネルを介して行われなければなら ない（SHALL）。

    有線接続

    有線接続は、組み込み認証機能を含むが、攻撃対象が最小限であるため、信頼できると仮定しても よい(MAY)。申請者は、信頼できるハードウェア(例えば、ケーブル、アダプタなど)を使用し、それらが危殆化 されていないことを確認するようアドバイスされるべきである[SHOULD]。

    ワイヤレスとハイブリッド接続

    ワイヤレスおよびネットワークベースのAuthenticator接続は、盗聴、インジェクション、リレー攻撃な どの脅威に対して潜在的に脆弱である。無線接続に対するこのような攻撃の可能性は、技術の有効範囲に依存する。認証者とエンドポイントの接続に対する脅威の攻撃面を最小化するために、認証プロセスは、200 メートルを超えない範囲の無線接続を確立することにより、認証者とエンドポイント間の物理的な近接を必要としなければならない（SHALL）。

    ワイヤレス接続およびハイブリッド接続は、以下のいずれかの方法で、認証局とエンドポイント間の暗号化通信のための鍵を確立しなければならない（SHALL）：

    1. 機器間の一時的な有線接続を介して。

    2. アソシエーション・プロセス（ペアリング・プロセスに似ているが、デバイス間の永続的な 関係を必要としない）を通して、認証局とエンドポイント間の暗号化通信のための鍵を確立する。アソシエーションプロセスでは、デバイス間でペアリングコード3または他の共有秘密を採用しなけれ ばならない（SHALL）。認証機またはエンドポイントのいずれかが、デバイスに印刷されてもよいペアリングコードを持たなければならない (MAY)。ペアリングコードの長さは、少なくとも10進数6桁（または同等）でなければならない（SHALL）。ペアリング・コードは、手動入力、または QR コードまたは光学的に通信される同様の表現を使用して、デバイス間で伝達されなければならない（SHALL）。

    有効範囲が1メートル未満の無線技術（例えば、NFC）を使用する場合、エンドポイントから認証機 器に送信される起動秘密は、機器間で確立された鍵を使用して暗号化されなければならない（SHALL）。認証された接続が使用されるべきである（SHOULD）。認証者が認証されたペアリングを要求するように設定されている場合、ペアリングコードを使用しなけれ ばならない(SHALL)。

    認証トランザクション全体ではなく、アクティブ化シークレットのみを暗号化すると、RP の ID などの機密情報が漏洩する可能性がある。PII を含み、認証されたペアリングを必要としないAuthenticatorには、特に注意が必要である。スキミング」攻撃や盗聴攻撃から情報を保護するために、暗号化を使用すべきで ある(SHOULD)。

    有効範囲が 1 メートル以上のワイヤレス技術（例：Bluetooth LE）およびネットワーク接続は、認証者とエンドポイント間で認証された暗号化接続を使用しなければならない（SHALL）。認証トランザクション全体は暗号化されなければならない（SHALL）。この例には、[SP800-73]で規定される仮想コンタクトインターフェースで使用されるペ アリングコードや、[CTAP2.2]プロトコルで規定されるハイブリッドトランスポートが含まれる。

    アソシエーションプロセスによって確立される鍵は、一時的(すなわち、限られたトランザクショ ン数または時間制限付きで有効)または永続的のいずれかである。エンドポイントが永続的な鍵を削除する仕組みを提供しなければならない[SHALL]。

    ランダム値

    乱数値は、ノンスや認証秘密など、認証プロセスで広く使用されている。別段の規定がない限り、本セクションを参照する乱数値は、[SP800-131A]の最新リビジョンで規定された最小セキュリティ強度（本書の日付時点で112ビット）以上を提供する、承認された乱数ビット生成器[RBG]4によって生成されなければならない（SHALL）。

    輸出可能性

    輸出可能性とは、認証者がその認証秘密（秘密鍵または対称鍵）を別のエンドポイントまたは 認証者と共有する能力のことである。一般に、エンドポイント上のソフトウェア（おそらくマルウェア）が認証秘 密にアクセスして漏えいする可能性があるため、認証秘密にアクセスできるエンドポイントは、 エクスポート可能とみなされる。非エクスポート可能な認証器は、より安全であると考えられ、したがって、AAL3 では非エクスポート可能な暗号化認証器が必要とされる。同期可能なAuthenticatorは、本質的にエクスポート可能である（付録Bを参照）。

    非エクスポート可能であるとみなされるためには、Authenticatorは、別個のハードウェア、または組込 みプロセッサもしくは実行環境（例えば、セキュアエレメント、TEE、またはトラステッドプラットフォームモ ジュール）のいずれかでなければならない（SHALL）。これらのハードウェア認証機能および組込みプロセッサは、ラップトップやモバイルデバイスの CPU のようなホストプロセッサとは別個のものである。非エクスポート型認証機能は、認証シークレットのホスト・プロセッサへのエクスポートを禁止す るように設計されなければならず（SHALL）、シークレットを抽出できるようにホスト・プロセッサに よって再プログラム可能であってはならない（SHALL NOT）。Authenticatorは、Authenticatorが使用される AAL の適用される[FIPS140]要件（適用される耐タンパー性要件 を含む）に従う。

    1. 無効化には、PKI ベースのAuthenticatorの失効、加入者アカウントからの削除など、いくつかの形態がある。

    2. 連邦企業システムには、政府請負業者、政府職員、およびミッション・パートナーなど、 PIV ガイダンスの対象範囲とみなされるものが含まれる。これには、政府対消費者または公共向けのユース・ケースは含まれない。

    3. 本節で使用するペアリングコードという用語は、持続的なペアリング処理（例えばBluetooth）が必ずしも使用されることを意味するものではない。

    4. 乱数値の生成に関する詳細な情報は、[SP800-90A]、[SP800-90B]、[SP800-90C]で構成されるNIST SP 800-90文書群に記載されています。

    認証イベント管理

    このセクションは規範的なものである。

    イベントは、サブスクライバのAuthenticatorの有効期間中に発生し、その使用に影響を与える。これらのイベントには、バインド、保守、紛失、盗難、危殆化、不正複製、期限切れ、および失効が含まれる。本セクションでは、これらのイベントに対する対応について説明する。

    認証バインディング

    Authenticatorバインディングとは、特定のAuthenticatorとサブスクライバアカウントとの関連付けを確立し、 Authenticatorがサブスクライバアカウントを認証できるようにすることである。

    Authenticatorは、いずれかによってサブスクライバ・アカウントにバインドされなければならない（SHALL）：

    • 登録の一部として CSP から発行されること。
    • CSP が許容するサブスクライバ提供のAuthenticatorを使用する。

    SP 800-63 の一連のガイドラインは、両方のオプションに対応するため、Authenticatorの発行ではなくバインドに言及している。

    デジタル ID のライフタイムを通じて、CSP は、各利用者アカウントにバインドされている、またはバインドされたことのあるすべての認証機能の記録を保持するものとする。CSP は、ベリファイアが各 AAL での要件への準拠を評価できるように、バインドされている認証機 能の特性（1 要素か多要素か、フィッシング耐性があるかないかなど）を判断しなければならない。この判定は、強力な証拠（認証者の証明など）、認証者を発行した当事者からの直接の情報、または認証者実装の典型的な特性（ユーザ検証ビットが設定されているかどうかなど）に基づいてもよい。

    CSP は、認証機関の検証要件を満たすために必要なその他の状態情報も保持するものとする。たとえば、第 3.2.2 節で説明する認証試行の絞り込みでは、CSP または検証者が、認証機で検証され たアクティブ化要素を除き、最近失敗した認証試行に関する状態情報を保持することが求められる。

    CSP によって作成されるレコードには、重要な認証機能のライフサイクル・イベント（サブスクライバ・アカウントへのバインディング、更新、アップデート、有効期限切れなど）の日時が含まれるものとする。レコードには、イベントに関連するデバイスのバインディング・ソース（IP アドレス、デバイス識別子など） に関する情報を含めるべきである（SHOULD）。

    バインディング・プロセスの一環として、CSP は、要求された AAL に適しているかどうかを判断するた めに、新しい認証機能またはその関連エンドポイントに関する追加情報を要求してもよい（MAY）。

    登録時のバインディング

    登録時のバインディングは登録プロセスの一部とみなされ、[SP800-63A]で議論されている。

    登録後のバインディング

    追加のAuthenticatorをバインドする

    アカウント回復の必要性を最小限に抑えるため、CSP およびベリファイアは、少なくとも 2つの別個の認証手段を維持するよう加入者に奨励すべきである（SHOULD）。たとえば、通常、物理的認証手段として OTP 認証手段を使用するサブスクライバは、物理的認証手段を紛失、 盗難、または破損した場合に使用するために、ルックアップ秘密認証手段を発行するか、帯域外認証 用デバイスを登録してもよい（MAY）。パスワードの置き換えに関する詳細については、第 4.2 節を参照のこと。

    したがって、CSP は、1 つのサブスクライバ・アカウントに複数の認証機能をバインドすることを許可す べきである（SHOULD）。新しいAuthenticatorがサブスクライバ・アカウントにバインドされる場合、CSP は、サブスクライバ・アカウントで現在使用可能な最大 AAL または新しいAuthenticatorが使用される最大 AAL のいずれか低い方の AAL での認証がプロセスで要求されるようにしなければならないものとする。たとえば、AAL2 での使用に適したAuthenticatorをバインドする場合、サブスクライバアカウントが現在 AAL1 認証能力しか持たない場合を除き、AAL2 での認証が必要となる。認証機能が追加された場合、CSP は、 第 4.6 節で説明するように、新しい認証機能をバインドするトランザク ションとは独立したメカニズムを介してサブスクライバに通知するものとする。

    外部認証バインディング

    外部Authenticatorバインディングとは、認証されたエンドポイントに接続されていない、または組み込まれていないAuthenticatorを加入者アカウントにバインドすることを指す。このプロセスは通常、新しいエンドポイントに組み込まれたAuthenticatorを追加する場合、または接続性の制限によって新しくバインドされたAuthenticatorが認証済みエンドポイントに接続できない場合に使用される。

    バインディング処理は、以下のいずれかの方法で進めなければならない[SHALL]：

    • CSP に対して認証を行ったエンドポイントは、CSP にバインディング・コードを要求する。バインディング・コードは、新しいAuthenticatorに関連付けられたエンドポイントに入力され、CSP に送信される。

    • 新しいAuthenticatorに関連付けられたエンドポイントは、CSP からバインディング・コードを取得する。バインディング・コードは認証済みエンドポイントに入力され、CSP に送信される。

    4.1.2.1項および4.2項の要件に加え、外部認証機関をバインドする場合、以下の要件が 適用されなければならない[SHALL]：

    • 認証された保護されたチャネルは、新しいAuthenticatorに関連付けられているエンドポイントと CSP によって確立されなければならない（SHALL）。

    • サブスクライバは、新しい認証機能に関連付けられたエンドポイント上で、CSP が自己を識別する 識別子を入力するようプロンプトされてもよい（MAY）。

    • CSP は、 Sec.3.2.12に記載されているように、承認済みのランダム・ビット・ジェネレータを使用してバイン ディング・コードを生成し、新しい認証者エンドポイントまたはバインディングを承認する認証済みエンドポ イントのいずれかに送信するものとする。バインディングコードは、前のステップで入力された識別子と共に使用される場合、 少なくとも40ビットの長さにしなければならない[SHALL]。そうでない場合、少なくとも112ビットの長さのバインディングコードを要求 されなければならない[SHALL]。

    • 加入者はバインディングコードをもう一方のエンドポイントに転送しなければならない[SHALL]。この転送は、手動またはローカルの帯域外の方法(例: QRコード)のいずれかでなければならない[SHALL]。バインディングコードは、安全でないチャネル(電子メールなど)を介して伝達されてはならない[SHALL NOT]。

    • バインディングコードは1回のみ使用可能で、最大10分間有効でなければならない（ SHALL）。

    • 新しい認証機能のバインディング（または PKI ベースの認証機能の場合は証明書の発行）に続いて、 CSP はサブスクライバに、プロセスが正常に完了したことを確認するために、新しい認証機能で認証するよう促すべきである（SHOULD）。

    • CSP は、Authenticatorのバインディングが誤った場合にサブスクライバが取るべき行動（例えば、誤ってバインドされ たAuthenticatorを迅速に無効化できるように、押せるボタンや使用できる連絡先アドレスなど）について、適切な場 合に明確な指示を提供するものとする。これは、セクション4.6で述べられているバインディング通知に加えて、 認証セッションで提供されてもよい[MAY]。

    外部Authenticatorのバインディングは、加入者が攻撃者に騙されてバインディングコードを使用したり、 攻撃者にバインディングコードを提供したりする可能性があるため、リスクが生じる可能性がある。場合によっては、信頼できるソース（例えば、認証されたセッション、特にその認証が耐フィッシング性 である場合）から取得された表現（例えば、QR コード）は、一般的にバインディング・コードに加えて CSP の URL を含むため、そのような攻撃に対してより堅牢であると考えられる。その結果、加入者が騙されてフィッシング・サイトでバインディング・コードを入力する可能性が低くなる。

    加入者が提供する認証機能へのバインディング

    加入者は、特定の AAL での認証に適したAuthenticatorをすでに所有している場合がある。たとえば、ソーシャル・ネットワーク・プロバイダの多要素認証機能（身元確認なしで AAL2 と見なされる）を持っており、IAL2 を必要とする RP でその認証機能を使用したい場合などである。この場合、おそらく別の CSP による IAL2 での 身元確認と、その CSP での登録時のAuthenticatorのバインドが必要になる。

    CSP は、現実的である場合、サブスクライバが提供するAuthenticatorに対応し、多数のAuthenticatorを管理するサブスク ライバの負担を軽減すべきである（SHOULD）。これらのAuthenticatorのバインディングは、 第 4.1.2 節の説明に従って行われるものとする。Authenticatorの強度が自明でない場合（例えば、あるタイプの 1 要素Authenticatorと多要素Authenticatorの間）、CSP は、より強 いAuthenticatorが使用されていることを証明できない限り（例えば、Authenticatorの発行者または製造者との検証によって）、より弱 いAuthenticatorが使用されていると見なされるものとする。

    リニューアル

    サブスクライバは、既存のAuthenticatorの有効期限が切れる前に、新規または更新された Authenticatorをバインドすべきである[SHOULD]。このプロセスは、 第 4.1.2 節に記載される追加Authenticatorのバインド・プロセスに準 拠すべきである（SHOULD）。CSP は、更新プロセスの一部として、または個別に、その他のアクション（連絡先アドレスの確認など）を 定期的に実行してもよい（MAY）。交換用Authenticatorの使用が成功した後、CSP は期限切れのAuthenticatorを無効にすべきである（SHOULD）。

    アカウント回復

    アカウントの回復とは、加入者が、希望する AAL で認証するために必要なAuthenticatorの制御を失っ た状態から回復することである。これは、身元確認プロセスの一部を繰り返すか、加入者アカウントにバインドされ ている加入者がまだ利用可能なAuthenticatorを使用することと併用して、1 つまたは複数の回復コードを提示す ることで達成できる。これが完了すると、加入者は、加入者アカウントに 1 つ以上の新しいAuthenticatorをバインドできる。アカウント回復イベントは、アカウント回復の不正使用を検出するために、常に 1 つ以上の通知を加入者に送信する。

    アカウント回復は、認証とはいくつかの点で異なる。アカウント回復が呼び出されることはめったにないため、一般に認証よりも利便性が低く、状況や CSP が提供する回復方法によっては、待ち時間が長くなることがある。

    アカウント回復方法

    アカウント回復方法の 4 つの一般的なクラスが認識される。CSP は、以下の 1 つ以上をサポートしなければならない（SHALL）：

    • 保存されたリカバリーコード
    • 発行されたリカバリーコード
    • リカバリー・コンタクトの使用
    • 繰り返される身分証明

    これらの方法に加えて、CSP は、サブスクライバのアカウントを回復するためのアプリケーション固有の方法 （例えば、CSP エージェントとの対話）をサポートしてもよい（MAY）。代替方法の使用は、リスク分析に基づき、CSP によって文書化されるものとします（SHALL）。

    保存されたリカバリーコード

    登録時に、このリカバリ・オプションをサポートする CSP は、サブスクライバにリカバリ・コードを発行す べきである（SHOULD）。回復コードには、承認されたランダム・ビット・ジェネレータによる少なくとも 64 ビットを含めなければならない（SHALL）。保存されたリカバリ・コードは、手動入力用の数字または英数字（Base64 など）として、またはリカバリ・コードを含む機械読み取り可能な光学ラベル（QR コードなど）として提示される。加入後の任意の時点で、加入者は回復コードの交換を要求してもよい（MAY）。交換用回復コードの発行により、第 4.6 節に記載されるように、アカウント回復通知が行われるも のとする。

    保存されたリカバリコードは、オフラインで維持され（例えば、印刷または書き留め）、将来の使用のためにサブスクライバによって安全に保存されることを意図している。保存された回復コードの検証は、 第 3.2.2 項のスロットリング要件に従うものとする。保存された回復コードは、第 3.1.1.2 項に記載されるように、承認された一方向性関数を使用して、ハッシュ化さ れた形で加入者アカウントに保存されなければならない。保存されたリカバリ・コードの使用後、CSP はそのリカバリ・コードを無効にし、新しい保存されたリカバリ・ コードをサブスクライバに発行しなければならないものとする。

    発行されたリカバリーコード

    このオプションをサポートする CSP では、加入者が 1 つ以上の回復アドレス（郵便、電子メール、テキスト・メッ セージ、または音声など）を保持することができる。回復が必要な場合、回復コードが請求者が選択したアドレスに送信される。発行されるリカバリーコードは、第 3.2.12 項に記載されているように、承認されたランダムビット発生器から得た少なくとも 10 進数 6 桁（またはそれと同等のもの）を含まなければならない（SHALL）。発行されたリカバリーコードは、手入力用の数字または英数字（Base64 など）、確認コードの表 示を含む安全な（https など）リンク、またはリカバリーコードを含む機械読み取り可能な光学ラベ ル（QR コードなど）として提示することができる。

    \クリアページ

    発行されたリカバリーコードの有効期限は、最大で次のとおりとする：

    • 米国本土内の住所に郵送する場合は21日間、
    • 米国本土以外の住所に送られた場合は30日間、
    • テキストメッセージまたは音声で送信された場合は10分間。
    • 電子メールアドレスに送られた場合は24時間。

    発行されたリカバリーコードの検証は、 第3.2.2節のスロットリング要件に従うものとする。

    回復アドレスを確立する場合、CSP は回復コードと同じ特性を持つ確認コードを、新しく確立された回復ア ドレスに送信しなければならないものとする。回復アドレスは、サブスクライバがその確認に成功した後にのみ確立されるものとする。CSP は、加入者が少なくとも 2 つの回復アドレスを確立できるようにしなければならない。

    回復連絡先

    リカバリ・コンタクトの使用をサポートする CSP は、サブスクライバが、発行されたリカバリ・コードを受 け取る信頼できるアソシエイトのアドレスを 1 つ以上指定できるようにしなければならない（SHALL）。回復連絡先の要件は、以下の例外を除き、発行済み回復コードの要件と非常に類似している：

    • 回復連絡先に送られた回復コードの有効時間は、回復連絡先が加入者に回復コードを伝達 するための追加時間を提供するために、24 時間延長してもよい（すなわち、テキストメッ セージで送られた場合、有効時間は 24 時間 10 分以内）。
    • 回復コードアドレスの確認は、回復連絡担当者が確認のコードを加入者に送信して入力できるよう に、24 時間延長してもよい（MAY）。

    繰り返される身分証明

    サブスクライバのアカウントが最低 IAL1 で身元証明された場合、CSP は身元証明プロセスの一部を繰 り返すことで、アカウントの回復をサポートすべきである（SHOULD）。CSP は、最初の身元証明のレベルと一致する身元証明の必要なステップを繰り返し、請求者の身元 が以前に確立されたアカウントと一致することを確認するものと する。CSP がユーザからのバイオメトリクス・サンプルまたは最初のプルーフィング中に使用された証拠のコ ピーを保持しており、それが十分な品質と解像度を持つ場合、CSP は[SP800-63A]に記載されているように、ID プルーフ ィング・プロセスの検証部分のみを繰り返してもよい（MAY）。

    IAL/AALによる回復要件

    加入者アカウントに関連するIALおよび最大AALに応じて、異なる回復方法が適用される。

    AAL1でのリカバリー

    ID プルーフには、サブスクライバが自分自身に関する個人情報にアクセスできるように、多要素認証に十 分なAuthenticatorを発行する必要があるため、AAL1 のサブスクライバ・アカウントには ID プルーフがなく、ID プルーフを繰り返すことはできない。CSP は、保存されたリカバリ・コード、発行されたリカバリ・コード、またはリカバリ・コンタクト の正常な使用を要求するものとする（SHALL）。

    AAL2でのリカバリー

    最大 AAL2 で認証できるアカウントを回復するには、CSP はサブスクライバに以下のいずれかを完了するよう要 求しなければならないものとする：

    • セット（保存、発行、リカバリーコンタクト）から異なる方法で取得した2つのリカバリーコード
    • セット（保存、発行、回復連絡先）から1つの回復コードと、加入者アカウントにバインドされた一要素認証機能による認証
    • 度重なる本人確認（ただし、加入者アカウントが本人確認済みであることを条件とする）

    AAL3でのリカバリー

    AAL3 で認証できるアカウントが IAL1 または IAL2 で ID プルーフされている場合、要件は AAL2 での回復の場合と同じである。

    AAL3 で認証可能なアカウントが IAL3 で身元証明されている場合、CSP は、 [SP800-63A]に記載されて いるように、現地で立会いの身元証明セッションにおいて、最初の身元証明セッション中に収集された バイオメトリック特性とのバイオメトリック比較に成功しなければならない（SHALL）。CSP はまた、最初の身元証明プロセスで使用された証拠の提示を要求してもよい（MAY）。

    アカウント回復通知

    すべての場合において、アカウント回復は、 セクション4.6で述べられているように、加入者 に通知を送らなければならない[SHALL]。

    紛失、盗難、破損、危殆化

    危殆化したAuthenticatorには、紛失、盗難、不正複製の対象となったもの、または加入者の管理下 にない起動要因を持つものが含まれる。一般に、紛失したAuthenticatorは、そのAuthenticatorの正当な所持者以外によって盗まれた、または危殆化され たと想定しなければならない。破損または故障したAuthenticatorも、Authenticatorの秘密が抜き取られる可能性から守るために、危殆化したとみなされる。特筆すべき例外は、攻撃者によって入手されたなど、危殆化の兆候なしに忘れられたパスワードで ある。

    CSP は、危殆化の検出後速やかに、サブスクライバのアカウントから危殆化した認証機能を一時停止、 無効化、または破棄するものとする。組織は、このプロセスに時間制限を設けるべきである（SHOULD）。

    認証情報の紛失、盗難、破損、または危殆化の安全な報告を促進するため、CSP は、バックアップま たは代替認証情報を使用して認証する方法をサブスクライバに提供するべきである（SHOULD）。このバックアップ認証機能は、パスワードまたは物理的な認証機能であるものとする。どちらを使用してもよいが、この報告には 1 つの認証要素のみが必要である。あるいは、サブスクライバは、身元確認中に収集された情報を CSP が検証するために、認証済みの保護されたチャネルを確立してもよい[MAY]。CSP は、連絡先住所（電子メール・アドレス、電話番号、郵便住所など）を検証し、危殆化したと報告された認証機 関を一時停止または無効にすることを選択してもよい。

    CSP は、危殆化の可能性が疑われるAuthenticatorの一時的な停止をサポートしてもよい（MAY）。一時停止がサポートされる場合、サブスクライバが有効な（つまり、一時停止されていない） 認証機能を使用して CSP への認証に成功し、一時停止された認証機能の再アクティベーションを要求した 場合、一時停止は取り消されるべきである（SHOULD）。CSP は、一時停止されたAuthenticatorを再アクティブ化できなくなるまでの制限時間を設定してもよい（MAY）。

    有効期限

    CSP は、期限切れのAuthenticatorを発行してもよい（MAY）。Authenticatorの有効期限が切れた場合、そのAuthenticatorは認証に使用できないものとする。期限切れのAuthenticatorを使用して認証が試行された場合、CSP はサブスクライバに対して、認証の失敗が他の 原因ではなく期限切れによるものであることを示すべきである（SHOULD）。

    CSP は、期限切れ後速やかに、個人情報を含むAuthenticatorを回収するか、そのゼロ化（消去）または破棄を提供す べきである（SHOULD）。

    期限切れのAuthenticatorの交換は、 4.1.2節に記載されている追加Authenticatorのバインドプロセスに準拠しなければならない 。

    無効化

    Authenticatorの無効化（失効または終了と呼ばれることもある）とは、Authenticatorとサブスクライバ・アカウントとの間のバインディングを取り除くことである。

    CSP は、サブスクライバのアカウントが存在しなくなったとき（サブスクライバの死亡、不正なサブスクライバの発 見など）、サブスクライバから要求されたとき、認証器が危殆化したとき、またはサブスクライバがその資格要件を 満たさなくなったと CSP が判断したときは、速やかに認証器を無効にするものとする。CSP は、サブスクライバからの無効化要求の真偽についてリスクベースの判定を行うものとし、危殆化した Authenticatorを無効化しなかった場合の結果は、誤って無効化した場合のサービス拒否の可能性よりも通常重大 であることに留意するものとする。

    CSP は、無効化後速やかに、個人情報を含むAuthenticatorを回収するか、そのゼロ化（消去）または破棄を提供す べきである（SHOULD）。

    PIV 認証機関の無効化に関するさらなる要件は、[FIPS201]に記載されている。

    アカウント通知

    Authenticatorのバインディングやアカウントの回復など、特定のサブスクライバアカウントイベントは、 サブスクライバに独自に通知する必要がある。これらの通知は、加入者が加入者アカウントに関連する不正の可能性を検知するのに役立つ。

    通知を必要とするイベントは、サブスクライバーアカウントに保存されている通知アドレ スに通知を送らなければならない[SHALL]。通知アドレスは

    • 住所
    • メールアドレス
    • テキストメッセージまたはボイスメッセージの送信先アドレス（電話番号など

    CSP は、サブスクライバ・アカウントごとに少なくとも 2 つの通知アドレスをサポートし、少なくとも 1 つは身元証明プロセス中に検証されるものとする（SHALL ）。CSP は、AAL2 以上（またはサブスクライバ・アカウントで使用可能な最高 AAL の場合は AAL1）の認証 を持つサブスクライバに、通知アドレスの更新を許可すべきです（SHOULD）。CSP は、サブスクライバが複数の通知アドレスを維持することを推奨すべきで ある。

    通知は、郵便アドレスを除くすべての通知アドレスに送られなければならない[SHALL]。しかしながら、加入者アカウントに他の形式の通知アドレスが保存されていない場合、あるいは通知がAAL3でのアカウント回復のためである場合、通知は郵便アドレスに送られなければならない[SHALL]。

    通知は、受信者が通知に関連する事象を否認する場合に備えて、連絡先情報を含む明 確な指示を提供しなければならない(SHALL)。

    セッション管理

    このセクションは規範的なものである。

    認証イベントが発生すると、サブスクライバが認証イベントを繰り返すことなく、 その後の複数のやりとりにおいてアプリケーションの使用を継続できるように することが望ましい場合が多い。これは特に、認証イベントが必然的にネットワーク全体の複数のコンポー ネントと当事者の調整を伴う、フェデレーションシナリオ(［SP800-63C］に記載)の場合 である。

    この動作を容易にするために、セッションは認証イベントに応答して開始され、 終了するまで継続してもよい[MAY]。セッションは、非活動時のタイムアウトや明示的なログアウトイベントなど、い かなる理由でも終了してもよい[MAY]。再認証イベント(セクション5.2で述べられている)でサブスクライバー が最初の認証プロセスの一部を繰り返すか、完全な認証を実行することで、 認証されたセッションを再確立し、セッションを拡張してもよい[MAY] 。

    セッション管理は、クレデンシャルの継続的な提示よりも望ましい。継続的な提示の使い勝手の悪 さは、回避策（アクティブ化要素のキャッシュなど）の誘因となることが多く、それによって認証の 意図が否定され、認証イベントの新鮮さが不明瞭になるからである。

    セッション・バインディング

    セッションは、サブスクライバが実行しているソフトウェア（すなわち、セッション・サブジェクト） （ブラウザ、アプリケーション、オペレーティング・システムなど）と、サブスクライバがアクセスしている RP または CSP（すなわち、セッション・ホスト）との間で発生する。セッションシークレットは、サブスクライバのソフトウェアとアクセスされるサービスの間で 共有されなければならない[SHALL]。このシークレットはセッションの両端をバインドし、サブスクライバが長期に渡ってサービスを使用し続けることを可能にする。シークレットは、サブスクライバのソフトウェアが直接提示するか、暗号化 メカニズムを使用してシークレットの所有を証明しなければ ならない[SHALL ]。

    認証されたセッションの継続性は、認証時にベリファイアが発行し、オプションでセッ ション中に更新されるセッションシークレットの所有に基づいていなければならない[SHALL]。セッションの性質はアプリケーションに依存する：

    • セッション」クッキーによるウェブ・ブラウザのセッション、または
    • セッション・シークレットを保持するモバイル・アプリケーションのインスタンス。

    セッション秘密鍵は、再起動または再起 動によって終了する特定のセッションに関連付けられるため、永続的であるべきで はない[SHOULD NOT](すなわち、関連するアプリケーションの再起動またはホス トデバイスの再起動にまたがって保持される)。Cookieや同様の「remember my browser」機能を認証の代わりに使用してはならない[SHALL NOT]。ただし、セクション2.2.3のAAL2における再認証に規定されているように、非アクティブ 制限を超えたが時間制限は超えていない場合を除く。

    セッションバインディングに使用される秘密鍵は、認証イベントに直接応答して セッションホストによって生成されなければならない[SHALL]。セッションは、その生成のトリガーとなった認証イベントのAALプロパティを継承するべきである[SHOULD]。セッションは認証イベントよりも低いAALで考慮されてもよい[MAY]が、 認証イベントよりも高いAALで考慮されてはならない[SHALL NOT]。

    セッションバインディングに使用されるシークレットは、以下のすべての要件を満たさなければならない[SHALL]：

    1. シークレットは、認証中または認証直後に確立される。
    2. 秘密は、3.2.12節に記述されているように、承認されたランダムビット生成器からの入力を用いて確立され、少なくとも64ビット長である。
    3. 秘密は、サブスクライバがログアウトすると、セッション・サブジェクトによって消去または無効化される。
    4. 秘密鍵は、セッション・ホストから RP または CSP に、認証された保護されたチャネルを介して転送されるか、有効で相互に認証された保護されたチャネルの確立の一部として確立される鍵から派生する。
    5. セクション2.1.3、セクション2.2.3、およびセクション2.3.3で規定されている、 AALに適した時間を経過すると、シークレットはタイムアウトし、受け付けられなくなる。
    6. シークレットは、ホストとサブスクライバーのエンドポイント間の仲介者は利用できない。

    加えて、セッションバインディングに使用されるシークレットは、サブスクライバーがログアウトしたとき、またはシークレットの有効期限が切れたとみなされたときに、サブスクライバーエンドポイント上で消去されるべきである[SHOULD]。クロスサイトスクリプティング(XSS)攻撃にローカルストレージがさらされる可能性があるため、安全でない場所(HTML5ローカルストレージなど)に置くべきではない(SHOULD NOT)。

    認証後、認証されたセッションは安全でないトランスポートにフォールバックしてはならない(SHALL NOT)。

    POST/PUTコンテンツは、クロスサイトリクエストフォージェリから保護するためにRPが検証しなければならないセッション識別子を含まなければならない(SHALL)。

    セッションを時間とともに管理するためのいくつかの仕組みが存在する。以下の節では、それぞれの例の技術について、異なる例、追加の要求事項、考慮事項を示します。追加の有益な手引きは OWASPセッション管理チートシート [OWASP-session]にあります。

    セッションは、相互作用が完了したときに加入者がセッションを終了(すなわちログオフ)する ための、容易にアクセスできる仕組みを提供するべきである[SHOULD]。セッションのログオフは、特にエンドポイントが他者からアクセスされるかもしれない状況において、加入者にセッションのセキュリティに対する更なる信頼と制御を与える。

    ブラウザクッキー

    ブラウザのクッキーは、加入者がサービスにアクセスするときにセッションが作成され、追跡される主要なメカニズムである。クッキーは認証手段ではないが、セッションの間の短期的な秘密として適している。

    セッションの維持に使用されるクッキー：

    1. 安全な(HTTPS)セッションでのみアクセスできるようにタグ付けされなければならない(SHALL)。
    2. 最低限実用的なホスト名とパスでアクセスできなければならない。
    3. JavaScript(HttpOnly)でアクセスできないようにタグ付けされるべきである。
    4. セッションの有効期間または有効期間の直後に失効するようにタグ付けされるべきです(SHOULD)。この要件はクッキーの蓄積を制限することを意図しているが、セッションのタイムアウトを強制するために依拠してはならない[SHALL NOT]。
    5. Host-"接頭辞を持ち、"Path=/"を設定すべきである。
    6. SameSite=Lax "または "SameSite=Strict "を設定すべきである。
    7. 不透 明な文字列(例えばセッション識別子)だけを含むべきであり(SHOULD NOT)、平文の個人情報を含んではならない(SHALL NOT)。

    アクセス・トークン

    アクセストークン(例えば、OAuth[RFC6749])は、認証イベント後にアプリケーションが サブスクライバーに代わって一連のサービスにアクセスすることを許可するために 使用される。他のシグナルがない場合、RPはOAuthアクセストークンの存在をサブスクライバーの 存在を示すものとして解釈してはならない[SHALL NOT]。OAuth アクセストークンと関連するリフレッシュトークンは、認証セッションが終了し、サブスクライバがアプリケーションから退出した後も有効である可能性がある。

    再認証

    セッションの定期的な再認証は、サブスクライバが認証されたセッションに継続して存在 していること(すなわち、サブスクライバがログアウトせずに立ち去っていないこ と)を確認するために実行されなければならない[SHALL]。

    セッション管理は、2種類のタイムアウトを使用する。オーバーオールタイムアウトは、認証または前回の再認証後、認証されたセッションの継続 時間を特定の期間に制限する。非活動タイムアウトは、サブスクライバが特定の期間活動を行わずにセッションを 終了させる。どちらのタイプのタイムアウトでも、RPはサブスクライバーにセッションが 終了しようとしていることを警告し、セッションが期限切れになる前に、 サブスクライバーにセッションをアクティブにするか、適切な再認証を許可してもよ い[MAY]。いずれかのタイムアウトが切れると、セッションを終了しなければならない[SHALL]。セッションのアクティ ビティは非アクティブ時のタイムアウトをリセットしなければならず[SHALL]、セッション 中に再認証に成功した場合は両方のタイムアウトをリセットしなければならない[SHALL]。

    全体および非活動時のタイムアウト期限は、セッションのAAL、セッションが実施される環境（例えば、サブスクライバが制限区域にいるかどうか）、使用されるエンドポイントのタイプ（例えば、モバイルアプリケーションまたはWebベース）、エンドポイントが管理対象デバイス1であるかどうか、およびアプリケーション自体の性質など、いくつかの要因に依存する。機関は、 [SP800-39]に記述されているようなシステムセキュリティプランにおいて実施される非アクティブ時 間及び全体的な時間制限を設定し、文書化しなければならない（SHALL）。

    各AALの詳細要件は、第2.1.3 節、第2.2.3節、および第2.3.3節に示されている。

    SP800-63C]に記載されているように、RP での認証にフェデレーション・プロトコルと IdP を使用する場合、セッション管理と再認証には特別な考慮事項が適用される。フェデレーション・プロトコルは、アサーションを使用して IdP での認証イベントを RP に伝達し、RP はこのアサーションの検証成功に基づいて認証セッションを開始する。IdP と RP は互いに別々にセッションを管理し、フェデレーション・プロトコルは IdP と RP 間のセッション管理を接続しないので、IdP と RP における加入者のセッションの終了は互いに独立している。同様に、複数の異なる RP における加入者のセッションは、互いに独立して確立および終了される。

    その結果、RP セッションが失効し、RP が再認証を要求する場合、IdP でのセッションが失効しておらず、サブスクライバを明示的に再認証しなくても、IdP でこのセッションから新しいアサーションが生成される可能性がある。IdP は、認証イベントの時間と詳細を RP に伝えることができるが、再認証要件が満たされているか どうかの判断は RP に任されている。SP800-63C]のセクション 4.7では、フェデレーションコンテキスト内のセッション管理 の詳細と要件が追加されている。

    セッション監視

    セッション・モニタリング（継続的認証と呼ばれることもある）とは、セッション中の 不正の可能性を検出するために、セッション特性を継続的に評価することである。

    セッション監視は、リスク削減策として、RP が CSP/ベリファイアと連携して実行してもよい（MAY）。セッション中に不正の可能性が検出された場合、RP は CSP/検証者と連携して、再認証、セッ ションの終了、または適切なサポート要員への通知などの措置を取るべきである（SHOULD）。評価してもよいセッション特性には、以下のものが含まれる：

    • 使用パターン、速度、タイミング
    • 行動バイオメトリック特性（例：タイピング・ケイデンス）
    • デバイスとブラウザの特性
    • ジオロケーション
    • IPアドレスの特徴（悪用で知られるブロックにIPアドレスがあるかどうかなど）

    これらの特性のほとんどはプライバシーに関係する。予想される加入者特性の収集、保存、およびセッション特性の処理は、第 7 節に記述されるプライバシーリスク評価に含めなければならない[SHALL]。

    1. 管理対象デバイスには、パーソナルコンピュータ、ノートパソコン、モバイルデバイス、仮想マシン、またはインフラストラクチャコンポーネントが含まれ、情報技術スタッフがそれらを発見、維持、制御できるようにする管理エージェントが装備されている。

    認証者の脅威

    Authenticatorを制御できる攻撃者は、多くの場合、Authenticatorの所有者になりすますことができる。Authenticatorに対する脅威は、Authenticatorを構成する認証要素のタイプに対する攻撃に基づいて分類すること ができる：

    • 「あなたが知っている何か」が攻撃者に開示される可能性がある。例えば、攻撃者はパスワードを推測することができる。Authenticatorが共有秘密である場合、攻撃者は CSP またはベリファイアにアクセスして秘密値を取得したり、その値のハッシュに対するディクショナリ攻撃を実行したりする可能性がある。攻撃者は、PIN またはパスコードの入力を観察したり、PIN またはパスコードの記録や日誌を発見したり、悪意のあるソフトウ ェア（キーボード・ロガーなど）をインストールして秘密を取得したりすることができる。さらに、攻撃者は、ベリファイアが保持するパスワード・データベースに対するオフライン攻撃を通じて、 秘密を特定することができる。

    • 「持っているもの」は、紛失、破損、所有者からの盗難、または攻撃者によるクローンかもしれない。たとえば、所有者のコン ピュータにアクセスした攻撃者は、ソフトウェア認証機能をコピーすることができる。ハードウェア認証機能は、盗まれたり、改ざんされたり、複製されたりする可能性がある。帯域外の秘密が攻撃者によって傍受され、自身のセッションを認証するために使用される可能性がある。加入者は、意図的な共謀なしに秘密へのアクセスを提供するように、社会的に 操作される可能性がある。

    • 「あなたがいる何か」は複製されるかもしれない。例えば、攻撃者は加入者のフィンガープリントのコピーを入手し、レプリカを構築するかもしれない。

    加入者は攻撃者と結託することがあり、認証の観点からは、このような攻撃を防 ぐためには事実上何もできない。この注意点を念頭に置いて、デジタル認証に使用されるAuthenticatorに対する脅威を、いくつかの例とともに表 2 に示す。

    エンドポイント上の悪意のあるコードが、多要素ソフトウェア暗号化認証機能を侵害する。 悪意のあるコードは、エンドポイントから認証鍵をプロキシしたり、認証鍵をエクスポートしたりする。

    認証機能の回復

    多くの認証メカニズムの弱点は、サブスクライバが1つまたは複数のオーセンティケー タの制御を失い、それらを交換する必要があるときに従うプロセスである。多くの場合、加入者を認証するためのオプションは限られており、経済的な懸念（コ ールセンターの維持費など）が、安価で多くの場合安全性の低いバックアップ認証方法を使用す る動機となっている。認証者のリカバリが人的支援である限り、ソーシャル・エンジニアリング攻撃も危険である。

    認証要素の完全性を維持するためには、ある認証要素を利用して別の認証要素のAuthenticator を取得できないことが不可欠である。例えば、パスワードは、ルックアップ秘密の新しいリストを取得するために使用可能であってはならない。

    セッション攻撃

    認証イベント後のセッションに対するハイジャック攻撃も、同様のセキュリティ上の 影響を与える可能性がある。第5節のセッション管理のガイドラインは、攻撃(例えば XSS)に対してセッションの完全性を維持するために不可欠です。また、表示されるすべての情報[OWASP-XSS-prevention]をサニタイズして、実行可能なコンテン ツを含まないようにすることも重要です。これらのガイドラインは、セッション秘密の流出に対する特別な防御を提供するために、セッション秘密をモ バイルコードからアクセスできないようにすることを推奨します。

    認証後のもう1つの脅威は、クロスサイト・リクエスト・フォージェリ（CSRF）であり、ユーザが複数のセッションを同時にアクティブにする傾向を利用する。有効なURLやリクエストが意図せず、あるいは悪意を持ってアクティブにされるのを防ぐために、ウェブ・リクエストにセッション識別子を埋め込み、検証することが不可欠です。

    プライバシー・リスク評価

    第 2 節の認証要件および第 5.3 節のオプションのセッション監視ガイドラインでは、CSP は記録保持のため にプライバシー・リスク評価を実施する必要がある。このようなプライバシー・リスク評価には、以下が含まれる：

    1. 情報への侵入や不正アクセスなど、記録の保持が加入者に問題を引き起こす可能性。
    2. そのような問題が起きた場合の影響

    CSP は、リスクの受容、軽減、および共有を含め、特定されたプライバシー・リスクへの対応を 合理的に正当化できるものとする。加入者の同意は、リスクを共有する一形態である。したがって、利用者の同意は、共有されるリスクを評価し、受け入れる能力があると合理的に期待できる場合にのみ使用することが適切である。

    プライバシー・コントロール

    セクション 2.4.3では、CSP に対して、適切に調整されたプライバシー管理を採用するよう要 求している。[SP800-53]は、CSP が認証メカニズムをデプロイする際に考慮すべき一連のプライバシー管理を提 供する。これには、通知、救済、および信頼できるデプロイを成功させるためのその他の重要な考慮事項 が含まれる。

    使用制限

    第 2.4.3 節では、CSP は、ID 証明、認証、認可、または属性アサーション以外の目的、関連する不正行為の 軽減、または法律や法的手続に従う目的以外の属性の処理から生じ得るプライバシー・リスクに見合う、予 測可能性（PII および情報システムによるその処理について、個人、所有者、および運用者が信頼できる 前提を可能にすること）および管理可能性（すなわち、変更、削除、および選択的な開示を含む、PII のきめ細かな管 理機能を提供すること）の目標を維持することを要求する[NISTIR8062]。

    CSP は、加入者への非 ID サービスの提供を含め、属性を処理する様々な事業目的を持つことができる。ただし、収集時に指定された目的以外で属性を処理すると、プライバシ ー・リスクが生じる可能性がある。CSP は、追加の処理から生じるプライバシー・リスクに見合った適切な手段を特定できる。例えば、適用される法律、規制、またはポリシーがない場合、加入者が要求する非アイデンティティ・ サービスを提供するために属性を処理する場合、同意を得る必要はないかもしれない。しかし、通知は、加入者が処理について信頼できる前提（すなわち、予測可能性）を維持す るのに役立つかもしれない。属性の他の処理は、同意を得ること、あるいは特定の属性の使用または開示について加入者により多くのコ ントロールを認めること（すなわち、管理可能性）を求める、異なるプライバシーリスクを伴うかもしれ ない。加入者の同意は、意味のあるものでなければならない。そのため、第 2.4.3 節に記載されているように、CSP が同意手段を使用する場合、加入者が追加の使用 を承諾することは、認証サービスを提供する条件であってはならない。

    提案された処理が、許可された処理の範囲外であるか、または適切なプライバシーリスク軽減措置の範囲外であるかについて疑問がある場合は、当該機関のSAOPに相談すること。

    省庁固有のプライバシー・コンプライアンス

    セクション 2.4.3では、連邦 CSP の具体的なコンプライアンス義務について説明する。デジタル認証システム開発の初期段階で省庁の SAOP を関与させ、プライバシー・リスクを評 価・軽減し、Authenticatorを発行または維持するために PII を収集することが、1974 年プライバシー法 （PrivacyAct ）または2002 年電子政府法（E-Government Act of 2002）（E-Gov）のPIA 実施要件に該当するかどうかなど、コンプライアンス要件について省庁に助言するこ とが重要である。たとえば、バイオメトリクスの集中管理に関しては、PII および認証に必要なその他の属性の収集および維持のために、プライバシ ー法要件がトリガされ、新規または既存のプライバシー法記録システム通知によるカバーが必要になる 可能性が高い。SAOP は、PIA が必要かどうかを判断する際にも、同様に機関を支援することができる。

    これらの考慮事項は、認証だけのためにプライバシー法 SORN または PIA を作成する要件とし て読まれるべきものではない。多くの場合、PIA および SORN は、デジタル ID プロセス全体を包含することができ、ま たは、機関が確立しようとしているオンライン・サービスまたは便益を論じるより大規 模なプログラム的 PIA の一部としてデジタル認証プロセスを含めることができる。

    デジタル認証には多くの構成要素があるため、SAOP は、各構成要素を認識し理解する必要がある。たとえば、連携 CSP または RP サービスを提供または使用する機関には、他のプライバシー成果物 が適用される場合がある（データ使用合意、コンピュータ・マッチング合意など）。SAOP は、どのような追加要件が適用されるかを機関が判断する際に役立つ。さらに、デジタル認証の個々のコンポーネントを十分に理解することで、SAOP は、コンプライ アンス・プロセスまたは他の手段を通じてプライバシー・リスクを評価し、軽減することができ るようになる。

    認証機能に関する一般的なユーザビリティの考慮事項

    認証システムを選択し、実装する場合、ユーザ、ユーザの目標、およびユーザの使用コンテク ストを考慮しながら、選択したAuthenticatorのライフタイム全体（例えば、典型的な使用と断続的な イベント）におけるユーザビリティを考慮する。

    通常、単一の認証タイプで全ユーザに対応できるわけではない。したがって、CSP は、可能な限り、また AAL 要件に基づいて、代替のAuthenticatorタイプをサポートし、ユー ザがニーズに最も適したタイプを選択できるようにする必要がある。タスクの即時性、認識されるコスト・ベネフィット・トレードオフ、および特定のAuthenticatorに対する不慣 れが、選択に影響を与えることが多い。ユーザは、その時点で最も負担やコストの少ないオプションを選択する傾向がある。例えば、あるタスクが情報システムへの即時アクセスを必要とする場合、ユーザは、より多くのステップを必要とするAuthenticatorを選択するよりも、新しい加入者アカウントとパスワードを作成することを好むかもしれない。あるいは、ユーザは、ID プロバイダの加入者アカウントをすでに持っている場合、適切な IAL、AAL、および FAL で承認される連携 ID オプションを選択することができる。ユーザは、他の認証機能より優れている認証機能を理解し、その理解と経験に基づく信頼レベ ルが異なる場合がある。

    肯定的なユーザ認証体験は、望ましいビジネス成果を達成するために不可欠である。したがって、組織は、ユーザの視点から認証者を考慮するよう努めなければならない。認証のユーザビリティの包括的な目標は、ユーザの負担と認証の摩擦（例えば、ユーザが 認証しなければならない回数、手順、追跡しなければならない情報量）を最小化することである。シングルサインオンは、このような最小化戦略の一例である。

    ほとんどの認証方式に適用できるユーザビリティに関する考慮事項を以下に示す。後続のセクションでは、特定の認証方式に特有のユーザビリティに関する考慮事項を記述する。

    ほとんどの認証機関に適用可能なユーザビリティの考慮事項には、以下が含まれる：

    • 認証機の使用および保守に関する情報（例えば、認証機が紛失または盗難された場合の対応）、 および特に初回使用または初期化について異なる要件がある場合には、使用説明書を提供する。

    • ユーザは、認証器をすぐに利用できるようにしておく必要があるため、認証器の可用性。元の認証器の紛失、損傷、その他の悪影響、および該当する場合はバッテリ電源の喪失の可能 性から保護するために、代替認証オプションの必要性を検討する。

    • 可能な限り、AAL 要件に基づく代替認証オプション。これにより、ユーザは、コンテキスト、目標、およびタスク（タスクの頻度や即時性 など）に基づいて、認証方式を選択することができる。代替認証オプションは、特定のAuthenticatorで発生する可能性のある可用性の問題に対処するのにも 役立つ。

    • ユーザー向けテキストの特徴
      • ユーザー向けのテキスト（指示、プロンプト、通知、エラーメッセージなど）は、想定される読者に向けて平易な言葉で書く。技術的な専門用語は避け、想定される読者の識字レベルに合わせて書く。
      • フォントのスタイル、サイズ、色、周囲の背景とのコントラストなど、ユーザー向けおよびユーザー入力テキストの読みやすさを考慮する。読みにくいテキストは、ユーザーの入力ミスを助長します。読みやすさを向上させるために、以下の使用を検討してください：
        • 高コントラスト（白地に黒など）
        • 電子ディスプレイにはサンセリフ書体、印刷物にはセリフ書体を使用する。
        • 混同しやすい文字を明確に区別できるフォント（例：大文字の「O」と数字のゼロ「0」）。
        • デバイスに表示するためにテキストが収まる限り、最小フォントサイズは12ポイント。
      • ブラウザのセキュリティ・インジケータと混同される可能性のあるアイコン（南京錠や盾など）の使用は避けること。
    • 認証入力時のユーザー体験：
      • マスクされたテキスト入力はエラーになりやすいので、入力中にテキストを表示するオプションを提供する。ある文字がユーザにとって十分な時間表示されると、その文字を非表示にすることができる。モバイル・デバイス（タブレットやスマートフォンなど）では、従来のデスクトップ・コン ピュータよりもパスワードの入力に時間がかかるため、マスキング遅延時間を決定す る際にはデバイスを考慮する。マスキングの遅延時間がユーザーのニーズと一致していることを確認する。
      • テキスト入力に許容される時間が適切であることを確認する（すなわち、入力画面が早期にタイムアウトしない）。テキスト入力の許容時間がユーザーのニーズと一致していることを確認する。
      • ユーザーの混乱とフラストレーションを軽減するために、入力エラーに対して明確で意味のある、実用的なフィードバックを提供する。ユーザがテキスト入力を誤ったことに気付かない場合、ユーザビリティに重大な影響が生じます。
      • ユーザによるAuthenticator出力の入力が必要なAuthenticatorについては、少なくとも 10 回の入力試行を許可す る。入力テキストが長く複雑であればあるほど、ユーザによる入力エラーの可能性は高くなる。
      • 残りの試行回数について、明確で意味のあるフィードバックを提供する。レート制限（スロットリング）については、次の試行までの待ち時間をユーザに通知する。
    • モバイル・デバイスの限られたタッチ・エリアやディスプレイ・エリアなど、フォーム・ファクターによる制約の影響を最小限に抑える：
      • 小型デバイスでの入力は、画面上のターゲットのサイズに対する入力メカニズム（指など）のサイズの関係で、フルサイズのキーボードで入力するよりもエラーが発生しやすく、時間がかかるためです。
      • 小型ディスプレイのための優れたユーザーインターフェースと情報デザインに従うこと。

    認証タイプ間の断続的なイベント（再認証、加入者アカウントのロックアウト、期限切れ、失効、損 傷、紛失、盗難、および機能しないソフトウェアなど）に対するユーザビリティの考慮事項には、以 下のようなものがある：

    • 非アクティブ・タイムアウトが発生する直前（たとえば、2分前）に、ユーザーに何らかのアクティビティを実行するよう促す。

    • ユーザーのアクティビティに関係なく、固定の再認証タイムアウトが発生する前に、ユーザーに作業の保存を促す。

    • 技術支援を受ける方法と場所を明確に伝える（例えば、オンラインセルフサービス機能、チャットセッション、ヘルプデスクサポートの電話番号へのリンクをユーザーに提供する）。理想的には、外部からの介入なしに、ユーザーが断続的な事象から自力で回復できるよう、十分な情報を提供する。

    • 加入者がセッションを終了（すなわちログオフ）するためのアクセス可能な手段を提供する。

    認証タイプ別のユーザビリティに関する考察

    以下のセクションでは、特定のAuthenticatorタイプに特有のその他のユーザビリティに関する考慮事 項について説明する。

    パスワード

    典型的な使用例

    ユーザーは、パスワード（パスフレーズまたはPINと呼ばれることもある）を手動で入力することが多い。あるいは、安全なパスワードの選択を支援し、認証されたサービスごとに異なるパスワードを管理するために、パスワードマネージャを使用することもある。別個のパスワードを使用することは、攻撃者が、あるサイトから漏洩したパスワードを、そのユーザもアカウントを持っている可能性のある他のサイトで使用する「パスワード・スタッフィング」攻撃を回避するために重要である。機関は、安全な実装の期待に応えるものであることを確認するために、推奨または義務付けを行う前に、パスワード管理者を慎重に評価すべきである。

    パスワード・マネージャーを使わない典型的な使用方法におけるユーザビリティの考慮点は、以下の通りである：

    • パスワードの記憶性
      • ユーザーが記憶すべき項目が多くなると、想起に失敗する可能性が高くなる。パスワードの数が少なければ、ユーザーは特定のRPに必要な特定のパスワードをより簡単に思い出すことができる。
      • 使用頻度の低いパスワードでは、記憶負担が大きくなる。
    • パスワード入力時のユーザー体験
      • パスフレーズを含むパスワード入力欄のコピー＆ペースト機能をサポート。

    断続的なイベント

    断続的なイベントに対するユーザビリティの考慮事項には、以下のようなものがある：

    • ユーザーがパスワードを作成・変更する場合
      • パスワードの作成および変更方法に関する情報を明確に伝える。
      • 第3.1.1項に明記されているように、パスワードの要件を明確に伝えること。
      • パスフレーズの使用をサポートするため、少なくとも64文字の長さを許可する。パスワードを好きなだけ長くし、暗記を助けるために好きな文字（スペースを含む）を使用するよう、ユーザーに奨励する。ユーザーインターフェースが十分な長さのパスワードをサポートするようにする。
      • パスワードに他の構成規則（異なる文字種の混合など）を課さない。
      • ユーザからの要求がない限り、または認証機関の危殆化の証拠がない限り、パスワードを任意に （例えば定期的に）変更することを要求しないこと（追加情報については第 3.1.1 節を参照）。
    • 選択したパスワードが拒否された場合（例：使用できないパスワードの「ブロックリスト」に表示されている場合、または以前に使用されたことがある場合）、明確で意味のある、実行可能なフィードバックを提供する。

    ルックアップの秘密

    典型的な使用例

    加入者は、ベリファイアのプロンプトに応答するために必要な適切な秘密を調べるために、 印刷されたまたは電子的なオーセンティケータを使用する。たとえば、利用者は、カードに表形式で印刷された数字または文字列の特定のサブセットを提供するよう求められることがある。

    一般的な使用におけるユーザビリティの考慮点は以下の通り：

    • ルックアップ秘密の入力時のユーザーエクスペリエンス。
      • プロンプトの複雑さとサイズを考慮する。ユーザに検索を促す秘密のサブセットが大きいほど、ユーザビリティに大きな影響がある。入力時の認知的作業負荷と物理的難易度の両方を考慮する必要がある。

    帯域外

    典型的な使用例

    帯域外認証では、ユーザがプライマリおよびセカンダリ通信チャネルにアクセスできるこ とが必要である。

    一般的な使用におけるユーザビリティの考慮点は以下の通り：

    • ロック可能なデバイスでシークレットを受け取ったことをユーザに通知する。帯域外デバイスがロックされている場合、秘密へのアクセスにはデバイスへの認証が必要である。

    • 実装によっては、ユーザーがモバイルデバイスでテキストを入力しなければならない場合に特に問題となる、フォームファクタの制約を考慮してください。より大きなタッチエリアを提供することで、モバイルデバイスで秘密を入力する際のユーザビリティが向上します。

    • プライマリ・チャネルとセカンダリ・チャネルが同じデバイス上にある場合、特に有 用である。例えば、ユーザがスマートフォンを使って認証シークレットを手動で転送するのは、帯域外アプリケーショ ンとプライマリ・チャネルを何度も往復しなければならないため、困難である。

    • 帯域外デバイスへのメッセージや通知には、アクセス先のサービス名など、ユーザーのコンテキスト情報を含めるべきである。

    • 帯域外メッセージは、加入者が疑わしい可能性のある認証要求を識別できるよう、一貫した方法およびスタイルで配信されるべきである。

    シングルファクターOTP

    典型的な使用例

    ユーザは、一要素 OTP 認証機能によって生成された OTP にアクセスする。通常、OTP 認証出力は認証機器に表示され、ユーザは認証セッション中に入力する。

    一般的な使用におけるユーザビリティの考慮点は以下の通り：

    • 認証機能出力は、少なくとも 1 分間は変更可能であるが、第 3.1.4.1 節に規定されているように、理想的には 2 分間は変更可能である。ユーザは、1 要素 OTP 認証機能と入力画面を行ったり来たりするなど、認証機能出力を入力す るのに十分な時間が必要である。

    • 実装によっては、実装者がさらにユーザビリティを考慮しなければならないことがある：

      • ユーザが認証出力を手動で入力する必要がないように、1 要素 OTP 認証器は、その出力を電子イン ターフェース（例えば、USB ポート）を介して供給することが望ましい。しかし、操作に物理的な入力（例えば、ボタンを押す）が必要な場合、USB ポートの位置がユーザビリティに難をもたらす可能性がある。例えば、一部のコンピュータの USB ポートは、コンピュータの背面に配置されており、ユー ザーが到達するのが困難な場合がある。
      • コンピュータに直接接続できるインターフェース（USBポートなど）の数が限られていると、使い勝手が悪くなる可能性がある。たとえば、ラップトップ・コンピュータの USB ポートの数は非常に限られていることが多い。このため、ユーザは、1 要素 OTP 認証を使用するために、他の USB 周辺機器を抜かざるを得ない場合がある。

    多要素OTP

    典型的な使用例

    ユーザは、多要素 OTP 認証機能によって生成された OTP に、第 2 の認証要素を通じてアクセ スする。OTP は通常、デバイスに表示され、ユーザは認証中のセッション中に手動で入力する。第 2 の認証要素は、パスワードを入力するための一体型入力パッド、一体型バイオメトリッ ク（指紋など）リーダ、または直接コンピュータ・インタフェース（USB ポートなど）により実現される。追加要素に対するユーザビリティの考慮も適用される（パスワードについては第 8.2.1 節を、多要素認証に使用されるバイオメトリクスについては第 8.4 節を参照）。

    \クリアページ

    一般的な使用におけるユーザビリティの考慮点は以下の通り：

    • Authenticator出力の手動入力時のユーザ体験
      • 時間ベースの OTP の場合は、OTP が表示される時間に加えて猶予時間を設ける。ユーザは、多要素 OTP 認証機と入力画面の間を行ったり来たりするなど、認証機出力を 入力するのに十分な時間が必要である。
      • ユーザが一体型入力パッドで多要素ワンタイムパスワード認証機能のロックを解除するか、 モバイルデバイスで認証機能出力を入力する必要がある場合は、フォームファクタの制約を考慮す る。小型デバイスでの入力は、従来のキーボードでの入力に比べ、エラーが発生しやすく、時間が かかります。より大きなタッチエリアを提供することで、多要素 OTP 認証機 能のロックを解除したり、モバイルデバイスで認証機能出力を入力したりする際の操作性が向 上する。
      • 直接コンピュータ・インタフェース（USB ポートなど）を利用できる場所が限られている場合 は、ユーザビリティに問題が生じる可能性がある。例えば、ラップトップ・コンピュータでは USB ポートの数が限られていることが多く、多要素 OTP 認証機能を使用するために他の USB 周辺機器を取り外さなければならない場合がある。

    単一要素暗号認証

    典型的な使用例

    ユーザーは、暗号鍵の所有と管理を証明することで認証を行う。

    一般的な使用におけるユーザビリティの考慮点は以下の通り：

    • ユーザが、どの暗号鍵をどの認証タスクに使用するかを認識し、思い出すことができるよう、 ユーザにとって意味のある適切な記述名を暗号鍵に付与する。これにより、ユーザは、複数の類似した曖昧な名前の暗号鍵を扱う必要がなくなる。画面サイズが小さいために暗号キーの名前が短くなっている場合、小型のモバイル・ デバイスで複数の暗号キーから選択することは、特に問題となる可能性がある。

    • 1 要素暗号化認証機能を操作するために物理的な入力（例えば、ボタンを押す）を必要とするこ とは、ユーザビリティを困難にする可能性がある。たとえば、一部の USB ポートはコンピュータの背面にあり、ユーザがポートに到達するのが困難である。

    • 接続型認証機の場合、直接コンピュータ・インタフェース（例えば USB ポート）を利用できる 場合が限られているため、使い勝手が悪くなる可能性がある。例えば、ラップトップ・コンピュータは、USB ポートの数が限られていることが多く、ユーザは認証器を使用するために他の USB 周辺機器を取り外さなければならない場合がある。

    多要素暗号認証

    典型的な使用例

    認証のために、ユーザは暗号鍵の所有と管理、および起動要素の管理を証明する。追加要素に対するユーザビリティの考慮も適用される（パスワードについては第 8.2.1 節を、 活性化要素として使用されるバイオメトリクスについては第 8.4 節を参照）。

    一般的な使用におけるユーザビリティの考慮点は以下の通り：

    • ユーザが、どの暗号鍵をどの認証タスクに使用するかを認識し、思い出すことができるよう、 ユーザにとって意味のある適切な記述名を暗号鍵に付与する。これにより、ユーザは、複数の類似した曖昧な名前の暗号鍵を扱う必要がなくなる。画面サイズが小さいために暗号キーの名前が短くなっている場合、小型のモバイル・ デバイスで複数の暗号キーから選択することは、特に問題となる可能性がある。

    • 認証後、外部多要素暗号化認証装置を接続しておくことをユーザに要求しない。ユーザは、認証を終了するときに、認証器を切断することを忘れる可能性がある（例 えば、スマートカードリーダにスマートカードを忘れて、コンピュータから離れる）。
      • ユーザは、認証器が接続された状態を維持する必要があるかどうかを知る必要がある。
    • 接続型認証機の場合、直接コンピュータ・インタフェース（例えば USB ポート）を利用できる 場合が限られているため、使い勝手が悪くなる可能性がある。例えば、ラップトップ・コンピュータは、USB ポートの数が限られていることが多く、ユーザは認証器を使用するために他の USB 周辺機器を取り外さなければならない場合がある。

    ユーザビリティに関する考察のまとめ

    図 4 は、各認証タイプについて、典型的な使用法と断続的なイベントに対するユーザビリティの 考慮事項をまとめたものである。行に示されているように、典型的な使用法に関するユーザビリティの考慮事項の多くは、ほとんどの認証機 器タイプに当てはまる。この表は、認証タイプ間で共通するユーザビリティ特性と異なるユーザビリティ特性を強調し ている。各列により、読者は、各認証 器のユーザビリティ属性を簡単に特定することができます。ユーザの目標や使用状況によっては、特定の属性が他の属性よりも重視される場合があります。可能な限り、代替の認証タイプ を提供し、ユーザがその中から選択できるようにする。

    多要素認証機能（多要素ワンタイムパスワードや多要素暗号など）も、その活性化要素の使用 可能性に関する考慮事項を継承する。バイオメトリクスは、多要素認証ソリューショ ンのアクティブ化要因としてのみ許可されるため、バイオメトリクスのユーザビリティ考慮事項は、図 4には含まれず、第 8.4 節で議論される。

    バイオメトリクスのユーザビリティ

    このセクションでは、バイオメトリクスの一般的なユーザビリティの概要について説明します。バイオメトリクスのユーザビリティに関するより詳細な考察は、「ユーザビリティとバイオメトリクス」、「バイオメトリクス・システムを確実に成功させるために 」[UsabilityBiometrics]に記載されています。

    ユーザが機器に慣れ、練習することは、すべてのモダリティのパフォーマンスを向上させる。デバイスのアフォーダンス（ユーザーがアクションを実行できるようにするデバイスの特性）、 フィードバック、および明確な指示は、ユーザーがバイオメトリクス・デバイスで成功するために 不可欠である。例えば、生存検出のために必要な動作について明確な指示を提供する。理想的には、ユーザは第 2 認証要素に最も使いやすいモダリティを選択できる。さまざまなユーザ集団は、他のバイオメトリクス・モダリティよりも、あるバイオメトリクス・モダリティ の方が快適で、慣れ親しんでおり、受け入れられるかもしれない。さらに、バイオメトリクスに関するユーザの経験も活性化要因である。許容される残りの試行回数について、明確で意味のあるフィードバックを提供する。例えば、レート制限（すなわち、スロットリング）の場合、次の試行まで待たなければならない期間をユーザに知らせる。

    典型的な使用例

    認証に最もよく使われる3つのバイオメトリック・モダリティは、指紋、顔、虹彩である。

    • 指紋の使いやすさへの配慮：
      • ユーザーは、最初の登録にどの指を使ったかを覚えておく必要がある。
      • 指の水分量は、センサが正常にキャプチャする能力に影響する。
      • 指紋キャプチャー品質に影響するその他の要因には、年齢、性別、職業（例えば、化学薬品を扱うユーザーや、手を使って広範囲に作業するユーザーは、摩擦隆起が劣化している可能性があります）などがあります。
    • 顔のユーザビリティへの配慮：
      • 利用者は、登録時にアーチファクト（メガネなど）をかけていたかどうかを覚えておく必要があり、顔認識精度に影響する。
      • 環境照明条件の違いは、顔認識精度に影響を与える可能性がある。
      • 顔の表情は顔認識精度に影響する（例：笑顔と無表情）。
      • 顔のポーズが顔認識精度に影響する（例：下を向く、カメラから目をそらす）。
    • 虹彩のユーザビリティに関する考察：
      • カラーコンタクトの着用は、虹彩認識精度に影響を与える可能性があります。
      • 目の手術を受けた方は、手術後に再登録が必要な場合があります。
      • 環境の照明条件の違いは、特に特定の虹彩の色について、虹彩認識精度に影響を与える可能性があります。

    断続的なイベント

    バイオメトリクスは、多要素認証の第 2 要素としてのみ許可されているため、第 1 要素に断続的な事象が発生した場合のユーザビリティへの配慮は依然として適用される。バイオメトリクスを使用した認識精度に影響を及ぼす可能性のある断続的な事象には、以下が含まれる：

    • 劣化した指紋または指の傷
    • 汚れた手、乾いた手、濡れた手、手袋の着用、マスクの着用
    • 顔や体重の自然な経年変化
    • 目の手術

    すべてのバイオメトリック・モダリティにおいて、断続的なイベントに対するユーザビリティの考慮点は以下の通りである：

    • 代替認証方法は、容易に利用可能であり、明確に伝えられなければならない。利用者は、バイオメトリクス認証を試みることを決して要求されるべきではなく、代 替の第 2 要素としてパスワードを使用することが許可されるべきである。
    • 技術支援のための規定を設けるべきである：
      • 技術支援を受ける方法と場所に関する情報を明確に伝える。例えば、オンライン・セルフサービス機能へのリンクやヘルプデスク・サポートの電話番号をユーザーに提供する。理想的には、断続的な事象が発生した場合に、外部からの介入を受けずにユーザーが自力で復旧できるよう、十分な情報を提供すること。
      • バイオメトリックセンサの感度に影響を与える可能性のある要因（例えば、センサの清潔さ） をユーザに知らせる。

    エクイティに関する考察

    このセクションは参考になる。

    正確で公平な認証サービスは、デジタル ID システムの不可欠な要素である。認証の正確さの側面は、主に本文書の他の箇所に記載されているセキュリティ要件の対象で あるが、行政命令 13985「Advancing Racial Equity and Support for Underserved Communities Through the Federal Government 」[EO13985]に規定されているように、政府サービスへの公平なアクセ スを提供するには、すべてのサブスクライバが確実に認証できることが必要である。衡平性リスクを評価する際、CSP は、その認証サービスのユーザ人口全体を考慮するものとする。さらに、CSP は、そのユーザ集団の中で、そのサービスを利用する際に不公平なアクセス、処遇、または 結果の原因となる可能性がある共通の特性を持つユーザ・グループを特定する。セクション 8では、認証サービスを利用するすべての人の全体的なユーザビリティおよび公平性を 確保するための考慮事項について説明する。

    衡平性の主要な側面は、CSP がその加入者集団のニーズを予測し、その集団に適した認証機能オプショ ンを提供する必要があることである。Authenticatorの適合性に関する問題の例をいくつか挙げる：

    • SMS ベースの帯域外認証は、携帯電話サービスのない地方の加入者には使用できない場合がある。
    • OTP 認証機能は、視力に問題のある加入者には読みにくい場合がある。
    • 音声電話で送信される帯域外認証秘密は、聴覚に障害のある加入者には理解しにくい場合がある。
    • 顔照合アルゴリズムは、すべての民族や眼鏡をかけている人の顔の特徴と同じように一致するとは限りません。
    • 加入者の中には、指が欠損していたり、指紋が劣化していたり（例えば、化学薬品を使用したり、手を酷使したりしたため）、指紋採取を妨げる器用さに問題があったりする場合がある。
    • ハードウェアベースのオーセンティケータのコストは、加入者によっては手に負えないかもしれない。
    • パスワードの正確な手入力は、運動能力や器用さに関連した身体障害を持つ加入者にとって困難な場合がある。
    • 知的障害、発達障害、学習障害、または神経認知障害のある利用者にとって、ある種の認証機 能は難しいかもしれない。
    • 低所得の加入者は、いくつかの認証方式で必要とされる最新のデバイスを持っている可能性が低い。
    • 低所得の加入者は、スマートフォンの使用が制限されているため、USB接続の認証機を使用できない可能性がある。
    • 技術的なスキルの低い加入者は、あるデバイスから別のデバイスへOTPコードを入力する際に助けが必要になるかもしれない。
    • 高齢の加入者は、いくつかの認証機の小型フォーム・ファクタに助けを必要とするかもしれない。
    • 依存症、性的搾取、その他のトラウマを経験した加入者は、パスワードやアクティベーションの秘密を思い出すのに苦労するかもしれない。

    CSP は、この分野における一般的かつ予想される問題を軽減することが求められるが、アプリ ケーションによって異なる潜在的な公平性の問題をすべて予測することは不可能である。したがって、CSP は、加入者が不公平な認証要件を報告するメカニズムを提供し、潜在的な代替認証戦略について助言する必要がある。

    本ガイドラインは、アカウント回復の必要性を最小化するために、追加認証器のバインドを推奨す る（第 4.2 項参照）。しかし、加入者は、バックアップとして 2 つ目のハードウェアベースの認証器を購入するための支援が必要かもしれない。この不公平は、ルックアップシークレット(Sec. 3.1.2 参照)のような安価な認証器を、認証器の故障または紛失時に使用できるようにすることで対処できる。

    CSP は、現在サポートしている認証機能を使用しても解決できない認証課題を経験する加入者 に対応する必要がある。これには、新しいAuthenticatorタイプをサポートするか、加入者のニーズを満たす信頼されたサービ スを介した連携認証を許可することが含まれる。

    パスワードの強度

    この付録は参考になる。

    本付録では、議論を容易にするため、「パスワード」という語を使用する。使用されている場合、パスフレーズ、PIN、パスワードを含むと解釈すべきである。

    はじめに

    パスワードは、ユーザビリティとセキュリティの両方の観点からその使用について 懸念があるにもかかわらず、広く使われている認証形式である[Persistence]。人間は複雑で任意の秘密を記憶する能力に限界があるため、容易に推測できるパスワードを選択することが多い。その結果生じるセキュリティ上の懸念に対処するため、オンラインサービスではパスワードの複雑さを増すルールを導入してきた。最も注目されるのは構成ルールで、ユーザーは複数の文字（例えば、少なくとも1つの数字、大文字、記号）を組み合わせて構成されたパスワードを選択する必要がある。しかし、漏洩したパスワードデータベースを分析した結果、このようなルールの利点は当初考えられていたほど大きくなく[Policies]、ユーザビリティと記憶性に深刻な影響を与えることが明らかになった。

    ユーザが選択したパスワードの複雑さは、情報理論のエントロピーの概念[Shannon]を用いて特徴付けられることが多い。エントロピーは、決定論的な分布関数を持つデータに対しては容易に計算することができるが、ユーザが選択したパスワードのエントロピーを推定することは困難であり、そのための過去の努力は特に正確ではなかった。このため、ここでは主にパスワードの長さに基づく、異なる、やや単純なアプローチを示す。

    パスワードの使用に関連する多くの攻撃は、パスワードの複雑さや長さに影響されません。キーストローク・ロギング、フィッシング、ソーシャル・エンジニアリング攻撃は、長くて複雑なパスワードに対しても、単純なパスワードと同様に有効です。これらの攻撃は、この付録の範囲外である。

    長さ

    パスワードの長さは、パスワードの強さ[Strength] [Composition]を特徴付ける主要な要素である。短すぎるパスワードはブルートフォース攻撃や辞書攻撃に屈する。必要最小限のパスワードの長さは、対応する脅威モデルによって異なる。攻撃者がパスワードを推測してログインを試みるオンライン攻撃は、許可されるログイン試行率を制限することで軽減できる。攻撃者（またはタイピングスキルの低い執拗な請求者）が、多くの不正確な推測を行うことで、素早く加入者にサービス拒否攻撃を与えることを防ぐには、推測が成功する確率が低くても、妥当な回数の試行が許可されるように、パスワードは十分に複雑である必要があり、推測が成功する可能性がかなり高くなる前に、レート制限を適用することができる。

    オフライン攻撃は、攻撃者がデータベース侵入によって1つ以上のハッシュ化されたパスワードを入手した場合に可能なことがある。攻撃者が1人以上のユーザーのパスワードを特定できるかどうかは、パスワードがどのように保存されているかに依存する。一般的に、パスワードはランダムな値で塩付けされ、好ましくは計算コストの高いアルゴリズムを使ってハッシュ化される。このような対策を講じたとしても、攻撃者がレート制限のないオフライン環境で毎秒何十億ものハッシュを計算できる現状では、パスワードはオンライン攻撃だけに耐えられると予想されるものより桁違いに複雑でなければならない。

    ユーザーは、合理的な範囲内で、自分のパスワードを好きなだけ長くすることを推奨されるべきである。ハッシュ化されたパスワードのサイズはその長さに依存しないので、ユーザが望むのであれば、長いパスワード（またはパスフレーズ）の使用を禁止する理由はない。極端に長いパスワード（おそらくメガバイト長）は、ハッシュ化に過大な処理時間を必要とする可能性があるので、ある程度の制限を設けることは合理的です。

    複雑さ

    構成ルールは、ユーザが選択したパスワードの推測の難易度を高めるために一般的に使用される。しかし、構成ルールによって課される要件に対して、ユーザは非常に予測可能な方法で反応することが研究で示されている[Policies]。例えば、パスワードとして「password」を選択したユーザーは、大文字と数字が必 須であれば「Password1」を、記号も必要であれば「Password1!

    また、オンライン・サービスが複雑なパスワードの作成を拒否した場合にも、ユーザーは不満を表明する。多くのサービスは、スペースや様々な特殊文字を含むパスワードを拒否する。受け付けられない文字は、その文字に依存する攻撃（SQLインジェクションなど）を避けるための努力の結果であることもある。しかし、ハッシュ化されていないパスワードがそのままデータベースに送信されることはないので、そのような予防措置は不要である。ユーザはまた、フレーズを使用できるように、スペース文字を含めることができるべきである。スペース文字はパスワードの複雑さにほとんど寄与せず、ユーザビリティの問題（例えば、1つではなく2つのスペースが使用されていることが検出されない）を引き起こす可能性があるため、入力されたパスワードで繰り返されるスペースを削除することは、初期検証に失敗した場合に有益である可能性があります。

    ユーザーのパスワード選択は予測可能であることが多いため、攻撃者は過去に成功したパスワードを推測する可能性が高い。このようなパスワードには、上記の「Password1!このため、ユーザーが選択したパスワードは、受け入れ不可能なパスワードのブロックリストと比較する必要があります。このリストには、過去の侵害コーパスのパスワード、パスワードとして使用された辞書的な単語、ユーザーが選択しそうな特定の単語（例えば、サービス自体の名前）を含める必要があります。最小限の長さの要件は、ユーザーのパスワードの選択にも適用されるため、この辞書には、その 要件を満たすエントリのみを含める必要がある。セクション3.1.1.2で述べられているように、ブロックリストの主な目的は、スロッ トリング制限が有効になる前に、オンライン攻撃で推測される可能性のある、ごく一般 的なパスワードの使用を防ぐことなので、ブロックリストが過度に大きかったり包括的だっ たりすることは有益ではない。過度に大きなブロックリストは、覚えやすいパスワードを選ぼうとするユーザーを挫折させる可能性が高い。

    高度に複雑なパスワードは、新たな潜在的脆弱性をもたらす。それは、記憶に残りにくく、安全でない方法で書き留められたり、電子的に保存されたりする可能性が高いからである。これらの慣行は必ずしも脆弱ではないが、そのような秘密を記録する方法によっては脆弱になる。これは、過度に長いパスワードや複雑なパスワードを要求しないことの、さらなる動機付けとなる。

    中央と地方の検証

    個別の認証要素として使用されるパスワードは、多くの場合、CSP の検証者によって集中的に検証さ れるが、多要素認証機能の起動要素として使用されるパスワードは、認証機能がローカルで検証するか、ま たは認証機能の出力を導き出すために使用される。これらの状況はいずれも、"ローカル検証 "と呼ばれる。

    中央検証とローカル検証では、攻撃対象と脆弱性が大きく異なる。したがって、中央で検証されるパスワードの要件は、ローカルで検証されるパスワードとは異なる。中央で検証されるパスワードは、検証者（すなわちオンラインリソース）が、すべての加入者のパスワー ドについて、塩付けされ、繰り返しハッシュ化された検証秘密を保存することを必要とする。ソルティングとハッシュ化プロセスは、ハッシュからパスワードを決定するための計算量を増加させるが、検証者は攻撃者にとって魅力的なターゲットであり、特に特定の加入者ではなく任意の加入者を危険にさらすことに興味を持つ攻撃者にとって魅力的なターゲットである。

    ローカル検証機には、中央オンライン検証機に対する大規模攻撃と同じ懸念はないが、認証機の 物理的セキュリティおよび関連するエンドポイントの完全性に大きく依存する。認証器が活性化因子を保存する範囲において、その因子は、認証器に対する物理的攻撃お よびサイドチャネル（例えば、電力およびタイミング分析）攻撃から保護されなければならない。活性化因子が関連するエンドポイントを通じて入力される場合、エンドポイントにはマルウェア （キー・ロギング・ソフトウェアなど）が存在しない必要がある。このような脅威はパスワードの長さや複雑さにあまり依存しないため、ローカル検証ではこれらの要件は緩和される。

    オンライン・パスワード推測攻撃は、中央およびローカルで検証されたパスワードに対する同様の脅威である。オンライン攻撃に対する主要な防御であるスロットリングは、失敗した試行に関する情報を安全に保存する認証機の能力が限られているため、ローカル検証機にとって特に困難である。スロットリングは、認証機関に無効な試行のカウントを保持するか、スロットリングを実行する CSP 検証者が拒否する認証機関の出力を生成することで実行できる。この場合、無効な出力が攻撃者に明らかであってはならない。そうでなければ、攻撃者は、有効に見える認証出 力が表示されるまでオフラインで試行することができる。

    概要

    ここで推奨されている以上の長さや複雑さの要件は、パスワード使用の難易度を著しく高め、ユーザーのフラストレーションを増大させます。その結果、ユーザはこれらの制限を逆生産的に回避することが多い。他の緩和策（ブロックリスト、ハッシュ化された安全なストレージ、機械が生成するランダムなパスワード、レート制限など）は、現代のブルートフォース攻撃を防ぐ上でより効果的であるため、複雑さの追加要件は課さない。

    同期可能な認証機能

    この付録は規範的なものである。

    はじめに

    Authenticatorを「同期」する機能（具体的には、Authenticatorの認証秘密をクラウドにコピー（すなわち、 クローン）し、そこからさらにAuthenticatorを追加する）は、認証における比較的新しい開発である。この付録では、同期可能なAuthenticatorの使用に関する追加ガイドラインを示す。

    認証キーのクローニング

    場合によっては、多要素暗号化認証機能（例えば、WebAuthn 標準[WebAuthn]に基づくもの）に 関連付けられた秘密鍵が、シンクファブリックに格納されることがある。これにより、鍵をバックアップし、他のデバイスに転送することができる。この方法で管理される鍵には、以下の要件が適用される：

    • すべての鍵は、承認された暗号技術を用いて生成されなければならない。
    • クローンされた秘密鍵またはデバイスからエクスポートされた秘密鍵は、暗号化された形でのみ保存 されなければならない。
    • すべての認証トランザクションは、デバイス上で生成された、またはシンクファブリック（クラウドストレージな ど）から復元された暗号鍵を使用して、ローカルデバイス上で秘密鍵操作を実行しなければならない（SHALL）。
    • クラウドベースのアカウントに保存される秘密鍵は、認証されたユーザのみがシンクファブリック内の 秘密鍵にアクセスできるよう、アクセス制御メカニズムによって保護されなければならない。
    • 同期ファブリック内の秘密鍵へのユーザ・アクセスは、同期された鍵を使用する認証プロトコルの完全 性を保持するために、AAL2相当のMFAによって保護されなければならない（SHALL）。
    • これらの一般的要件及び同期可能なAuthenticatorを使用するためのその他の機関固有の要件は、該当す る場合には、一般向けウェブサイト及びデジタル・サービス・ポリシーを含め、文書化し、伝えなければならない（SHALL）。

    連邦企業1が同期可能なAuthenticatorを使用するための追加要件：

    • 連邦企業の秘密鍵（すなわち、連邦鍵）は、FISMA Moderate の保護または同等の保護を達成したシンク ファブリックに保管しなければならない。
    • 連邦政府の企業秘密鍵を含むAuthenticatorを生成、保管、同期するデバイス（携帯電話、ノートパソコン、タ ブレットなど）は、モバイル・デバイス管理ソフトウェアまたはその他のデバイス・コンフィギュレーショ ン・コントロールによって保護され、権限のないデバイスまたは同期ファブリックへの鍵の同期または共 有を防止しなければならない。
    • 同期ファブリックへのアクセスは、秘密鍵のライフサイクルに対するエンタープライズ・コントロールを維持するた め、省庁が管理するアカウント（例えば、中央アイデンティティ・アクセス管理ソリューションまたはプラッ トフォーム・ベースの管理アカウント）によって制御されなければならない（SHALL）。
    • 秘密鍵を生成する認証機関は、認証機関の能力およびソースを検証するために使用できる認証 機能をサポートすべきである（例えば、エンタープライズ認証）。

    これらの管理は、特に同期をサポートし、[FIPS140]検証を含む既存の多要素暗号認証要件および AAL2 要件に付加的に考慮されるべきである。

    認証鍵の同期は、本質的に、鍵をエクスポートできることを意味する。AAL2での認証は、上記の要件に従ってサポートされる場合がある。ただし、同期することは、AAL3の非エクスポート可能要件に違反する。AAL3の他の要件を満たす、エクスポート可能な方法で保管されていない鍵を使用する 類似のプロトコルを使用してもよい。

    実施要件

    多くの同期可能な認証機能は、W3C の[WebAuthn]仕様に基づいて構築されている。この仕様 は、共通のデータ構造、チャレンジ・レスポンス暗号プロトコル、公開鍵認証情報を利用するための API を提供する。この仕様は柔軟で適応性があるため、WebAuthn クレデンシャルのすべてのデプロイメントが連邦機関の実装要件を満たすとは限らない。

    この仕様には、認証イベントの追加コンテキストを提供し、それが RP のアクセス・ポリシーに適合しているかどうかを判断するために、RP アプリケーションが認証者に要求できる一連のフラグがある。本節では、RP として活動する連邦機関が、NIST AAL2 ガイドラインに沿った同期可能な 認証機能の実装を構築する際に理解し、問い合わせる必要がある、WebAuthn 仕様の特定のフラ グについて説明する。

    WebAuthnレベル3フラグには以下の要件が適用されます：

    ユーザープレゼント（UP）
    User Presentフラグは、「プレゼンス」テストが、ユーザが認証者と対話したこと（例： USBポートに挿入されたハードウェアトークンをタップする）を確認するために使用されたことを示す。これは、セクション3.2.8で説明される認証意図に対応する。ベリファイアは、User Present フラグが設定されていることを確認すべきである。
    ユーザー認証済み（UV）
    User Verifiedフラグは、認証者が、利用可能な「ユーザー検証」方法のいずれかを使用して、 ユーザーをローカルで認証したことを示す。ベリファイアは、UVが優先されることを示し、UVフラグの値を確認するために応答を検査しなければならない[ SHALL]。これは、認証者が多要素暗号化認証者として扱えるかどうかを示す。ユーザが検証されていない場合、機関は認証者を一要素暗号化認証者として扱わなけれ ばならない[SHALL]。WebAuthnレベル3仕様のさらなる拡張（[WebAuthn]の第10.3項参照）は、機関がローカル認証イベ ントのコンテキストを得ようとする場合、検証方法に関する追加データを提供する。
    バックアップ対象
    Backup Eligible フラグは、Authenticatorを別のデバイスに同期できるかどうか（すなわち、鍵を別の場所に保存できるかどうか）を示す。Authenticatorが同期可能だからといって、同期済みであるとは限らないことに注意することが重要である。ベリファイアは、同期可能なAuthenticatorの使用を制限するポリシーを設定するために、 このフラグを使用してもよい[MAY]。このフラグは、デバイスにバインドされるAuthenticatorと、複数のデバイスにクローンされる Authenticatorを区別するために必要である。
    バックアップ状態
    バックアップ状態フラグは、Authenticatorが別のデバイスに同期されたかどうかを示す。ベリファイアは、このフラグを使用して、他のデバイスに同期されたAuthenticatorに対する制限を設定してもよい（MAY）。機関は、ユーザエクスペリエンス上の懸念から、一般向けアプリケーションでこのフラグに基 づいて受理を条件付けるべきでない(SHOULD NOT)。このフラグは、特定のアプリケーションで同期可能なAuthenticatorの制限をサポートするために、エ ンタープライズアプリケーションに使用してもよい（MAY）。

    上記で指定されたフラグに加えて、機関は、実装し受け入れることを選択した同期可能な認証機 関の出所および能力に関する追加情報を得ることを望むかもしれない。FIDO2 WebAuthn のコンテキストでは、認証者の中には、特定の認証者の能力と製造者を判断するために 使用できる認証機能をサポートしているものがある。企業ユースケースの場合、各機関は、プラットフォームプロバイダが提供する機能に基づいて、認証 機能を実装するべきである（SHOULD）。これは、RP が認証者の識別情報を要求するエンタープライズ認証の形をとる。

    認証は、広範な公衆向けアプリケーションでの同期可能なAuthenticatorの使用をブロックすべきで はない（SHOULD NOT）。コンシューマ製品での利用可能性が限られているため、その使用を義務付けると、フィッシングに脆弱な、より安全性の低い認証オプション（PSTN ベースの帯域外認証など）にユーザが誘導される可能性が高い。ローカルのアクティベーション要素の使用を示すUser Verifiedフラグなどの認証トランザク ションのメタデータは、WebAuthnレスポンスで利用可能であるが、認証は、トランザクショ ンで使用されるAuthenticatorの特性について、より強力な保証を提供することができる。 RPは、同期可能なAuthenticatorに対する信頼度を決定するために、証明書を使用してもよい[MAY]。

    RP がフラグおよび認証データを要求しても、認証局は要求された情報をすべて返さないかもしれず、ま たはリソースへのアクセスに義務付けられる期待応答と矛盾する情報を返すかもしれない。機関は、同期可能な認証局のユースケースを評価し、返された情報に基づいて、適切なアクセ スポリシー決定を行うことを決定しなければならない。

    シェアリング

    サイバーセキュリティ・ガイドラインは、歴史的に、ユーザ間でAuthenticatorを共有しないよう注意を喚起しており、異なるユーザが独自のAuthenticatorを保持することを期待している。このガイダンスにもかかわらず、一部のユーザ・グループやアプリケーションでは、個人がデジタル・アカウントへのアクセスを共有できるようにするために、Authenticatorとパスワードの共有が行われている。

    表 5 に示すように、一部の同期可能認証機能実装は、このようなユーザの行動を受け入れ、 異なるユーザ間で認証キーを共有する方法を確立している。さらに、一部の実装では、一般的なサービスのパスワードを共有するよりも便利で安全な代替手段 として、同期可能Authenticatorの共有を積極的に推奨している。

    企業ユースケースの場合、鍵の共有に関する懸念は、鍵が承認されたデバイスや同期 ファブリックから移動されることを制限するデバイス管理技術を使用することで、効果的に 軽減することができる。しかし、公衆向けのユースケースでは、同様の緩和策は現在のところ利用できないため、RP は同期可能な認証プロバイダが採用する共有モデルに依存することになる。公衆向けアプリケーションの所有者は、共有Authenticatorに関連するリスクを認識する必要がある。一般ユーザと対話する場合、機関は、ユーザがどの特定のAuthenticatorを使用しているかにつ いての可視性が限られており、すべての同期可能なAuthenticatorが共有の対象となる可能性があること を想定する必要がある。多くの共有モデルには、リスクを最小化する実質的な制御（例えば、共有を許可するためにデバイス間 の近接を要求する）があるが、他の実装はそれほど制限的ではない。

    この新しいクラスの認証者がもたらす共有リスクは、特別なものではない。このリスクは、同期可能なAuthenticatorより弱いものも含め、すべてのAuthenticatorに適用される。どのようなAuthenticatorも、それを共有すると決めたユーザによって共有される可能性がある。ユーザは、パスワード、OTP、帯域外Authenticator、さらには被指定者（正式か否かを問わない）がエンドユ ーザの代理として認証を行うことを許可するプッシュ認証イベントを積極的に共有することができる。

    各省庁は、直面する特定のリスク、脅威、およびユーザビリティの考慮事項に基づいて、どの 認証機関をアプリケーションに受け入れるかを決定する。同期可能な認証方式は、AAL2 までの実装を求めるアプリケーションの新しいオプションとして提供されるかもしれない。この技術のトレードオフは、セキュリ ティ、プライバシー、公平性、ユーザビリティに期待される結果に基づいて、うまくバランスを取るべ きである。

    活性化
    認証に使用できるようにするために、多要素認証機能に起動要素を入力するプロセス。
    活性化因子

    アサーション

    認証する
    認証を参照。
    認証済み保護チャネル
    接続の開始者（クライアント）が受信者（サーバー）を認証している、承認された暗号を使用する暗号化通信チャネル。認証された保護されたチャネルは、機密性とアクティブな仲介者からの保護を提供するために暗号化され、ユーザー認証プロセスで頻繁に使用される。トランスポートレイヤーセキュリティ(TLS)およびデータグラムトランス ポートレイヤーセキュリティ(DTLS)[RFC9325]は、受信者が提示する証明書がイニ シエータによって検証される、認証された保護されたチャネルの例である。別段の指定がない限り、認証付き保護チャネルは、サーバーがクライア ントを認証することを要求しない。サーバーの認証は、多くの場合、各サーバーで個別に行われるのでは なく、信頼されたルートにつながる証明書チェーンを通して達成される。

    認証
    請求者が、加入者アカウントにバインドされた 1 つまたは複数のAuthenticatorの所有および管理を証明し、 そのアカウントに関連する加入者であることを証明するプロセス。
    認証保証レベル（AAL）
    認証プロセスの強度を表すカテゴリー。
    認証要素
    認証要素には、「知っているもの」、「持っているもの」、「存在するもの」の 3 種類がある。すべての認証者は、1 つ以上の認証要素を持っている。
    認証の意図
    認証フローにおいてユーザの介入を要求することで、請求者の 認証または再認証の 意図を確認するプロセス。一部のAuthenticator（OTP など）は、動作の一部として認証の意図を確立する。また、意思を確立するために、ボタンを押すなどの特定のステップを必要とするものもある。認証意図は、加入者が知らないうちに攻撃者を認証するためのプロキシとして、エンドポイン トでマルウェアが使用することに対する対策である。
    認証プロトコル
    請求者と検証者の間で定義される一連のメッセージであり、請求者が自己の身元を証明するため に 1 つ以上の有効な認証子を所有し管理していることを示し、オプションとして、請求者 が意図する検証者と通信していることを示す。
    認証秘密
    攻撃者が認証プロトコルで加入者になりすますために使用できる秘密値の総称。

    これらはさらに、攻撃者にとって限られた期間のみ有効な短期認証秘密 (short-term authentication secret)と、攻撃者が手動でリセットするまで加入者になりすますこ とを可能にする長期認証秘密(long-term authentication secret)に分けられる。Authenticator秘密鍵は長期認証秘密鍵の典型的な例であり、Authenticator出力(Authenticator秘密鍵と異なる場合)は通常、短期認証秘密鍵である。

    認証者
    加入者が所有し管理するもの（暗号モジュールまたはパスワードなど）で、請求者の身元を認証 するために使用されるもの。Authenticatorタイプおよび多要素Authenticatorを参照。
    Authenticator結合
    特定の認証機関と加入者アカウントとの間の関連付けを確立し、その認証機関を、場合によっては他の 認証機関と組み合わせて、その加入者アカウントの認証に使用できるようにすること。
    認証出力
    Authenticatorによって生成される出力値。有効な認証機能出力をオンデマンドで生成する能力は、請求者が認証機能を所有し、 管理していることを証明する。ベリファイアに送信されるプロトコルメッセージは、Authenticator出力に依存するが、Authenticator出力を 明示的に含む場合もあれば、含まない場合もある。
    認証者秘密
    認証子に含まれる秘密値。
    認証タイプ
    提供する認証要素の種類や動作メカニズムなど、共通の特徴を持つAuthenticatorのカテゴリー。
    信憑性
    データがその出所とされる場所に由来するという性質。
    認可する
    通常、対象者の 属性を評価することによって自動化される。

    クレーマー

    信任状

    例えば、銀行のウェブサイトが CSRF 攻撃に脆弱な場合、銀行への接続が別のブラウザウィンドウで開かれている間に、電子メール内の悪意のあるリンクをクリックすることで、加入者が意図せずに多額の送金を承認してしまう可能性がある。

    クロスサイトスクリプティング（XSS）

    認証鍵をクローンし、他のストレージにエクスポートすることで、それらの鍵を他の認証機 器（すなわちデバイス）に同期させることができるソフトウェアまたはハードウェアの暗号化認証機器。
    レコードシステム（SOR）
    SORは、個人に関する情報を含む、機関の管理下にある記録のコレクションである。記録は、個人の名前または識別番号、記号、その他の識別子によって検索することができる。
    記録システム通知（SORN）
    連邦政府機関が連邦官報（Federal Register）に掲載する、記録システムに関する通知。
